# 意健险研发团队 AI 工程化体系建设方案

## 1. 简介

本方案是在公司已有研发基础设施、研发流程和 DevOps 工具链基础上，建设意健险研发团队专属的 **AI 工程化体系**。

该体系通过把 **大模型、Coding Agent、团队知识库、研发规范、质量门禁、自动化测试和 DevOps 工具** 连接起来，形成可持续、可度量、可治理的 AI 研发能力。

本方案是目标项目落地蓝图和 V1 需求来源，不替代具体执行计划、项目进度台账或单个资产详细设计。当前仓库执行以 `docs/V1_IMPLEMENTATION_PLAN.md` 和 `docs/PROGRESS.md` 为准。

建设范围覆盖研发全流程，但 V1 只交付可人工执行、可审计、可复制的 Markdown-first Harness 资产：

```text
需求输入与评审
    ↓
技术设计辅助
    ↓
编码计划与 ADR 判断
    ↓
TDD 与辅助编码
    ↓
代码审查与提交检查
    ↓
测试场景生成与验证
    ↓
发布前检查
    ↓
运维监控分析（V3）
```

核心目标：

1. **提升研发效率**：缩短需求澄清、开发实现、测试验证、缺陷修复和新人上手周期。
    
2. **提升研发质量**：提升需求完整性、设计合理性、代码规范性、测试覆盖率和发布可靠性。
    
3. **沉淀团队工程资产**：形成意健险领域知识库、上下文文档、Prompt 模板、代码规则、测试规则和缺陷案例库。
    
4. **建立可控 AI 研发闭环**：通过上下文、约束、验证、反馈和修正机制，让 AI 能力稳定服务研发流程。
    

**限制**

该体系需考虑的限制有：

1. 面对大型、陈旧、架构复杂、非规范化的保险核心系统。
2. V1 主要面向 OpenCode 试点落地，使用模型为 DeepSeek V4 Flash，虽然上下文窗口较大，但推理稳定性、工具能力和算力仍需按受限模型处理。
3. 团队成员 AI 使用能力参差不齐，资产必须短、清晰、结构化，降低选择和执行成本。
4. V1 不做自动生成核心交易代码、自动合并、自动发布、自动修改生产配置或直接访问生产数据。

## 1.1 方案定位与范围边界

本方案用于说明意健险研发团队如何在目标项目中落地 AI Harness 工程化体系。

V1 的定位是：

```text
OpenCode 优先
Markdown-first
少量核心 Skill
短入口 + 按需加载 + 结构化输出
提醒型门禁为主
敏感信息和 P0 风险必须给出阻断建议
仓库资产验收与试点效果验收分层
```

V1 不把以下事项作为首批核心交付：

```text
自动生成正式 PRD
完整设计方案生成
正式实现计划自动拆解
多 Agent 编排
自动化测试平台接入
Git Hook / CI / MR / 发布平台深度集成
监控分析实时预警
组织排班、发布窗口或人员绩效管理
```

这些能力可以作为 V2/V3 集成方向，但 V1 应先把可读、可用、可审计、可复盘的 Harness 资产做稳。

# 2. 目标及建设原则

## 2.1 目标

目标分为两类：

1. **结果指标**：衡量最终效率和质量是否提升。
    
2. **过程指标**：衡量 AI 工程化体系是否真正被使用、被治理、被沉淀。
    

---

## 2.1.1 结果指标

### 1）效率指标

|指标|现状|目标建议|说明|数据来源|
|---|--:|--:|---|---|
|需求到开发启动时间|待统计|降低 20%-40%|从需求提出到完成澄清、影响分析、可进入开发的时间|需求管理系统、神兵流程|
|开发周期|待统计|降低 15%-30%|Story 从开发启动到提测的耗时|需求/任务流转记录|
|缺陷修复耗时|待统计|降低 20%-40%|Bug 从定位到修复完成的时间|缺陷管理系统|
|新人上手时间|待统计|降低 30%-50%|新人理解系统并完成首个有效任务的周期|任务记录、导师反馈|
|代码 Review 周期|待统计|降低 20%-30%|MR 从提交到合并的平均耗时|Git 平台|
|单测编写耗时|待统计|降低 30%-50%|开发编写单元测试所需时间|开发自评、任务记录|

V1 阶段建议先采集 1 个月基线数据，再确认正式目标值，避免目标不可验收。

### 2）质量指标

|指标|现状|目标建议|说明|数据来源|
|---|--:|--:|---|---|
|单测覆盖率|待统计|提升 10%-30%|AI 生成测试是否提升覆盖|Sonar、CI|
|代码缺陷密度|待统计|降低 10%-25%|每千行代码缺陷数|缺陷系统、代码统计|
|人均缺陷密度|待统计|降低 10%-20%|每人每月有效缺陷数|缺陷系统|

|AI Review 有效问题率|待统计|≥60%|AI 提出的问题被人工确认有效的比例|Review 反馈|
|生产事故数|待统计|不增加，逐步下降|是否因 AI 生成代码引入事故|生产事件记录|
|回归缺陷率|待统计|降低 10%-20%|测试后或上线后发现的回归问题|缺陷系统|
|发布遗漏数|待统计|降低 30%-50%|漏合代码、漏发 SQL、漏改配置等|发布复盘|

### 1）规范指标

|指标|目标|说明|数据来源|
|---|--:|---|---|

### 2）安全指标

|指标|目标|说明|
|---|--:|---|
|敏感数据泄露事件|0|不允许向外部模型提交客户、生产、密钥等敏感数据|
|AI 直接操作生产事件|0|AI 不得直接操作生产环境|
|未经 Review 的 AI 代码合入|0|AI 生成代码必须人工 Review|
|高风险命令未审批执行|0|删除、覆盖、生产连接等高风险命令必须审批|
|AI 生成代码引发严重生产事故|0|AI 不能绕过质量门禁|
|密钥/Token 被 AI 读取或输出|0|禁止 AI 读取敏感配置文件|
|生产日志未脱敏输入 AI|0|日志进入 AI 前必须脱敏|

# 2.2 建设原则

## 2.2.1 AI 辅助，人工决策

AI 主要负责：

```text
生成建议
提炼上下文
发现风险
生成草稿
辅助编码
生成测试
执行检查
辅助分析
```

人工必须负责：

```text
需求确认
设计决策
核心逻辑判断
代码最终质量
测试结果确认
发布决策
生产问题处置
```

以下场景必须人工主导：

```text
保费计算
退保金额计算
责任判断
核保规则
理赔规则
账务处理
客户隐私处理
权限控制
生产配置
发布操作
```

---

## 2.2.2 先低风险，再核心逻辑

V1 优先推进：

```text
需求评审
编码计划辅助
代码解释
TDD 用例生成
单元测试生成
代码审查
提交前检查
发布前检查
核心场景测试
```

V2/V3 再推进：

```text
设计方案生成
正式实现计划自动拆解
ADR 草稿生成
自动化测试生成
监控分析
工程引导
多 Agent 编排
```

暂不建议 V1 推进：

```text
自动生成核心交易代码
AI 自动合并 MR
AI 自动发布
AI 自动修改生产配置
AI 直接访问生产数据
```

---

## 2.2.3 Harness 闭环原则

AI 工程化体系不是简单调用大模型，而是建立如下闭环：

```text
上下文 Context
    ↓
计划 Plan
    ↓
约束 Constraint
    ↓
执行 Action
    ↓
验证 Verification
    ↓
反馈 Feedback
    ↓
修正 Correction
    ↓
沉淀 Knowledge
```

对应到研发流程中：

|Harness 要素|在研发中的体现|
|---|---|
|上下文|需求文档、领域知识、代码上下文、架构说明、历史缺陷|
|计划|编码计划、测试计划、发布计划、ADR 判断|
|约束|编码规范、测试规范、安全规范、架构规则、发布规则|
|执行|消费需求输入、辅助编码、生成测试、代码审查、发布检查|
|验证|单测、静态扫描、AI Review、门禁、人工评审|
|反馈|测试失败、Review 意见、缺陷、发布问题|
|修正|AI 辅助修改、补测试、补设计、补文档|
|沉淀|更新 `docs/` 文档、`.ai-harness/` 规则、缺陷案例库|

---

## 2.2.4 规范统一

统一内容包括：

```text
统一 Agent 入口文件
统一上下文文档
统一 Prompt 模板
统一 Skill 输入输出
统一代码规范
统一测试规范
统一 Review 标准
统一提交检查
统一发布检查
统一安全边界
统一度量指标
```

避免不同项目、不同开发人员各自一套 Prompt 和规则，否则后续很难治理。

---

## 2.2.5 可度量、可验收、可复盘

每个 AI 工具都必须明确：

```text
解决什么问题
输入是什么
输出是什么
谁来使用
何时触发
如何评价效果
失败如何处理
结果如何沉淀
```

否则容易变成“看起来很智能，但无法证明价值”。

---

## 2.2.6 短入口、按需加载、结构化输出

V1 面向 OpenCode 和受限模型落地，必须避免把所有业务知识、规则、模板一次性塞进入口文件。

设计原则：

1. `AGENTS.md` 只做入口、边界、读取顺序和关键禁令。
2. Prompt 和 Skill 必须绑定具体研发场景，不写万能 Prompt。
3. 每次只读取当前场景必要上下文，不默认全量读取所有文档。
4. 输出固定包含结论、依据、风险等级、建议动作、待确认问题和上下文更新建议。
5. 高风险结论必须说明依据来源；缺少输入时列为待确认，不补编业务规则。

---

## 2.2.7 统一风险分级与场景矩阵

金融核心系统风险必须统一分级，供 Prompt、Skill、门禁和验收资产共同引用。

|等级|含义|处理要求|
|---|---|---|
|P0|可能导致资金、保单、客户权益错误；敏感信息泄露；越权；生产事故；不可回滚发布|必须修复或给出阻断建议|
|P1|可能导致核心流程异常、边界条件错误、数据不一致、重要测试缺失|修复或由负责人明确豁免|
|P2|规范、可维护性、日志、性能、测试质量问题|建议修复并记录|
|P3|风格、可读性、局部优化建议|可选采纳|

不同场景的必检重点：

|场景|必检风险|
|---|---|
|需求评审|业务规则缺失、验收标准不清、金额/日期/状态/权限规则缺口|
|编码计划|影响范围遗漏、事务边界、幂等策略、外部依赖、ADR 触发点|
|TDD|正常、边界、异常、重复请求、并发、Mock、断言和测试数据|
|辅助编码|超范围修改、核心逻辑错误、金额精度、异常处理、日志审计、测试缺失|
|代码审查|需求一致性、业务逻辑、权限、安全、性能、可维护性、测试覆盖|
|提交检查|需求关联、分支、CI、单测、敏感信息、SQL/配置/脚本遗漏|
|核心场景测试|必测场景、回归范围、历史缺陷相似场景、自动化可行性|
|发布检查|需求清单、分支制品、SQL、配置、缺陷、验证点、回滚方案|

---

## 2.2.8 Review 指令精简与分层引用

Review 规则必须短、明确、可路由，不能把所有语言规则、安全规则、业务规则和历史缺陷都塞进一个超长文件。

外部工具约束示例：GitHub Copilot Code Review 对自定义指令文件存在前 4000 个字符的读取限制，超出部分不会影响 Review 结果。这说明 V1 设计 Review 规则时必须优先保证“前置总则足够短、关键禁令足够靠前、细则按需引用”。

V1 Review 规则采用如下分层：

```text
docs/REVIEW_RULES.md                   # 通用 Review 总则、风险分级、加载路由
docs/review-rules/java.md              # Java 规则
docs/review-rules/sql.md               # SQL 规则
docs/review-rules/security.md          # 安全规则
docs/review-rules/insurance-domain.md  # 保险业务规则
```

`.ai-harness/prompts/code-review.md` 不复制所有 Review 细则，而是根据本次 diff、技术栈、SQL/配置变更、安全风险和保险业务影响，按需引用上述规则文件。

设计要求：

1. `docs/REVIEW_RULES.md` 只保留通用总则、P0/P1/P2/P3 分级、必读输入和场景路由。
2. Java、SQL、安全、保险业务等细则拆入 `docs/review-rules/`。
3. 高风险禁令、敏感信息、P0 风险和人工 Review 要求必须放在总则前部。
4. Review Prompt 或 Skill 每次只加载与本次变更相关的规则文件。
5. 兼容 Copilot 等短指令工具时，只把压缩后的总则映射到自定义指令文件，详细规则保留在 Harness 文档中。

# 3. 方案

## 3.1 总览

针对研发全流程，围绕 LLM 和 Coding Agent，建设意健险研发团队专属的 AI Harness 工程。

在需求和设计阶段，重点是 **明确上下文**；  
在编码和测试阶段，重点是 **建立计划、约束、验证、反馈和修正闭环**；  
在发布和运维阶段，重点是 **降低风险、辅助验证、快速定位问题**；  
在整体流程层面，重点是 **节点门禁和工程引导**。

---

## 3.1.1 全流程能力总览

|阶段|能力|版本|核心目标|主要产出|
|---|---|--:|---|---|
|需求|生成需求文档|外部前置|由公司既有 AI 工具或需求管理流程生成 PRD 草稿|需求文档|
|需求|需求评审|V1|审计需求完整性、歧义、风险|需求评审报告、`PRD_CONTEXT.md`|
|设计|生成设计方案|V2|生成架构、接口、数据、非功能设计草案|技术设计方案|
|编码|编码计划辅助|V1|生成轻量编码计划，并判断是否需要 ADR|编码计划、ADR 判断|
|编码|正式实现计划/用户故事拆解|V2|拆解任务、理清逻辑、识别冲突|正式实现计划|
|编码|TDD|V1|先生成测试场景、测试用例和验证点，再辅助实现|单测、测试用例|
|编码|辅助编码|V1|代码生成、解释、重构、修复|代码、代码说明|
|编码|代码审查|V1|基于规则、需求、案例进行 Review|代码审查报告|
|编码|提交检查|V1|提交前检查单测、规范、故事覆盖|提交检查报告|
|测试|核心场景测试|V1|建立核心业务测试场景库|测试场景、用例|
|测试|自动化测试|V3|生成并维护自动化测试脚本|自动化脚本、报告|
|发布|发布前检查|V1|核对需求、提交、分支、配置、验证点|发布检查报告|
|运维|监控分析|V3|分析日志、指标、告警并预警|监控分析报告|
|整体|节点门禁|V1|在关键流转节点做风险检查|门禁报告|
|整体|工程引导|V3|引导开发完成下一步工程任务|下一步建议、任务清单|

---

## 3.1.2 V1 聚焦范围

V1 建议聚焦“低风险、高收益、易执行、可审计”的能力。

|能力|优先级|V1 定位|原因|
|---|--:|---|---|
|需求评审|P0|必做|最容易提前减少返工|
|编码计划辅助|P0/P1|重点需求必做，普通需求可选|为 TDD、辅助编码、代码审查提供计划输入|
|TDD|P0|必做|对质量提升最直接|
|辅助编码|P0|必做|开发者日常提效明显|
|代码审查|P0|必做|易嵌入 MR 流程|
|提交检查|P0|必做|可快速降低低级错误|
|核心场景测试|P1|必做|沉淀团队测试资产|
|发布前检查|P0|必做|直接降低发布风险|
|节点门禁|P0|必做|把 AI 能力变成流程保障|

V1 暂不重点做：

|能力|原因|
|---|---|
|自动生成正式 PRD|PRD 生成由公司既有 AI 工具或需求流程提供，V1 Harness 消费其输出|
|完整设计方案生成|依赖架构上下文和历史设计资产，建议 V2|
|正式实现计划自动生成|需要深度读取代码库和需求系统，建议 V2|
|ADR 自动更新并合入|ADR 属于架构决策记录，必须由技术负责人确认，建议 V2|
|自动化测试全面生成|依赖测试平台和稳定用例资产，建议 V3|
|Git Hook / CI / MR 深度集成|V1 先交付 Markdown-first 资产，平台集成放入 V2/V3|
|监控分析实时预警|依赖日志、指标、链路和发布数据打通，建议 V3|
|工程引导|依赖全流程状态感知，建议 V3|

## 3.3.3 `AGENTS.md` 定位

`AGENTS.md` 是 AI Coding Agent 进入项目后第一个读取的文件，应放在项目根目录。

它的职责是：

```text
告诉 Agent 当前项目是什么
告诉 Agent 应该按场景读取哪些文档
告诉 Agent 不同任务应遵守什么流程
告诉 Agent 哪些目录可以读写
告诉 Agent 哪些行为禁止
告诉 Agent 如何更新上下文
告诉 Agent 如何输出结果
```

`AGENTS.md` 不应写成冗长知识库，而应作为 **索引、导航和规则入口**。

```markdown
## 4. 安全规则

禁止执行以下行为：

1. 读取或输出生产账号、密码、Token、密钥。
2. 读取 `.env`、生产配置文件、密钥文件。
3. 向外部模型提交客户敏感信息。
4. 直接操作生产环境。
5. 自动合并 MR。
6. 自动发布。
7. 自动修改生产配置。
8. 执行破坏性命令。
9. 删除未确认的代码、测试、配置或脚本。

涉及以下信息时必须脱敏：

- 姓名
- 身份证号
- 手机号
- 银行卡号
- 保单号
- 客户号
- 地址
- 邮箱
- Token
- Cookie
- Session
- 数据库连接串
- 生产 IP
- 内部密钥

## 5. 编码规则摘要

详细规则见 `docs/CODING_RULES.md`。

关键要求：

1. 不允许在 controller 中编写业务逻辑。
2. 金额计算必须使用 BigDecimal。
3. 日期计算必须明确起止边界。
4. 外部系统调用必须有超时、异常处理和日志。
5. 核心交易必须记录 businessTraceId。
6. 不允许单元测试访问真实数据库。
7. 不允许提交生产密钥、Token、客户敏感信息。

---

## 6. 测试规则摘要

详细规则见 `docs/TESTING_GUIDE.md`。

关键要求：

1. 核心业务逻辑必须有单元测试。
2. 测试必须覆盖正常、边界、异常场景。
3. 外部系统必须 Mock。
4. 金额断言必须使用 BigDecimal。
5. 日期相关测试必须覆盖起保日、终保日、越界和闰年场景。

## 7. 上下文更新规则

当任务产生新的领域知识、业务规则、关键流程、历史风险或技术决策时，应提出文档更新建议。

允许建议更新：

- `docs/DOMAIN_KNOWLEDGE.md`
- `docs/PROJ_CONTEXT.md`
- `docs/PRD_CONTEXT.md`
- `docs/DEFECT_CASES.md`
- `docs/ADR/`

更新原则：

1. 不直接覆盖重要内容。
2. 标明新增、修改或废弃。
3. 对不确定内容标注“待确认”。
4. 涉及架构决策时，只生成 ADR 草稿，不自动合入。
```

# 4.2 编码阶段

编码阶段的核心是：基于明确上下文和工程约束，形成 **计划 → 测试 → 编码 → 审查 → 修正 → 提交** 的闭环。

建议编码阶段标准流程：

```text

    ↓

    ↓

    ↓

    ↓
辅助编码实现
    ↓
执行单测和静态检查
    ↓
AI 代码审查
    ↓
修正问题
    ↓
提交前检查
    ↓
创建 MR
```

## 4.2.1 编码计划辅助 Skill

### 工具定位

建设 **编码计划辅助 Skill**。该 Skill 基于需求上下文、代码上下文、编码规范和测试指引，辅助开发人员在正式编码前生成轻量编码计划，并判断本次需求是否需要新增或更新 ADR。

V1 阶段不建设完整自动化实现计划 Agent，不强制所有需求都必须生成正式实现计划。V1 重点是通过轻量编码计划，帮助开发人员明确：

```text
改什么
改哪里
怎么改
怎么测
有什么风险
是否涉及架构决策
```

### 目标

|目标|说明|
|---|---|
|明确开发范围|在编码前明确涉及的模块、类、接口、表、配置等|
|降低编码启动成本|帮助开发人员快速理解需求和改动路径|
|限定 AI 编码边界|避免辅助编码阶段大范围、无约束修改代码|
|支撑 TDD|为 TDD Skill 提供测试范围、验证点和风险点|
|支撑代码审查|为代码审查提供“本次应实现什么”的依据|
|识别 ADR 需求|判断本次变更是否涉及架构决策，需要新增或更新 ADR|

### 适用范围

V1 阶段建议对以下需求使用编码计划辅助：

```text
重点需求
跨模块需求
核心业务逻辑需求
涉及数据库变更的需求
涉及接口变更的需求
涉及金额、日期、状态流转的需求
涉及外部系统交互的需求
涉及配置变更的需求
存在较多澄清点或风险点的需求
```

普通小需求、简单字段调整、简单文案修改、低风险 Bug 修复，可以不强制生成编码计划。

### 输入

|输入|说明|
|---|---|
|`AGENTS.md`|Agent 第一入口，说明项目规则、上下文路径和安全边界|
|`docs/PRD_CONTEXT.md`|单个需求上下文，包括需求要点、业务规则、澄清点、风险点|
|`docs/PROJ_CONTEXT.md`|当前代码仓库或模块上下文|
|`docs/CODING_RULES.md`|编码规范、分层规范、异常、日志、事务、幂等规则|
|`docs/TESTING_GUIDE.md`|测试规范、单测要求、Mock 规则、断言规则|
|`docs/DOMAIN_KNOWLEDGE.md`|意健险领域知识和业务规则|
|`docs/DEFECT_CASES.md`|历史缺陷案例和防范规则|
|相关代码片段|本次需求可能涉及的类、接口、配置、SQL 等|
|需求/故事链接|神兵或需求管理系统中的需求链接|

### 处理逻辑

|分析项|内容|
|---|---|
|需求摘要|提炼本次需求目标和核心业务规则|
|影响范围|识别涉及系统、模块、类、接口、数据库、配置、批处理|
|实现步骤|给出建议开发顺序|
|风险识别|识别金额、日期、状态、事务、幂等、兼容、安全等风险|
|测试建议|提炼 TDD 和核心场景测试需要覆盖的内容|
|待确认问题|输出开发前必须澄清的问题|
|ADR 判断|判断是否需要新增或更新 ADR|
|上下文更新|建议需要补充到 `docs/PROJ_CONTEXT.md` 或 `docs/PRD_CONTEXT.md` 的内容|

### ADR 判断规则

V1 阶段不要求 AI 自动更新 ADR，但要求 AI 判断本次需求是否可能需要 ADR。

需要 ADR 的典型场景包括：

|场景|是否需要 ADR|
|---|---|
|修改系统架构分层|需要|
|引入新技术组件|需要|
|改变核心数据模型|需要|
|改变关键接口契约|需要|
|改变事务边界|需要|
|改变幂等策略|需要|
|改变一致性方案|需要|
|改变发布或回滚策略|需要|
|改变核心业务处理链路|需要|
|普通字段新增|通常不需要|
|简单 Bug 修复|通常不需要|
|小范围页面调整|通常不需要|
|普通文案修改|不需要|

## 4.2.4 代码审查 Skill

### 工具定位

代码审查 Skill 是独立 Agent，用于在人工 Review 前，对本次变更进行 AI Review。

它不是替代人工 Reviewer，而是帮助提前发现常见问题，让人工 Review 聚焦核心业务判断。

### 目标

|目标|说明|
|---|---|
|发现代码问题|识别规范、逻辑、异常、性能、安全问题|
|检查需求一致性|判断代码是否覆盖 `docs/PRD_CONTEXT.md`|
|提升 Review 效率|降低人工 Review 低级问题负担|
|沉淀规则|将常见问题反哺 `docs/REVIEW_RULES.md`、`docs/review-rules/` 和 `docs/DEFECT_CASES.md`|

### 输入

```text
docs/PROJ_CONTEXT.md

docs/REVIEW_RULES.md

docs/review-rules/insurance-domain.md（按需）
docs/TESTING_GUIDE.md
docs/DEFECT_CASES.md
编码计划
单测结果
```

### Review 规则加载策略

代码审查 Skill 先读取 `docs/REVIEW_RULES.md`，再根据本次 diff 按需读取分场景规则：

|触发条件|补充读取|
|---|---|
|涉及 Java 代码、Spring、事务、异常、日志、分层或外部调用|`docs/review-rules/java.md`|
|涉及 SQL、DDL、DML、索引、批量更新、数据修复或配置脚本|`docs/review-rules/sql.md`|
|涉及权限、认证、脱敏、日志、密钥、外部接口或客户敏感信息|`docs/review-rules/security.md`|
|涉及投保、批改、退保、核保、理赔前置、收付费、责任、费率、保单/批单状态|`docs/review-rules/insurance-domain.md`|

`.ai-harness/prompts/code-review.md` 负责把上述规则作为引用路径和加载决策写清楚，不把全部规则正文复制进 Prompt。

### 检查维度

|维度|检查内容|
|---|---|
|需求一致性|是否实现需求要点，是否遗漏验收标准|
|架构规范|是否符合分层、职责、依赖方向|
|业务逻辑|金额、日期、状态、权限、幂等是否正确|
|异常处理|是否吞异常、异常类型是否正确|
|日志审计|核心交易是否有必要日志|
|性能风险|循环查库、N+1 查询、大事务、重复远程调用|
|安全风险|敏感信息、权限绕过、SQL 注入、日志泄露|
|测试覆盖|是否覆盖正常、边界、异常|
|可维护性|是否过度复杂、重复代码、命名不清|

## 4.6 Harness 资产效果评估与回归测试

V1 资产必须可验证。Prompt、Skill、门禁和 Schema 每次新增或修改后，应通过固定样例做轻量回归，避免规则漂移、格式失控或高风险场景漏检。

### 回归资产

|资产|内容|
|---|---|
|示例输入|脱敏需求、脱敏 diff、测试结果、发布清单、历史缺陷摘要|
|期望输出|结构化报告章节、风险等级、建议动作、待确认问题|
|高风险样例|金额精度、状态流转、幂等、权限、隐私、SQL、配置、回滚|
|误报/漏报记录|AI 提出但人工判定无效的问题，以及 AI 漏掉但人工发现的问题|
|复盘记录|Prompt、Skill、门禁规则需要如何调整|

### 评估指标

|指标|说明|
|---|---|
|格式稳定性|输出是否稳定包含结论、依据、风险、建议动作、待确认问题|
|有效问题率|AI 发现的问题被人工确认有效的比例|
|误报率|AI 报告中被人工判定无效的问题比例|
|漏报率|人工发现但 AI 未识别的关键问题比例|
|高风险覆盖率|P0/P1 样例是否被正确识别和分级|
|安全执行率|是否拒绝或脱敏处理敏感信息、密钥和生产数据|

### 使用规则

```text
1. 新增 Prompt 或 Skill 时必须提供至少一个最小示例。
2. 新增门禁规则时必须提供通过、有条件通过、不通过三类样例。
3. 修改跨场景规则时必须回归需求评审、代码审查、发布检查至少三个样例。
4. 所有样例不得包含真实客户敏感信息、生产数据、密钥或生产配置。
5. 回归结果只作为资产质量证据，不替代试点运行效果评估。
```

# 5. 权限与安全治理

## 5.1 Agent 权限分级

|等级|权限|适用场景|
|---|---|---|
|L0|只读分析|需求评审、代码解释、日志分析|
|L1|生成建议和补丁|编码计划、测试用例、代码建议|
|L2|修改本地代码|辅助编码、重构、单测生成|
|L3|执行受限命令|运行单测、构建、静态扫描|
|L4|接入 CI/CD 或发布平台|仅限平台服务账号，严格审批|

V1 阶段建议控制在 L0-L2，部分项目可开放 L3 的受限命令执行。L4 不作为 V1 人工试点的默认能力。

---

## 5.2 工具权限与操作审计矩阵

|场景 / Skill|文件读取|文件修改|命令执行|禁止或审批事项|审计证据|
|---|---|---|---|---|---|
|需求评审|需求文档、`docs/` 上下文、脱敏历史材料|仅输出 `PRD_CONTEXT.md` 更新建议|不需要|不得读取客户敏感信息原文或生产数据|需求评审报告|
|编码计划|`AGENTS.md`、`PRD_CONTEXT.md`、`PROJ_CONTEXT.md`、规范文档、相关代码片段|仅输出计划和上下文更新建议|不需要|不得直接修改代码或生成完整实现|编码计划报告|
|TDD|需求上下文、测试规范、相关代码、已有测试|可生成测试草稿；是否落盘由开发确认|可运行受限单测|不得访问真实数据库或生产依赖|TDD 报告、测试结果|
|辅助编码|需求上下文、编码计划、TDD、相关代码|可修改本地代码和测试|可运行测试、构建、静态检查|不得读密钥、生产配置、未脱敏数据；不得超范围修改|变更摘要、测试结果|
|代码审查|diff、需求上下文、规范、测试结果、历史缺陷|不修改代码|不需要|不得替代人工 Review 或直接合并|代码审查报告|
|提交检查|diff、分支信息、测试结果、配置/SQL/脚本清单|不修改代码；只输出 MR 描述建议|可读取本地检查结果|不得自动提交、自动合并、隐藏风险|提交检查报告|
|核心场景测试|需求、领域知识、测试规范、历史缺陷、已有用例|只输出测试场景和用例建议|可运行受限测试|不得生成含真实客户信息的测试数据|核心场景测试报告|
|发布检查|发布清单、分支制品、测试报告、缺陷状态、SQL/配置摘要|不修改生产配置或发布材料|不执行发布命令|不得自动发布、自动改生产配置、连接生产环境|发布检查报告|
|节点门禁|对应节点输入和前序报告|不修改业务代码|V1 默认不执行命令|敏感信息、P0 风险、生产越权必须给出阻断建议|门禁报告|

受限命令仅包括本地测试、构建、静态检查、格式检查等非生产命令。删除、覆盖、生产连接、生产配置修改、发布、回滚、数据库变更等高风险操作必须禁止或走既有审批流程。

## 5.3 禁止行为


# 8. V2/V3 规划

## 8.1 V2：流程集成阶段

目标：将 AI 能力嵌入研发流程，形成半自动化协作。

### 重点能力

```text
1. 设计方案生成
2. 正式实现计划 Agent

6. 需求、代码、测试、发布数据打通

```

### 正式实现计划与 ADR 辅助

在 V1 轻量编码计划基础上，V2 建设正式实现计划 Agent。

该 Agent 基于需求、设计方案、代码仓库、历史缺陷、测试资产和架构上下文，生成任务级、文件级、测试级实现计划，并在需要时生成 ADR 草稿。

V2 能力包括：

```text
1. 自动分析代码仓库影响范围。
2. 生成任务级实现计划。
3. 生成文件级修改建议。
4. 生成测试计划。
5. 生成 Review 关注点。
6. 判断并生成 ADR 草稿。
7. 由技术负责人确认后更新 ADR。
```

ADR 草稿仍然必须人工确认：

```text
ADR 属于架构决策记录，AI 可以生成草稿，但不能自动决策、自动合入。
正式 ADR 必须由技术负责人或架构师确认。
```

# 10. 最终方案总结

本方案通过建设意健险研发团队专属 AI Harness 工程体系，将需求、代码、测试、发布、运维等研发上下文结构化沉淀，并通过 Skill / Agent 在关键节点执行生成、审查、验证和门禁，形成：

```text
上下文 → 计划 → 约束 → 执行 → 验证 → 反馈 → 修正 → 沉淀
```

的工程闭环，从而提升研发效率和质量。

本方案的关键调整包括：

1. 将 `AGENTS.md` 作为 AI Coding Agent 第一入口文件，放在项目根目录。
    
2. 将用户需要阅读和维护的知识、规范、上下文、ADR 放入目标项目 `docs/`，其中项目研发上下文统一命名为 `PROJ_CONTEXT.md`。
    
3. 将 Prompt、Skill、门禁规则、Schema、示例等工程化资产放入目标项目 `.ai-harness/`；本仓库通过 `harness-assets/` 可见源目录维护。
    
4. V1 以 OpenCode 试点落地为主，采用短入口、按需加载、结构化输出、Review 规则分层引用和少量核心 Skill。
    
5. V1 不做自动生成正式 PRD、完整设计方案、自动化平台集成、自动合并、自动发布或生产操作。
    
6. V1 聚焦需求评审、编码计划、TDD、辅助编码、代码审查、提交检查、核心场景测试、发布检查和节点门禁。
    
7. V1 建立统一风险分级、权限审计矩阵、资产回归评估和分层验收机制。
    
8. V2/V3 再推进正式实现计划、设计方案生成、ADR 草稿生成、流程集成、工程引导、多 Agent 编排、自动化测试和监控分析。
    

最小可执行落地路径：

```text
先建 AGENTS.md
    ↓
再建 docs/ 上下文和规范
    ↓
再建 Prompt、Skill、门禁、Schema 和示例资产
    ↓
先试点需求评审、编码计划、TDD、代码审查
    ↓
再接入提交检查、发布检查和节点门禁
    ↓
最后做流程集成和平台化运营
```

V1 最小资产包：

```text
1. AGENTS.md
2. docs/DOMAIN_KNOWLEDGE.md
3. docs/PROJ_CONTEXT.md
4. docs/PRD_CONTEXT.md
5. docs/CODING_RULES.md / TESTING_GUIDE.md / REVIEW_RULES.md / docs/review-rules/ / RELEASE_GUIDE.md
6. docs/DEFECT_CASES.md / docs/ADR/ / docs/workflows/
7. .ai-harness/prompts/
8. .ai-harness/skills/
9. .ai-harness/gates/
10. .ai-harness/schemas/
11. .ai-harness/examples/
12. 统一风险分级与场景矩阵
13. 工具权限与操作审计矩阵

```
