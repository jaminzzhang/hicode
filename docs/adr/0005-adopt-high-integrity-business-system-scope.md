# ADR 0005: Adopt High-Integrity Business System Scope

## Status

Accepted

## Context

hicode 过去以保险和金融核心系统作为默认服务对象，这建立了严格的金额、交易一致性、状态、幂等、权限、审计、隐私、监管和生产变更风险基线，但也把适用范围表达成了行业边界。支付、清结算、账务、计费、订单履约等类金融系统，以及其他具有同等级业务严谨性和一致性要求的系统，会遇到相同的研发风险，却可能因行业名称不匹配而无法自然采用 hicode。

## Decision

hicode 将“高严谨业务系统”作为统一目标项目术语。准入依据是系统是否对业务规则严谨性、逻辑边界、交易或数据一致性、可审计性有高要求，而不是是否属于某一个行业。

金融、保险、支付、清结算、账务、计费和订单履约等金融或类金融系统继续作为重点场景。保险特有规则保留为领域实例，不再定义 hicode 的整体边界。统一风险基线扩展为：领域业务逻辑严谨性、金额与关键数值精度、交易与数据一致性、状态流转、幂等、并发、权限、审计、隐私、适用监管与合规、生产变更、回滚和发布准入。

hicode 不因此变成面向所有软件项目的通用低风险工具。普通展示型、内容型或不具备上述高严谨特征的项目，不是默认服务对象；具体领域规则仍必须来自目标项目证据，不得用通用风险清单替代。

## Consequences

入口规则、术语表、Plugin 描述、Skill、Agent、模板、验收资产和离线评估文案统一使用“高严谨业务系统”口径。金融与保险关键词可以保留为重点场景和检索入口，但不得再表达为唯一目标范围。

风险判断将更多依赖系统特征与具体证据。新领域接入时必须补充领域术语、关键规则、不变量、状态和一致性要求；没有领域证据时应标注待确认，不能把金融或保险规则直接套用到其他领域。

## Alternatives Considered

1. 继续只服务保险与金融核心系统：领域聚焦清晰，但会排除风险形态相同的类金融和其他高严谨系统。
2. 改为面向所有业务系统：覆盖最广，但会弱化 hicode 的风险基线和差异化价值。
3. 以“高严谨业务系统”定义特征边界，并保留金融、保险及类金融重点场景：既拓宽适用范围，也保持严格治理标准。

选择方案 3。
