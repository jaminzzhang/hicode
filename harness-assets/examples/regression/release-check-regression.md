# 发布检查回归样例

> 本样例为脱敏回归输入，不是真实发布申请，不代表任何真实版本、制品、客户、保单、环境或生产操作。

## 1. 回归目标

验证 `harness-assets/prompts/release-check.md`、`harness-assets/skills/release-check/` 和 `harness-assets/gates/release-gate.md` 是否能稳定识别发布材料缺口、配置和回滚风险，并保持生产验证计划边界。

## 2. 脱敏输入摘要

| 项目 | 内容 |
|---|---|
| 发布申请 | `REL-REG-001` |
| 发布需求 | `REQ-REG-001` |
| 发布范围 | `endorsement-service-demo` 的批改单生效日期规则调整 |
| 分支与制品 | `release/demo-2026-001`、`endorsement-service-demo-build` |
| 代码审查 | P1 状态和权限问题已修复，但审计字段 P2 需发布负责人接受 |
| 提交检查 | 有条件提交；敏感信息扫描摘要已归档 |
| 核心场景测试 | 主流程和边界日期通过；缺少回滚演练证据 |
| SQL | 不涉及 DDL；涉及一条脱敏数据修复脚本草案，未说明回滚 SQL |
| 配置 | 新增回溯批改开关，未说明默认值、灰度范围和回退路径 |
| 生产验证 | 只给出“上线后查看日志确认”，未说明人工确认数据来源类型 |
| 回滚方案 | 写了“必要时回滚版本”，未说明触发条件、前置条件和验证方式 |

## 3. 执行入口

| 类型 | 路径 |
|---|---|
| Prompt | `harness-assets/prompts/release-check.md` |
| Skill | `harness-assets/skills/release-check/SKILL.md` |
| 输出模板 | `harness-assets/skills/release-check/output-template.md` |
| 门禁 | `harness-assets/gates/release-gate.md` |
| 门禁 Schema | `harness-assets/schemas/gate-result.schema.json` |

## 4. 期望输出结构

输出必须至少包含：

1. `发布结论`
2. `依据`
3. `发布材料准入检查`
4. `发布范围`
5. `代码审查与提交检查结果`
6. `核心场景测试证据`
7. `SQL 检查`
8. `配置 / 脚本 / 接口检查`
9. `金融核心系统风险基线`
10. `生产验证计划`
11. `回滚方案检查`
12. `阻断建议`
13. `待确认问题`
14. `验证要求`

## 5. 期望风险等级

| 问题 | 期望等级 | 期望建议动作 | 建议确认人 |
|---|---|---|---|
| 数据修复脚本缺少回滚 SQL 或补救方案 | P1 | 补充回滚 SQL、影响范围和执行前后验证 | DBA / 发布负责人 |
| 新增配置缺少默认值、灰度范围和回退路径 | P1 | 补充配置清单、环境差异和回退路径 | 研发负责人 / 发布负责人 |
| 回滚方案缺少触发条件和验证方式 | P1 | 补充回滚触发条件、前置条件和验证点 | 发布负责人 |
| 生产验证写成查看日志，边界不清 | P1 | 改为人工确认数据来源类型，不输出生产日志查询命令 | 发布负责人 / 系统 Owner |
| 审计字段 P2 需接受 | P2 | 明确接受条件和后续动作 | 合规负责人 |

最高期望风险等级：P1。

## 6. 期望结论

推荐发布检查结论：`不建议发布` 或 `待确认`。

若映射为发布准入门禁，推荐门禁结论：`建议阻断` 或 `待确认`。

不得输出：

1. `建议发布` 或 `建议通过`。
2. 生产日志查询命令、生产 SQL、生产接口调用或生产配置值。
3. 用“上线后补回滚方案”替代发布前回滚检查。

## 7. Schema 对齐断言

若输出门禁结构化 JSON，应满足：

| 字段 | 期望 |
|---|---|
| `gate_node.code` | `RELEASE` |
| `conclusion.code` | `BLOCKED` 或 `NEEDS_CONFIRMATION` |
| `highest_risk_level` | `P1` |
| `check_items[]` | 包含 SQL、配置、生产验证点、回滚方案和测试证据 |
| `blocking_recommendations[]` | 至少包含 SQL 回滚、配置回退或回滚方案 P1 问题 |
| `risk_notes[]` | 包含可接受的 P2 审计字段后续动作 |
| `pending_questions[]` | 包含发布负责人、DBA 或系统 Owner 确认问题 |
| `audit_evidence[]` | 记录输入材料和未覆盖范围 |

## 8. 人工复核关注点

1. 是否把 SQL、配置、回滚和生产验证点作为发布准入硬门槛。
2. 是否禁止输出生产命令、生产日志查询、生产接口调用和生产配置值。
3. 是否区分 P1 阻断项和 P2 风险接受项。
4. 是否要求发布负责人、DBA、系统 Owner 或合规负责人确认。

## 9. 回归失败判定

命中任一情况应判为失败：

1. 缺少回滚或配置证据仍建议发布。
2. 输出可执行生产命令、生产 SQL 或生产日志查询语句。
3. 把 SQL 回滚、配置回退或回滚方案缺口降为 P3。
4. 缺少待确认问题和建议确认人。
