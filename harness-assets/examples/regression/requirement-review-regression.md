# 需求评审回归样例

> 本样例为脱敏回归输入，不是真实需求，不代表任何真实产品、客户、保单或系统行为。

## 1. 回归目标

验证 `harness-assets/prompts/requirement-review.md` 和 `harness-assets/skills/requirement-review/` 是否能在需求信息不完整时稳定输出：

1. 有条件或待确认结论。
2. P1 级业务规则、金额和状态流转风险。
3. 具体待确认问题和建议确认人。
4. `docs/PRD_CONTEXT.md`、领域知识或缺陷案例更新建议。

## 2. 脱敏输入摘要

| 项目 | 内容 |
|---|---|
| 需求编号 | `REQ-REG-001` |
| 需求名称 | 批改单生效日期允许回溯 |
| 业务背景 | 为支持指定批改原因，允许批改单生效日期选择保单起保日至申请日之间的日期 |
| 已知规则 | 生效日期不得早于保单起保日；不得晚于申请日 |
| 缺失规则 | 未说明已收付费、已退保、已终止、已满期、已发生理赔前置标记时是否允许回溯 |
| 金额影响 | 未说明回溯后保费重算、退补费、舍入和交易一致性规则 |
| 审计要求 | 仅写“需要留痕”，未说明审计字段、回溯原因和操作来源 |
| 验收标准 | 只有主流程，没有边界日期、异常状态和权限校验 |

## 3. 执行入口

| 类型 | 路径 |
|---|---|
| Prompt | `harness-assets/prompts/requirement-review.md` |
| Skill | `harness-assets/skills/requirement-review/SKILL.md` |
| 输出模板 | `harness-assets/skills/requirement-review/output-template.md` |

## 4. 期望输出结构

输出必须至少包含：

1. `评审结论`
2. `依据`
3. `完整性检查`
4. `一致性与明确性检查`
5. `风险清单`
6. `业务规则缺失`
7. `边界与异常场景缺失`
8. `待确认问题`
9. `验证要求`
10. `上下文更新建议`

## 5. 期望风险等级

| 风险 | 期望等级 | 期望建议动作 | 建议确认人 |
|---|---|---|---|
| 回溯日期影响责任期间和保费重算规则缺失 | P1 | 补充责任期间、保费重算、退补费和舍入规则 | 业务负责人 / 财务负责人 |
| 终态保单是否允许批改不明确 | P1 | 补充状态矩阵和禁止状态 | 核心系统 Owner |
| 理赔前置状态是否限制批改不明确 | P1 | 明确理赔前置标记和批改关系 | 理赔或核心系统 Owner |
| 审计留痕字段不明确 | P2 | 补充回溯原因、操作来源、操作人和时间字段 | 合规负责人 |
| 权限边界不明确 | P2 | 明确哪些角色或渠道可选择回溯日期 | 研发负责人 / 安全负责人 |

最高期望风险等级：P1。

## 6. 期望结论

推荐结论：`有条件通过` 或 `待确认`。

不得输出：

1. `通过` 且无条件进入开发。
2. 已补齐业务规则的表述。
3. 将保费重算、状态流转或理赔前置风险降为 P3。

## 7. 人工复核关注点

1. 输出是否识别金额、状态流转、理赔前置、权限和审计风险。
2. 待确认问题是否具体到业务规则、数据、接口、测试或负责人。
3. 是否明确缺少规则时不能进入编码计划或只能有条件进入。
4. 是否避免编造真实产品、客户、保单或生产规则。

## 8. Schema 对齐提示

本样例不要求输出 JSON，但若后续工具化提取结构化结果，应至少能映射：

| 字段 | 期望 |
|---|---|
| `highest_risk_level` | `P1` |
| `pending_questions[]` | 包含状态、金额、理赔前置和审计问题 |
| `recommended_actions[]` | 包含补规则、补验收标准和补测试关注点 |
| `context_update_suggestions[]` | 建议更新 `docs/PRD_CONTEXT.md` 或领域知识 |

## 9. 回归失败判定

命中任一情况应判为失败：

1. 输出无条件通过。
2. 遗漏金额、状态流转或理赔前置任一 P1 风险。
3. 编造未提供的业务规则。
4. 输出真实客户、生产数据、密钥或生产配置。
