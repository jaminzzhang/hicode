# 发布检查 Skill 示例案例

> 本示例为 P4-WP7 脱敏结构示例，不是真实试点结果，不代表任何真实客户、保单、系统、分支或发布记录。

## 1. 适用 Skill

`harness-assets/skills/release-check/SKILL.md`

## 2. 脱敏输入摘要

发布目标：批改单生效日期规则调整进入示例发布申请。

发布材料：

1. 发布版本：`DEMO-REL-2026-001`
2. 发布需求：`REQ-DEMO-2026-001`
3. 分支：`feature/demo-endorsement-date-rule` 合入 `release/demo-2026-001`
4. 制品：`endorsement-service-demo-build`
5. 代码审查报告：示例中 P0/P1 已关闭。
6. 提交检查报告：有条件提交，CI 链接已在示例中补齐。
7. 核心场景测试报告：有条件进入发布检查，接口测试和审计字段验收已补齐。
8. SQL/配置：不涉及数据库结构变更，不涉及生产配置。
9. 回滚方案：恢复原日期校验规则，保留审计记录。

## 3. 处理重点

1. 发布范围、需求清单、分支制品和测试证据入口是硬门槛。
2. 缺少核心场景测试报告时不得无条件建议发布。
3. 生产验证只能写计划和人工确认入口，不输出生产命令。
4. 不替代发布负责人最终审批。

| 审计事件异常 | 回溯原因无法追溯 | 暂停回溯批改入口并回滚制品 | 系统 Owner 确认影响 | 有条件通过 |

## 5. 人工复核关注点

1. 发布范围、分支、制品和需求清单是否真实一致。
2. 核心场景测试报告是否由测试负责人确认。
3. 生产验证计划是否只有人工确认口径，没有可执行生产命令。
4. 回滚方案是否由发布负责人确认。

## 6. 禁止误用说明

1. 不得把本示例当作真实发布审批。
2. 不得输出或执行生产命令、生产 SQL、生产日志查询或生产接口调用。
3. 不得使用真实客户、保单、证件、手机号、银行卡或地址信息。
