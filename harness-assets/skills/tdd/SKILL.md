# TDD Skill

## 1. 定位

本 Skill 用于编码前和编码中的测试先行设计，负责把需求上下文、编码计划、项目测试规范和代码上下文转化为可执行的 TDD 循环：一个可观察行为、一个测试、一个最小实现、一次验证、一次必要重构。

本 Skill 参考 TDD 的 RED-GREEN-REFACTOR 纪律：测试应验证公开接口和可观察行为，不测试私有实现细节；禁止先批量写完所有测试再批量实现；禁止在 RED 状态重构。

本 Skill 不替代测试负责人、研发负责人或模块 Owner 的最终判断，不访问真实数据库、真实外部系统、生产配置或未脱敏客户数据。

## 2. 适用场景

使用本 Skill 的场景：

1. 编码计划已形成，准备进入实现、修复或重构前，需要明确测试先行证据。
2. 需求涉及保险核心业务逻辑、金额、日期、状态、交易一致性、幂等、权限、审计、隐私、监管、外部系统或历史缺陷。
3. 需要为辅助编码 Skill 提供测试场景、Given-When-Then、Mock、断言、测试数据和 RED-GREEN-REFACTOR 记录。
4. 需要补充或修正测试，以保护已有行为、缺陷复现或重构行为不变。
5. 用户明确要求使用 TDD、测试先行或 red-green-refactor。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 需求目标、业务规则、影响范围或 P0/P1 待确认问题尚未关闭；应回到 `requirement-review` 或 `coding-plan` Skill。
2. 当前任务是纯只读解释，不修改代码、测试、配置、脚本或行为。
3. 用户要求跳过测试、先实现再补测试、删除测试、降低断言或隐藏测试风险。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
5. 用户要求连接生产、执行数据库变更、自动合并、自动提交、自动发布或修改生产配置。

## 4. 必读上下文

执行前按需读取：

1. `AGENTS.md`
2. `docs/workflows/tdd.md`
3. `.ai-harness/prompts/tdd.md`
4. `docs/PRD_CONTEXT.md`
5. `docs/PROJ_CONTEXT.md`
6. `docs/CODING_RULES.md`
7. `docs/TESTING_GUIDE.md`
8. `docs/DOMAIN_KNOWLEDGE.md`
9. `docs/DEFECT_CASES.md`
10. 编码计划或 `coding-plan` Skill 输出
11. 相关代码、公开接口、已有测试、测试框架和测试命令，如已有
12. 本目录的 `output-template.md`

只读取当前任务必要上下文。缺少上下文时，输出缺口、影响和补齐动作，不补编业务规则、类、表、接口、测试结果或验收结论。

## 5. 输入材料

要求用户或调用方提供：

1. 需求编号、需求摘要或 `docs/PRD_CONTEXT.md`。
2. 编码计划，包括改动目标、影响范围、风险和建议实现步骤。
3. 相关模块、类、公开接口、方法、表、配置或外部依赖范围。
4. 已有测试、测试框架、测试命令、失败信息、覆盖率摘要或 CI 摘要，如已有。
5. 历史缺陷、Review 问题、核心场景测试要求或发布限制，如已有。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否满足 TDD 准入：编码计划存在、上下文清晰门槛已满足、P0/P1 待确认问题已关闭、影响范围足以设计测试。
3. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止并要求先脱敏或改走人工安全流程。
4. 读取必需上下文，列出已读取材料、缺失材料和缺失影响。
5. 识别需要保护的可观察行为，优先覆盖核心业务路径、高风险金融场景、历史缺陷和复杂逻辑。
6. 设计垂直切片 TDD 循环，不做横向切片：
   - 每轮只选一个可观察行为。
   - 先写一个通过公开接口验证行为的测试。
   - 确认 RED：测试因目标行为缺失或缺陷复现而失败。
   - 只写足够通过当前测试的最小实现。
   - 确认 GREEN：当前测试和相关回归通过。
   - 只在 GREEN 状态重构，并在每次重构后重新运行相关测试。
7. 输出测试场景、Given-When-Then、边界、异常、Mock、测试数据、断言规则和测试代码草案。
8. 记录受限命令执行建议或实际执行结果。未执行时说明原因、风险和人工验证动作。
9. 输出风险、待确认问题、验证要求和上下文更新建议。

## 7. 输出

必须输出：

1. 结论：建议进入辅助编码 / 有条件进入辅助编码 / 不建议进入辅助编码 / 待确认。
2. 依据和输入范围。
3. TDD 准入检查。
4. 测试目标与范围。
5. 行为清单和优先级。
6. RED-GREEN-REFACTOR 循环计划。
7. 测试场景清单。
8. Given-When-Then 用例。
9. 边界场景和异常场景。
10. Mock 策略。
11. 测试数据和脱敏要求。
12. 断言规则。
13. 测试代码草案或示例片段。
14. 受限命令执行记录。
15. 风险、待确认问题、验证要求和上下文更新建议。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的 TDD Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 8. 质量标准

输出必须满足：

1. 测试验证可观察行为和公开接口，不测试私有方法、内部协作者或实现细节。
2. 每个关键测试建议可追溯到需求、编码计划、项目上下文、测试规范、代码证据或历史缺陷。
3. TDD 循环必须体现 RED-GREEN-REFACTOR，且一轮只覆盖一个行为。
4. 不批量写完所有测试再实现，不输出投机性未来功能。
5. 核心逻辑必须覆盖正常、边界、异常和历史缺陷场景。
6. 金额断言必须使用精确数值口径；日期测试必须覆盖边界、时区和特殊日期；外部依赖必须 Mock。
7. 测试代码草案必须标注未执行前不得视为已通过。
8. 风险按 P0/P1/P2/P3 分级，并说明影响、建议动作和建议确认人。

## 9. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不访问真实数据库、真实外部系统、生产配置、生产数据或未脱敏客户数据。
6. 不直接操作生产环境，不直接给出生产操作命令。
7. 不自动提交，不自动合并 MR / PR，不自动发布，不自动修改生产配置。
8. 不删除测试，不降低断言，不为了通过检查而绕过测试或隐藏风险。
9. 受限命令仅限本地单元测试、构建、静态检查或格式检查；不得执行数据库变更、发布、回滚、生产连接或删除未确认文件。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 10. 验证要求

完成后必须说明：

1. 实际读取的上下文。
2. TDD 准入是否满足。
3. 每个测试行为的依据来源。
4. RED、GREEN 和重构验证命令是否实际执行；如未执行，说明原因和风险。
5. 是否存在测试框架、代码上下文、Mock 依赖、测试数据或负责人确认缺口。
6. 是否需要更新目标项目 `docs/`、目标项目 `.ai-harness/` 或本仓库 `harness-assets/` 源资产。

## 11. 上下文更新

如发现稳定测试规则、历史缺陷、模块测试策略、Mock 约束、断言规范或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库、测试规范或 ADR 前，必须由对应负责人确认。
