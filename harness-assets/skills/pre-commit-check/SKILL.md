# 提交检查 Skill

## 1. 定位

本 Skill 用于 Git 提交前、创建 MR/PR 前或进入提测前的开发侧材料准入检查，负责汇总代码审查结果、分支和需求关联、测试构建扫描、SQL/配置/脚本、敏感信息、MR/PR 描述草稿和阻断建议。

本 Skill 默认消费 `code-review` Skill 的双轴审查报告，但不重复做深度代码审查。缺少代码审查结果时，可以继续检查提交材料缺口，但不得给出无条件建议提交。

本 Skill 不替代开发人员、Reviewer、研发负责人、测试负责人、CI/CD、门禁系统或发布负责人的最终判断，不自动提交、自动推送、自动合并或自动发布。

## 2. 适用场景

使用本 Skill 的场景：

1. 准备 Git 提交、创建 MR/PR 或进入提测前，需要检查提交材料是否齐备。
2. 已有代码审查报告，需要确认 P0/P1/P2/P3 问题、规范轴和需求轴结论是否允许进入提交流程。
3. 需要检查分支、目标分支、关联需求、测试构建扫描、覆盖率、敏感信息、SQL、配置、脚本、接口文档、发布影响和回滚关注点。
4. 需要生成 MR/PR 描述草稿、风险提示、待处理问题和上下文更新建议。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 当前目标是深度审查代码逻辑、架构、安全或测试覆盖细节；应先使用 `code-review` Skill。
2. 没有 diff、分支信息、变更范围或关联需求，无法判断提交范围。
3. 用户要求自动提交、自动推送、自动合并、自动发布、修改代码、修改生产配置或执行生产操作。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
5. 用户要求删除测试、降低断言、跳过 Review、隐藏 P0/P1 风险或绕过门禁。

执行前按需读取：

1. `AGENTS.md`
2. `CONTEXT.md`
3. `docs/workflows/pre-commit-check.md`
4. `.hicode/prompts/pre-commit-check.md`
5. `docs/PRD_CONTEXT.md`
6. `docs/CODING_RULES.md`
7. `docs/TESTING_GUIDE.md`
8. `docs/REVIEW_RULES.md`
9. `docs/RELEASE_GUIDE.md`，涉及 SQL、配置、脚本或发布影响时读取
10. `docs/DEFECT_CASES.md`
11. 编码计划、TDD 测试设计报告、辅助编码结果和代码审查报告，如已有
12. 当前分支、目标分支、关联需求、变更范围和 diff
13. 单测、构建、静态扫描、覆盖率、敏感信息扫描或 CI 结果，如已有
14. SQL、配置、脚本、接口文档、发布说明、回滚说明和验证点清单，如有
15. 本目录的 `output-template.md`

只读取当前提交检查必要上下文。缺少上下文时，输出缺口和影响，不补编需求、业务规则、测试结果、扫描结果、Review 结论或提交结论。

## 5. 输入材料

要求用户或调用方提供：

1. 当前分支、目标分支、关联需求/故事链接和 MR/PR 目标。
2. `git diff`、MR/PR diff、补丁文件、变更文件列表或当前工作区 diff。
3. 代码审查报告，包括规范轴、需求轴、P0/P1/P2/P3 问题和是否建议进入提交检查。
4. 单元测试、构建、静态扫描、覆盖率、格式检查、敏感信息扫描或 CI 结果。
5. SQL、配置、脚本、接口文档、发布说明、回滚说明或验证点清单，如有。
6. 已知豁免、待确认问题、负责人确认、风险接受说明或未执行命令原因，如有。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止检查并要求先脱敏或改走人工安全流程。
3. 检查提交范围：当前分支、目标分支、关联需求、diff 范围、变更文件和 MR/PR 目标是否明确。
4. 检查代码审查证据：
   - 是否提供 `code-review` 报告。
   - 是否包含规范轴和需求轴结果。
   - 是否存在未关闭 P0/P1。
   - 是否存在负责人豁免或待确认问题。
   - 缺少代码审查结果时，输出审查证据缺口，不给无条件建议提交。
5. 检查需求关联、编码计划、TDD、辅助编码结果和测试证据是否可追溯。
6. 检查测试、构建、静态扫描、覆盖率、格式检查、敏感信息扫描和 CI 结果是否齐备。
7. 检查 SQL、配置、脚本、接口文档、发布说明、回滚说明和验证点是否齐备。
8. 在具备条件时执行受限命令，并记录命令、范围、结果、失败摘要和验证边界。
9. 输出建议提交 / 有条件提交 / 不建议提交 / 待确认、阻断建议、待处理问题、MR/PR 描述草稿、验证要求和上下文更新建议。

## 7. 受限命令边界

可执行的受限命令：

1. `git status`
2. `git diff --stat`
3. `git diff`
4. `git log`
5. 本地单元测试
6. 构建
7. 静态检查
8. 格式检查
9. 敏感信息扫描

执行前提：

1. 命令来自项目已有脚本、文档、工具配置或用户确认。
2. 命令不连接生产、不读取密钥、不处理未脱敏生产数据。
3. 命令不执行数据库变更、发布、回滚、提交、推送、合并或删除未确认文件。

敏感信息扫描命中时，不输出完整敏感值，必须掩码并建议人工处理。

## 8. 输出

必须输出：

1. 检查结论：建议提交 / 有条件提交 / 不建议提交 / 待确认。
2. 依据。
3. 提交范围。
4. 代码审查报告输入检查。
5. 规范轴/需求轴未关闭问题统计。
6. 需求关联检查。
7. 分支与变更检查。
8. 测试、构建与扫描结果。
9. SQL / 配置 / 脚本 / 接口文档检查。
10. 敏感信息检查。
11. 受限命令执行记录。
12. 阻断建议。
13. 待处理问题。
14. MR / PR 描述草稿。
15. 验证要求。
16. 上下文更新建议。
17. 下一步建议。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的提交检查 Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 9. 质量标准

输出必须满足：

1. 结论只能使用建议提交 / 有条件提交 / 不建议提交 / 待确认，不输出允许提交、批准提交或已通过最终门禁。
2. 缺少代码审查报告时，不得给出无条件建议提交。
3. 存在未关闭 P0 Review 问题、敏感信息、密钥、未脱敏生产数据、自动合并/发布或生产越权时，必须给出不建议提交和阻断建议。
4. 存在未关闭 P1 问题时，必须建议修复或负责人明确豁免。
5. SQL、配置、脚本、接口文档和发布影响必须说明范围、回滚或验证关注点。
6. 每个检查结论都能追溯到输入、工具结果、代码审查报告或负责人确认。
7. MR/PR 描述只作为草稿，不代表正式提交、推送、合并或发布动作。

## 10. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不自动提交，不自动推送，不自动合并 MR / PR，不自动发布，不自动修改生产配置。
6. 不直接操作生产环境，不直接给出生产操作命令。
7. 不修改代码，不删除测试，不降低断言，不隐藏 P0/P1 风险。
8. 不执行数据库变更、生产连接、发布、回滚或删除未确认文件。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 11. 验证要求

完成后必须说明：

1. 实际读取的上下文、提交范围和代码审查报告。
2. 是否取得当前分支、目标分支、关联需求和 diff。
3. 代码审查报告是否包含规范轴、需求轴和 P0/P1/P2/P3 结果。
4. 实际执行的受限命令、结果、失败摘要和未执行原因。
5. 是否存在测试、构建、扫描、覆盖率、SQL、配置、脚本、接口文档、发布或回滚证据缺口。
6. 是否需要开发人员、Reviewer、研发负责人、测试负责人、安全负责人、发布负责人或业务负责人确认。
7. 是否需要更新目标项目 `docs/`、目标项目 `.hicode/`、目标项目 `.hicode/gates/` 或本仓库 `harness-assets/` 源资产。

## 12. 上下文更新

如发现稳定提交检查规则、缺陷模式、Review 规则、测试规范、发布材料要求、权限边界或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库、Review 规则、发布指南或门禁资产前，必须由对应负责人确认。
