# 需求评审 Skill

## 1. 定位

本 Skill 用于需求进入编码计划前的研发视角评审，负责路由需求上下文、执行输入准入检查、引用需求评审 Prompt、记录只读验证动作，并使用本目录输出模板生成需求评审报告。

本 Skill 不生成正式 PRD，不替代业务负责人、产品/BA、研发负责人、测试负责人或架构师对需求范围、业务规则、验收口径和进入开发的最终判断。

## 2. 适用场景

使用本 Skill 的场景：

1. 新需求、变更需求或缺陷修复需求准备进入编码计划前，需要检查完整性、一致性、明确性、可开发性和可测试性。
2. 用户提供了需求摘要、PRD、故事链接、会议纪要、原型、流程图或 `docs/PRD_CONTEXT.md`，需要输出澄清问题、风险、测试关注点和上下文更新建议。
3. 需求涉及保险/金融核心系统风险，需要提前检查保险核心业务逻辑严谨性、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
4. 用户明确要求使用需求评审 Skill，且输入材料已经脱敏并不违反安全规则。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 目标是生成编码任务、实现方案或 ADR 判断；应使用 `coding-plan` Skill 或 `.ai-harness/prompts/coding-plan.md`。
2. 目标是设计测试用例、生成测试草案或执行 TDD；应使用 `tdd` Skill 或 `.ai-harness/prompts/tdd.md`。
3. 目标是代码 diff 审查、提交检查、核心场景测试或发布检查；应使用对应 Skill 或 Prompt。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
5. 用户要求本 Skill 修改业务代码、自动通过需求、自动进入开发、自动合并、自动发布或操作生产环境。

## 4. 必读上下文

执行前按需读取：

1. `AGENTS.md`
2. `docs/workflows/requirement-review.md`
3. `.ai-harness/prompts/requirement-review.md`
4. `docs/DOMAIN_KNOWLEDGE.md`
5. `docs/PROJ_CONTEXT.md`
6. `docs/PRD_CONTEXT.md`，如已有
7. `docs/DEFECT_CASES.md`
8. 本目录的 `output-template.md`

只读取当前需求必要上下文。缺少上下文时，输出缺口、影响和建议补充材料，不补编业务规则、系统行为或验收标准。

## 5. 输入材料

要求用户或调用方提供：

1. 需求名称、需求编号、故事链接或版本标识。
2. 需求文档、需求摘要、流程说明、页面原型、接口说明、会议纪要或用户故事。
3. 已知业务规则、验收标准、范围内/范围外说明。
4. 涉及系统、模块、接口、数据、配置、批处理、权限或外部依赖范围，如已知。
5. 已脱敏的历史需求、历史缺陷、相似需求或负责人确认说明，如有。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止评审并要求先脱敏或改走人工安全流程。
3. 读取必需上下文，列出已读取材料、缺失材料和缺失影响。
4. 按 `.ai-harness/prompts/requirement-review.md` 的检查维度执行需求评审，覆盖完整性、一致性、明确性、可开发性、可测试性、业务风险、系统影响、非功能要求和历史缺陷匹配。
5. 对保险/金融核心系统风险按 P0/P1/P2/P3 分级；P0 澄清问题未关闭时，不得建议直接进入开发。
6. 使用本目录 `output-template.md` 生成需求评审报告。
7. 输出 `docs/PRD_CONTEXT.md`、`docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md`、`docs/DEFECT_CASES.md` 或 Harness 资产的更新建议；正式沉淀前必须由对应负责人确认。
8. 记录验证动作、未执行命令原因和证据缺口。本 Skill 默认为 L0 只读分析，不需要执行命令。

## 7. 输出

必须输出：

1. 评审结论：通过 / 有条件通过 / 不建议进入开发 / 待确认。
2. 依据和输入范围。
3. 需求摘要。
4. 完整性检查。
5. 一致性与明确性检查。
6. 风险清单。
7. 业务规则缺失、边界与异常场景缺失。
8. 影响范围初判。
9. 测试关注点。
10. 待确认问题。
11. 验证要求和证据缺口。
12. 上下文更新建议。
13. 后续建议动作。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的需求评审 Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 8. 质量标准

输出必须满足：

1. 结论可追溯到需求输入、上下文文档、历史缺陷、工具结果或负责人确认。
2. 覆盖完整性、一致性、明确性、可开发性、可测试性、业务风险、系统影响、非功能要求和历史缺陷匹配。
3. 风险按 P0/P1/P2/P3 分级，并说明影响、建议动作和建议确认人。
4. 不确定内容标注 `需确认` 或 `待确认`。
5. 澄清问题具体到规则、流程、数据、接口、测试、权限、审计、发布或负责人。
6. 输出可作为 `coding-plan` Skill 的输入，但不得替代编码计划。
7. 不复制需求评审 Prompt 全文，不维护第二套需求评审规则。

## 9. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不直接操作生产环境，不直接给出生产操作命令。
6. 不自动通过需求，不自动进入开发，不自动合并 MR / PR，不自动发布，不自动修改生产配置。
7. 不为了推动需求进入开发而隐藏 P0/P1 风险、降低风险等级或删减待确认问题。
8. 不把一次有效输出描述为普适可靠结论，必须说明验证边界和证据缺口。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 10. 验证要求

完成后必须说明：

1. 实际读取的上下文。
2. 是否读取了需求原文、摘要或路径。
3. 是否匹配过历史缺陷或领域高风险主题。
4. 是否存在无法判断的需求范围、规则、系统影响或测试证据缺口。
5. 是否需要业务负责人、产品/BA、研发负责人、测试负责人、安全负责人或架构师确认。
6. 是否需要更新目标项目 `docs/`、目标项目 `.ai-harness/` 或本仓库 `harness-assets/` 源资产。

## 11. 上下文更新

如发现稳定业务规则、术语、历史缺陷、模块影响、接口约束、ADR 决策或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库或 ADR 前，必须由对应负责人确认。
