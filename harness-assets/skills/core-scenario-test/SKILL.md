# 核心场景测试 Skill

## 1. 定位

本 Skill 用于测试阶段、提测前或发布检查前的核心场景测试证据产出，负责按证据分层识别必测场景、回归场景、边界、异常、测试数据、Mock/环境约束、自动化可行性、受限命令执行记录和测试缺口风险。

本 Skill 是测试侧证据产出节点，不是发布前证据汇总节点。它的输出可作为 `release-check` Skill 的重要输入，但不替代测试负责人、研发负责人、模块 Owner 或发布负责人的最终判断。

## 2. 适用场景

使用本 Skill 的场景：

1. 需求进入测试阶段，需要从需求和变更证据提炼核心业务测试场景。
2. 变更涉及保险核心业务逻辑、金额、日期、交易一致性、状态流转、幂等、权限、审计、隐私、监管、外部系统或历史缺陷。
3. 发布检查前需要整理核心场景测试报告、测试证据、缺口风险和补测建议。
4. 已有测试用例、测试报告或测试命令，需要核对是否覆盖本次需求和核心回归范围。
5. 用户要求生成必测场景、回归场景、边界异常测试、Mock 策略或测试侧发布检查输入证据。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 当前目标是编码前单元测试设计或 TDD 循环；应使用 `tdd` Skill。
2. 当前目标是发布范围、制品、SQL、配置、缺陷、回滚和生产验证点汇总；应使用 `release-check` Skill。
3. 缺少需求层或变更层证据，无法判断本次测试影响范围；只能输出待确认和补证据动作。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
5. 用户要求连接生产、访问真实外部系统、执行数据库变更、发布、回滚、删除未确认文件或绕过测试负责人确认。

## 4. 必读上下文

执行前按需读取：

1. `AGENTS.md`
2. `CONTEXT.md`
3. `docs/workflows/core-scenario-test.md`
4. `.ai-harness/prompts/core-scenario-test.md`
5. `docs/PRD_CONTEXT.md`
6. `docs/PROJ_CONTEXT.md`
7. `docs/DOMAIN_KNOWLEDGE.md`
8. `docs/TESTING_GUIDE.md`
9. `docs/DEFECT_CASES.md`
10. `docs/CODING_RULES.md`，涉及代码、配置、日志、权限或敏感信息时读取
11. `docs/RELEASE_GUIDE.md`，输出将用于发布检查时读取
12. 编码计划、TDD 测试设计报告、辅助编码结果、代码审查报告、提交检查报告，如已有
13. 代码 diff、MR/PR diff、补丁文件、变更文件列表或关键代码片段
14. 已有测试用例、测试报告、测试命令、覆盖率结果、CI 结果或缺陷清单，如已有
15. 本目录的 `output-template.md`

只读取当前测试分析必要上下文。缺少上下文时，输出缺口和影响，不补编需求、业务规则、核心场景、类、表、接口、测试结果或覆盖率结论。

## 5. 输入材料

要求用户或调用方提供：

1. 需求编号、需求摘要、验收标准或 `docs/PRD_CONTEXT.md`。
2. 本次变更范围，包括模块、接口、表、配置、外部依赖、分支、diff 或 MR/PR 信息。
3. 编码计划、TDD 报告、辅助编码结果、代码审查报告或提交检查报告，如已有。
4. 已有测试用例、测试报告、测试命令、覆盖率、CI 结果或手工测试记录，如已有。
5. 历史缺陷、相似问题、回归范围、发布限制或负责人风险接受说明，如有。
6. 可在目标项目本地非生产环境执行的受限测试命令，如需 Agent 执行。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 准入规则

进入可建议状态前必须满足：

1. 有需求层证据，说明本次要验证什么。
2. 有变更层证据，说明本次实际改了什么。
3. 需求层或变更层缺失时，只能输出 `待确认`，不得建议进入发布检查。
4. 测试层证据缺失时，可以输出场景和补测建议，但必须按缺口严重程度给出有条件进入发布检查或不建议进入发布检查。
5. P0/P1 测试缺口、核心业务场景未覆盖、测试失败未解释、敏感数据或真实依赖风险存在时，必须给出不建议进入发布检查和阻断建议。

## 7. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止分析并要求先脱敏或改走人工安全流程。
3. 按需求层、项目层、领域层、测试层、缺陷层和变更层整理证据，标注证据缺口和影响。
4. 执行需求层和变更层准入检查。
5. 识别本次需求影响业务场景，覆盖投保、批改、退保、核保、理赔前置、收付费、责任判断、状态流转、渠道交互等相关域。
6. 生成必测场景、建议回归场景、边界测试、异常测试、安全与隐私测试。
7. 明确测试数据要求、脱敏要求、Mock/环境约束和禁止事项。
8. 判断自动化可行性，区分单元测试、接口测试、集成测试、回归自动化和人工验证。
9. 在具备条件时执行受限测试命令；无法执行时说明原因、风险和人工验证动作。
10. 关联历史缺陷和防范规则。
11. 输出测试侧结论、风险、待确认问题、验证要求和上下文更新建议。

## 8. 受限命令边界

可执行的受限命令：

1. 本地单元测试。
2. 指定模块测试。
3. 回归测试。
4. 覆盖率。
5. 构建。
6. 静态检查。
7. 格式检查。

执行前提：

1. 命令来自项目已有脚本、文档、工具配置或用户确认。
2. 命令运行在目标项目本地非生产环境。
3. 命令不连接真实数据库、真实外部系统或生产环境。
4. 命令不读取密钥、不读取生产配置、不处理未脱敏生产数据。
5. 命令不执行数据库变更、发布、回滚、生产连接或删除未确认文件。

必须记录命令、范围、是否执行、结果、失败摘要或未执行原因、验证边界。

## 9. 输出

必须输出：

1. 测试侧结论：建议进入发布检查 / 有条件进入发布检查 / 不建议进入发布检查 / 待确认。
2. 依据与证据分层。
3. 需求层和变更层准入结果。
4. 本次需求摘要。
5. 变更范围。
6. 影响业务场景。
7. 必测场景。
8. 建议回归场景。
9. 边界测试。
10. 异常测试。
11. 安全与隐私测试。
12. 测试数据要求。
13. Mock 与环境约束。
14. 自动化可行性。
15. 历史缺陷关联。
16. 受限命令执行记录。
17. 测试缺口与风险。
18. 待确认问题。
19. 验证要求。
20. 上下文更新建议。
21. 下一步建议。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的核心场景测试 Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 10. 质量标准

输出必须满足：

1. 每个场景和结论都能追溯到需求、项目、领域、测试、缺陷或变更证据。
2. 需求层或变更层缺失时，不得输出建议进入发布检查或有条件进入发布检查。
3. 必测场景覆盖本次需求直接影响范围、主流程、关键业务规则和验收标准。
4. 回归场景覆盖相关核心链路、相邻业务域、相似历史缺陷和高风险依赖。
5. 金融核心系统风险覆盖保险核心业务逻辑、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
6. 测试数据必须虚构或脱敏，外部依赖必须 Mock 或说明人工验证约束。
7. 未实际执行的测试不得写成已通过。
8. 输出能作为 `release-check` Skill 的输入证据，但不替代测试负责人最终确认。

## 11. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不访问真实数据库、真实外部系统、生产配置、生产数据或未脱敏客户数据。
6. 不直接操作生产环境，不直接给出生产操作命令。
7. 不执行数据库变更、发布、回滚、生产连接或删除未确认文件。
8. 不为了推动发布而隐藏测试缺口、降低断言、删除测试或绕过负责人确认。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 12. 验证要求

完成后必须说明：

1. 实际读取的上下文。
2. 需求层、项目层、领域层、测试层、缺陷层和变更层证据是否齐备。
3. 需求层和变更层准入是否满足。
4. 实际执行的受限命令、结果、失败摘要和未执行原因。
5. 是否存在测试数据、Mock、环境、自动化、核心场景或负责人确认缺口。
6. 是否需要测试负责人、研发负责人、模块 Owner、安全负责人或发布负责人确认。
7. 是否需要更新目标项目 `docs/`、目标项目 `.ai-harness/`、目标项目 `.ai-harness/gates/` 或本仓库 `harness-assets/` 源资产。

## 13. 上下文更新

如发现稳定测试规则、核心场景、历史缺陷、Mock 约束、测试数据要求、发布检查证据缺口或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库、测试规范、发布指南或门禁资产前，必须由对应负责人确认。
