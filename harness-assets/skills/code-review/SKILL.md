# 代码审查 Skill

## 1. 定位

本 Skill 用于开发完成后、人工 Review 前或提交检查前的 AI 代码审查，负责固定审查范围、分离规范轴和需求轴、按 P0/P1/P2/P3 输出问题，并判断是否建议进入提交检查。

本 Skill 参考双轴 Review 纪律：规范轴检查变更是否符合已文档化标准，需求轴检查变更是否符合来源需求、PRD、编码计划和 TDD 证据。两轴结果必须分别呈现，再汇总为本仓库统一风险分级结论。

本 Skill 不替代人工 Reviewer、架构师、研发负责人、安全负责人、测试负责人或发布负责人的最终判断，不自动修改代码、不自动提交、不自动合并、不自动发布。

## 2. 适用场景

使用本 Skill 的场景：

1. 用户要求审查一个分支、PR、MR、patch、当前工作区 diff 或指定基准点以来的变更。
2. 开发完成后，需要在人工 Review 或提交检查前先识别质量、安全、测试、需求偏差和金融核心系统风险。
3. 变更涉及保险核心业务逻辑、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、SQL、配置、脚本、发布影响或历史缺陷。
4. 需要基于明确 diff 和来源规格，输出规范轴、需求轴、分级问题、测试缺口和是否建议进入提交检查。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 缺少 diff、补丁、固定基准点、MR/PR diff、当前工作区 diff 或变更文件列表，无法确定审查范围。
2. 当前目标是提交材料检查、生成 MR/PR 描述草稿或执行提交前命令；应使用 `pre-commit-check` Skill。
3. 当前目标是修复问题、重构或修改代码；应先回到 `coding-plan`、`tdd` 或 `coding-assistant` Skill。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
5. 用户要求自动修改代码、自动提交、自动合并、自动发布、修改生产配置或直接操作生产环境。

## 5. 输入材料

要求用户或调用方提供：

1. 固定基准点，例如分支、commit、tag、`main`、`HEAD~N`；未提供时先追问，不擅自假设。
2. 审查范围：`git diff <fixed-point>...HEAD`、MR/PR diff、补丁文件、当前工作区 diff 或变更文件列表。
3. 提交列表：优先使用 `git log <fixed-point>..HEAD --oneline`，如可取得。
4. 来源需求、PRD、需求编号、故事链接、Issue、编码计划、TDD 报告或辅助编码结果。
5. 单元测试、构建、静态扫描、覆盖率、敏感信息扫描或 CI 结果，如已有。
6. SQL、配置、脚本、接口、外部依赖、发布影响、负责人豁免或已知 Review 关注点，如有。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止审查并要求先脱敏或改走人工安全流程。
3. 固定审查范围：
   - 有固定基准点时，优先记录 `git diff <fixed-point>...HEAD`、`git diff --stat <fixed-point>...HEAD` 和 `git log <fixed-point>..HEAD --oneline`。
   - 无固定基准点时，先追问审查相对哪个分支、commit 或 `main`。
   - 只能取得 MR/PR diff、补丁或当前工作区 diff 时，标注范围边界和降级原因。
4. 识别需求轴来源，按顺序查找 Issue/MR/PRD 引用、用户提供路径、`docs/`、`specs/`、`.scratch/`、编码计划、TDD 报告和辅助编码结果。
5. 识别规范轴标准来源，读取入口规则、术语上下文、Review 总则、编码规范、测试规范、缺陷案例、按需 Review 细则、相关 ADR 和机器配置约束。
6. 执行规范轴审查：只基于已文档化标准和已配置工具约束检查违反项，区分硬性违规和判断性建议；工具可自动检查的内容只记录需运行工具或引用已有结果。
7. 执行需求轴审查：检查缺失需求、部分实现、实现偏差、范围蔓延和需求未要求的行为。找不到需求来源时，需求轴标注无可用需求来源，不编造需求一致性结论。
8. 将两轴结果汇总到 P0/P1/P2/P3 问题清单；严重程度按影响校准，不把风格或偏好问题升级为 P0/P1。
9. 检查测试覆盖、安全敏感信息、SQL/配置/脚本/发布影响和金融核心系统风险。
10. 输出是否建议进入提交检查、验证要求、待确认问题和上下文更新建议。

如果目标 Agent 支持并行子任务，可分别执行规范轴和需求轴；如不支持，也必须按两个独立章节、独立依据和独立发现输出。

## 7. 输出

必须输出：

1. 审查结论：通过 / 有条件通过 / 不建议提交 / 待确认。
2. 是否建议进入提交检查：是 / 否 / 有条件 / 待确认。
3. 固定基准点、diff 范围、提交列表和范围边界。
4. 标准来源与需求来源。
5. 规范轴审查结果。
6. 需求轴审查结果。
7. P0/P1/P2/P3 统一问题清单。
8. 测试覆盖问题。
9. 安全与敏感信息检查。
10. SQL / 配置 / 脚本 / 发布影响。
11. 验证要求。
12. 待确认问题。
13. 上下文更新建议。
14. 下一步建议。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的代码审查 Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 8. 质量标准

输出必须满足：

1. 审查范围明确；缺少固定基准点或完整 diff 时，不得给出无条件通过结论。
2. 规范轴和需求轴分别列出依据、发现和证据缺口，不混成泛泛评价。
3. 规范轴问题必须引用文档化标准、相关 ADR、Review 细则或工具约束；不得凭个人偏好输出高严重度问题。
4. 需求轴问题必须追溯到需求、PRD、Issue、编码计划、TDD 或负责人确认；找不到来源时标注降级。
5. 每个 P0/P1/P2 问题都包含文件/位置、问题说明、影响、依据、修复建议和责任角色。
6. 金融核心系统风险必须覆盖保险核心业务逻辑、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
7. 测试缺口必须具体到场景、类型、缺口和建议补充方式。
8. 输出能支持人工 Reviewer 判断是否进入提交检查，但不替代人工 Review 最终结论。

## 9. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不自动修改代码，不自动提交，不自动合并 MR / PR，不自动发布，不自动修改生产配置。
6. 不直接操作生产环境，不直接给出生产操作命令。
7. 不删除测试，不降低断言，不为了推动提交而隐藏 P0/P1 风险或跳过 Review。
8. 默认只读；除用户明确要求且命令属于安全的 diff、log、status 外，不运行测试构建命令。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 10. 验证要求

完成后必须说明：

1. 实际读取的上下文、标准来源、需求来源和 Review 细则。
2. 固定基准点、diff 命令、提交列表是否取得；如未取得，说明原因和范围边界。
3. 规范轴和需求轴是否完整执行；需求轴是否降级。
4. 是否存在无法判断的测试、CI、覆盖率、扫描、SQL、配置、脚本或发布影响证据缺口。
5. 是否需要人工 Reviewer、研发负责人、测试负责人、安全负责人、架构师、业务负责人或发布负责人确认。
6. 是否需要更新目标项目 `docs/`、目标项目 `.hicode/` 或本仓库 `harness-assets/` 源资产。

## 11. 上下文更新

如发现稳定 Review 规则、缺陷模式、编码规则、测试规则、ADR 决策、需求上下文或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库、Review 规则或 ADR 前，必须由对应负责人确认。
