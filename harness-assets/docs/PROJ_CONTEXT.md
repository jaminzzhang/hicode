# PROJ_CONTEXT

## 1. 文档定位

本文档描述目标项目的研发上下文，包括模块结构、关键流程、接口、数据、约束和历史风险。它不替代术语表、领域知识文档，也不记录单个需求的临时上下文。

- 维护对象：研发负责人、架构师、模块 Owner
- 更新时机：模块职责变化、核心流程变化、关键接口变化、重要技术约束变化后
- 主要使用者：编码计划、TDD、辅助编码、代码审查 Agent
- 使用边界：单需求信息写入 `docs/PRD_CONTEXT.md`，领域知识写入 `docs/DOMAIN_KNOWLEDGE.md`

### 3.1 大型代码库导航地图

当目标项目是多模块单体、Monorepo、多仓服务群或长期遗留系统时，维护本节，帮助 Agent 先定位再读取。

| 路径/仓库 | 层级 | 职责摘要 | 主要语言/框架 | 关键入口 | Owner | 注意事项 |
|---|---|---|---|---|---|---|

**Owner 要求**：每个模块必须填写 Owner 或标注"待指定"。整体 hicode 配置维护 DRI 应写入 `AGENTS.md` 项目定位表的"技术负责人"字段。Owner 缺位时，Agent 输出中应标注"本模块无确认 Owner，上下文更新建议未获确认，需项目负责人复核"。

**子目录上下文文件**：当目录超过 3 层或模块超过 10 个时，应在关键模块目录下创建子目录级上下文文件（如 `module/AGENTS.md` 或 `module/.hicode/CONTEXT.md`），包含模块职责摘要、本地命令、关键入口、排除路径和 Owner。导航地图表的"注意事项"列应标注哪些模块已有子目录上下文文件及文件路径。

| 排除路径 | 排除原因 | 是否可被任务显式恢复读取 | 恢复读取条件 | 注意事项 |
|---|---|---|---|---|

**排除路径可执行化**：排除路径表中的每一行应同步写入 `.hicode/settings.json` 的 `permissions.deny` 或等价平台排除配置，确保 Agent 在搜索时自动跳过，而不需要人工记住排除规则。配置示例：

```json
{
  "permissions": {
    "deny": [
      { "path": "build/", "reason": "构建产物" },
      { "path": "generated/", "reason": "生成代码" },
      { "path": "node_modules/", "reason": "第三方依赖" }
    ]
  }
}
```

安装器或适配层应将本排除路径表转化为平台原生排除规则（如 `.gitignore`、`.cursorignore` 等）。排除规则不能掩盖安全、合规或生产配置风险。

## 16. Agent 使用提示

Agent 读取本文档时应重点确认：

1. 本次需求涉及哪些模块、接口、表、配置和外部依赖。
2. 是否涉及金融核心系统风险基线，尤其是保险核心业务逻辑严谨性、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
3. 是否存在历史风险点或相关 ADR。
4. 在大型代码库中，是否已依据导航地图缩小目录范围、确认局部命令、排除低价值路径，并说明搜索证据。
5. 是否需要提出 `docs/PRD_CONTEXT.md`、`docs/DOMAIN_KNOWLEDGE.md`、`docs/DEFECT_CASES.md` 或 ADR 更新建议。
