# PILOT_USAGE_LOG

## 1. 文档定位

本文档用于 P6-WP3，作为 V1 Prompt、Skill 和门禁在试点中的脱敏使用事件台账。它按一次使用记录一条事件，用于观察 V1 资产在需求评审、编码计划、TDD、辅助编码、代码审查、提交检查、核心场景测试和发布检查中的使用情况。

本文档不保存完整 Prompt、完整模型输出、原始代码 diff、生产日志、生产 SQL、客户材料、生产配置、生产地址、账号、密钥、Token、Cookie、Session、连接串或保单号。

- 维护对象：AI 工程化小组、试点项目负责人、研发负责人、测试负责人、安全负责人
- 主要使用者：试点研发代表、测试代表、产品/BA 代表、架构师或技术负责人、Reviewer、发布负责人
- 关联资产：`harness-assets/docs/pilot/PILOT_PROJECTS.md`、`harness-assets/docs/pilot/BASELINE_METRICS_PLAN.md`、`harness-assets/docs/pilot/V1_REVIEW_REPORT_TEMPLATE.md`

## 2. 使用规则

1. 每次使用记录一个脱敏 `usage_record_id`，格式建议为 `USAGE-YYYY-NNN`，例如 `USAGE-2026-001`。
2. 每条记录必须关联 `pilot_project_id`；若尚未确定试点项目，填写 `待确认` 并说明原因。
3. 可选关联 `metric_id`，用于支撑覆盖率、问题数、有效问题率、误报漏报、安全事件等统计。
4. 输入材料只记录来源类型和脱敏说明，不粘贴原文。
5. 输出内容只记录结论摘要、风险等级、采纳情况和问题摘要，不保存完整模型回复。
6. 采纳情况必须体现人工判断；不得把 AI 输出写成已自动执行、已自动合并、已自动发布或已替代负责人确认。
7. 发现敏感信息、生产越权、未经 Review 的 AI 代码合入、P0/P1 未关闭仍推进等问题时，必须记录风险并转人工确认。

## 5. 场景记录指引

### 5.1 重点需求评审

| 记录项 | 填写要求 |
|---|---|
| 输入来源类型 | 脱敏需求材料、`PRD_CONTEXT`、领域知识、历史缺陷摘要 |
| 输出结论摘要 | 记录澄清问题数量、P0/P1 缺口、建议动作 |
| 采纳情况 | 由产品/BA 代表、研发负责人或业务负责人复核 |
| 禁止记录 | 客户原文、保单号、未脱敏需求附件、真实审批意见全文 |

### 5.2 编码计划辅助和 ADR 判断

| 记录项 | 填写要求 |
|---|---|
| 输入来源类型 | 需求评审报告、`PRD_CONTEXT`、`PROJ_CONTEXT`、相关代码摘要、ADR 摘要 |
| 输出结论摘要 | 记录计划范围、上下文清晰门槛、ADR 判断、待确认问题 |
| 采纳情况 | 由研发负责人、架构师或技术负责人复核 |
| 禁止记录 | 未脱敏代码片段、生产配置、内部系统敏感编号 |

### 5.3 核心逻辑 TDD 和辅助编码

| 记录项 | 填写要求 |
|---|---|
| 输入来源类型 | 编码计划、TDD 报告、测试规范、公开接口摘要 |
| 输出结论摘要 | 记录测试场景、RED-GREEN-REFACTOR 证据摘要、修改建议摘要 |
| 采纳情况 | 由研发代表和 Reviewer 复核 |
| 禁止记录 | 完整 AI 代码回复、未经 Review 的合入结论、真实数据库或外部系统信息 |

### 5.4 AI Review 和提交前检查

| 记录项 | 填写要求 |
|---|---|
| 输入来源类型 | 固定基准点、脱敏 diff 摘要、需求来源、TDD 证据、Review 规则 |
| 输出结论摘要 | 记录 P0/P1/P2/P3 问题数量、阻断建议、提交建议 |
| 采纳情况 | 由 Reviewer 或研发负责人复核 |
| 禁止记录 | 原始完整 diff、密钥扫描命中原文、未脱敏配置或 SQL 详情 |

### 5.5 核心场景测试

| 记录项 | 填写要求 |
|---|---|
| 输入来源类型 | 需求、变更范围、领域知识、测试报告、历史缺陷摘要 |
| 输出结论摘要 | 记录核心场景数量、缺口、是否建议进入发布检查 |
| 采纳情况 | 由测试代表或测试负责人复核 |
| 禁止记录 | 未脱敏测试数据、客户样本、生产数据或真实外部系统信息 |

### 5.6 发布前检查

| 记录项 | 填写要求 |
|---|---|
| 输入来源类型 | 发布范围摘要、制品信息摘要、测试报告、缺陷摘要、SQL/配置摘要、回滚方案摘要 |
| 输出结论摘要 | 记录建议发布、有条件发布、不建议发布或待确认，以及关键风险 |
| 采纳情况 | 由发布负责人、测试负责人或研发负责人复核 |
| 禁止记录 | 生产命令、生产 SQL、生产日志、生产配置值、真实发布平台敏感链接 |

## 9. 禁止误用

1. 不保存完整 Prompt、完整模型输出、原始代码 diff、生产日志、生产 SQL、客户材料、生产配置、生产地址、账号、密钥、Token、Cookie、Session、连接串或保单号。
2. 不把 AI 输出写成自动执行结果、自动合并、自动发布、最终审批或负责人确认。
3. 不把使用次数、采纳情况或问题数编造成试点效果结论。
4. 不为了提高采纳率删除未采纳记录、误报记录、漏报记录或 P0/P1 问题。
5. 不记录真实人员名单、工号、手机号、邮箱或 IM 账号。
6. 不替代项目负责人、研发负责人、测试负责人、安全负责人、发布负责人、Reviewer 或业务负责人的最终确认。
