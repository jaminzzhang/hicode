# V1_REVIEW_REPORT_TEMPLATE

## 1. 文档定位

本文档用于 P6-WP4，作为 V1 试点复盘报告模板。它基于 `PILOT_USAGE_LOG.md` 和 `BASELINE_METRICS_PLAN.md` 汇总试点事实数据，区分分析判断和建议动作，形成 V1 后续优化、继续试点、条件推广或暂缓推广的依据。

本文档不替代项目负责人、研发负责人、测试负责人、安全负责人、发布负责人、业务负责人或管理层的最终验收和推广审批。未采集的数据必须标注缺口，不得编造成试点效果或推广结论。

- 维护对象：AI 工程化小组、试点项目负责人、研发负责人、测试负责人、安全负责人
- 主要使用者：项目负责人、试点团队、研发管理者、后续 V2 规划负责人
- 主要输入：`harness-assets/docs/pilot/PILOT_PROJECTS.md`、`harness-assets/docs/pilot/BASELINE_METRICS_PLAN.md`、`harness-assets/docs/pilot/PILOT_USAGE_LOG.md`

## 2. 使用规则

1. 每条复盘事项使用一个 `review_item_id`，格式建议为 `REVIEW-YYYY-NNN`，例如 `REVIEW-2026-001`。
2. 事实数据必须追溯到 `pilot_project_id`、`metric_id` 或 `usage_record_id`。
3. 分析判断必须说明依据、样本范围和不确定性，不能把单个事件扩大为总体结论。
4. 建议动作必须写清责任角色、优先级、前置条件和验收方式。
5. 结论只能使用建议性质表述：`建议继续试点`、`建议有条件推广`、`不建议推广`、`待补充证据`。
6. 存在 P0 安全问题、敏感信息泄露、生产越权、未经 Review 的 AI 代码合入或关键数据缺失时，不得输出无条件推广建议。
7. 本模板不得记录真实客户敏感信息、生产数据、生产日志、生产 SQL、生产地址、账号、密钥、Token、Cookie、Session、连接串或保单号。

## 15. 禁止误用

1. 不把复盘报告写成验收证书、推广审批、最终发布批准或管理层决策。
2. 不编造试点数据、覆盖率、有效问题率、满意度、安全结论或推广结论。
3. 不把目标建议写成当前已达成。
4. 不把未采集的数据、单个事件或局部反馈扩大为整体结论。
5. 不记录真实客户敏感信息、生产数据、生产日志、生产 SQL、生产配置、生产地址、账号、密钥、Token、Cookie、Session、连接串或保单号。
6. 不为了收口删除无效场景、误报、漏报、安全问题、阻断项或待确认问题。
7. 不替代项目负责人、研发负责人、测试负责人、安全负责人、发布负责人、业务负责人或管理层最终确认。
