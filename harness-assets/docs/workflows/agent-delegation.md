# 子 Agent 委托工作流

## 1. 触发条件

当用户任务需要专门角色进行需求评审、编码计划、TDD、受控辅助编码、代码审查、安全审查、Java 专项审查或发布风险审查时，触发本工作流。

本工作流用于统一任务路由和证据闭环，不替代具体 Prompt、Skill、门禁或人工审批。

## 2. 必读上下文

1. `AGENTS.md`
2. `CONTEXT.md`
3. `docs/PRD_CONTEXT.md`，涉及单需求时读取
4. `docs/PROJ_CONTEXT.md`，涉及代码、模块、接口或数据时读取
5. `docs/DOMAIN_KNOWLEDGE.md`，涉及保险核心业务规则时读取
6. `docs/CODING_RULES.md`，涉及编码或代码审查时读取
7. `docs/TESTING_GUIDE.md`，涉及测试、TDD、提测或发布证据时读取
8. `docs/REVIEW_RULES.md`，涉及 Review 或合并前检查时读取
9. `docs/RELEASE_GUIDE.md`，涉及发布、SQL、配置、回滚或生产验证计划时读取
10. `docs/DEFECT_CASES.md`，涉及历史缺陷、回归或风险复盘时读取
11. `.hicode/agents/` 中当前任务对应 Agent
12. `.hicode/prompts/`、`.hicode/skills/`、`.hicode/gates/` 和 `.hicode/schemas/` 中当前任务对应资产

只读取当前任务必要上下文。缺少上下文时，输出证据缺口和影响，不补编事实。

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 用户任务 | 目标、范围、期望输出、是否允许修改文件 |
| 需求材料 | 需求编号、PRD、故事、会议纪要、流程图或脱敏摘要 |
| 代码材料 | 分支、基准点、diff、提交列表、文件路径或代码片段 |
| 测试材料 | TDD 报告、单测结果、构建结果、覆盖率、核心场景测试报告 |
| Review 材料 | AI Review、人工 Review、讨论、风险接受或豁免记录 |
| 发布材料 | 发布范围、需求清单、分支制品、SQL、配置、回滚、生产验证计划 |
| 规则资产 | 对应 Agent、Prompt、Skill、Gate、Schema 和专项规则 |

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 4. 执行步骤

1. 接收用户任务，确认任务目标、范围和是否允许修改文件。
2. 先做安全红线检查；命中密钥、未脱敏客户信息、生产数据、生产操作、自动合并或自动发布诉求时停止推进。
3. 判断任务阶段：需求、计划、TDD、编码、Review、专项审查、发布或其他。
4. 判断是否需要委托 Agent：
   - 需要稳定角色边界、专项审查或高风险降噪时，委托 Agent。
   - 简单规则生成或单一报告可直接使用 Prompt/Skill。
5. 选择 Agent 或直接使用 Prompt/Skill。
6. 固定输入边界和证据来源，包括需求范围、diff 基准、测试证据、发布材料或负责人确认。
7. 读取对应 Prompt、Skill、Gate、Schema 和专项规则。
8. 按 Agent 角色边界和规则源执行分析、计划、审查或受控修改。
9. 记录验证动作、受限命令、未执行原因和降级情况。
10. 输出 Agent 层建议结论、底层规则源原始结论、风险等级、依据、建议动作、待确认问题和上下文更新建议。
11. 如需门禁，进入对应 Gate 形成建议性质准入报告，但不替代审批。
12. 如发现长期上下文、规则、缺陷或 ADR 缺口，只输出更新建议，不自动写入长期上下文。

## 5. Agent 选择规则

| 任务阶段或风险 | 推荐 Agent | 说明 |
|---|---|---|
| 需求进入开发前 | `requirement-reviewer` | 识别澄清点、业务风险、测试关注点和需求准入证据 |
| 编码启动前 | `coding-planner` | 输出影响范围、实现切片、TDD 重点和 ADR 判断 |
| 测试先行 | `tdd-guide` | 生成 TDD 证据；用户确认后可改测试相关文件 |
| 受控实现、修复、重构、解释 | `coding-assistant` | 生产代码修改必须有编码计划、TDD 证据和用户确认 |
| 开发完成后代码审查 | `code-reviewer` | 执行规范轴和需求轴审查 |
| 权限、密钥、隐私、生产数据、安全红线 | `security-reviewer` | 专项审查，不替代安全负责人确认 |
| Java、Spring、事务、SQL、批处理、保险核心后端 | `java-reviewer` | 专项审查，不替代 code-reviewer |
| 发布申请前或发布准入前 | `release-reviewer` | 审查发布材料、证据、回滚和生产验证计划 |

## 6. 直接使用 Prompt/Skill 的场景

以下场景可以不委托 Agent，直接使用 Prompt 或 Skill：

1. 用户明确要求运行某个 Prompt 或 Skill，且任务边界清晰。
2. 任务低风险、单一、无需专门角色审查。
3. 目标项目未安装对应 Agent，但已安装对应 Prompt/Skill。
4. 只是生成一次性报告草稿，且不会进入门禁、合并或发布流程。

直接使用 Prompt/Skill 时，仍必须遵守安全红线、上下文读取、验证记录和输出要求。

## 7. 降级处理

缺少资产时按以下方式处理：

1. 缺 Agent：直接使用对应 Prompt/Skill，并说明未进行子 Agent 委托。
2. 缺 Prompt：使用 Skill 和上下文输出受限建议，说明缺少详细规则源。
3. 缺 Skill：使用 Prompt 生成报告，说明缺少流程路由、输入准入和验证记录约束。
4. 缺 Gate：输出风险和建议动作，但不得输出准入建议结论。
5. 缺 Schema：输出 Markdown 报告，并说明未做结构化校验。
6. 缺上下文：输出证据缺口和待确认问题，不编造业务规则。
7. 缺专项规则：降级为普通审查或建议人工专项审查。

命中安全红线、生产越权、自动合并/发布诉求或无法判断 P0/P1 风险时，不得降级推进。

## 8. 输出

必须输出：

1. Agent 或 Prompt/Skill 路由结论。
2. 委托范围和输入边界。
3. 已读取资产和缺失资产。
4. 底层规则源原始结论。
5. Agent 层建议结论。
6. 风险等级：P0 / P1 / P2 / P3。
7. 依据或证据来源。
8. 建议动作。
9. 待确认问题。
10. 验证动作、受限命令执行记录或未执行原因。
11. 降级情况，如有。
12. 上下文更新建议。
13. 下一步建议。

不得输出准许合并、准许发布、门禁通过、可以上线或最终审批通过。

## 9. 质量标准

1. 每个建议结论必须能追溯到输入材料、上下文、Prompt、Skill、Gate、Schema、测试结果、Review 报告或负责人确认。
2. P0/P1 问题必须说明触发条件、失败场景、影响、依据、建议动作和建议确认人。
3. 角色路由必须说明为什么选择某个 Agent 或为什么直接使用 Prompt/Skill。
4. 降级输出必须说明缺失资产、影响和补齐建议。
5. 不确定内容必须标注 `待确认`。
6. 不制造发现；没有证据的问题应降级、标为待确认或不输出。
7. 输出保持短、清晰、可维护，不复制 Prompt、Skill、门禁或需求草案全文。

## 10. 安全约束

1. 不读取或输出生产账号、密码、Token、Cookie、Session、连接串、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏客户敏感信息或未脱敏生产数据。
4. 不向外部服务提交客户敏感信息、生产数据或密钥。
5. 不直接操作生产环境。
6. 不自动提交、自动推送、自动合并、自动发布或自动回滚。
7. 不自动修改生产配置。
8. 不通过删除测试、降低断言、跳过 Review 或隐藏风险推动流程通过。

## 11. 下一步

1. 需要目标项目入口路由时，进入 `V2-P2-WP2` 更新 `harness-assets/AGENTS.md`。
2. 需要选择性安装时，进入 `V2-P3` 建立 `harness-assets/install/`。
3. 需要自动触发提醒或阻断设计时，进入 `V2-P4` Hook 化设计。
4. 发现稳定新术语时，输出 `CONTEXT.md` 更新建议；正式写入前需负责人确认。
5. 发现规则、缺陷或 ADR 缺口时，输出对应文档更新建议；不得自动把推断写成事实。
