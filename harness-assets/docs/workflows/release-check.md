# 发布检查工作流

## 1. 触发条件

发布申请前、发布准入门禁前，或发布范围发生变化时，触发本工作流。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/RELEASE_GUIDE.md`
3. `docs/PRD_CONTEXT.md`
4. `docs/PROJ_CONTEXT.md`
5. `docs/TESTING_GUIDE.md`
6. `docs/DEFECT_CASES.md`

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 神兵版本链接或版本标识 | 发布版本来源 |
| 发布需求清单 | 本次发布范围 |
| 代码分支 | 发布分支 |
| Commit / MR 清单 | 已合入变更 |
| 制品版本 | 待发布制品 |
| SQL / 配置 | 发布相关脚本和配置 |
| 测试报告 | 测试通过情况 |
| 缺陷状态 | P0/P1 缺陷关闭情况 |
| 回滚方案 | 回滚条件和步骤 |

## 4. 执行步骤

1. 核对发布需求清单。
2. 核对分支、制品和 Commit / MR。
3. 检查金融核心系统风险基线是否已有结论，尤其是保险核心业务逻辑、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
4. 检查 SQL、配置、接口兼容性和脚本。
5. 检查测试报告、核心场景和缺陷关闭情况。
6. 检查生产验证点。
7. 检查回滚方案。
8. 输出发布风险和上线后观察建议。

## 5. 输出

1. 发布前检查报告。
2. 发布结论：建议发布 / 有条件发布 / 不建议发布。
3. 发布范围。
4. 需求、分支、制品、Commit / MR、SQL、配置、测试、缺陷检查结果。
5. 生产验证点。
6. 回滚方案。
7. 发布风险。
8. 上线后观察建议。

## 6. 质量标准

1. 发布需求清单完整。
2. 分支和制品一致。
3. 测试通过。
4. 无未关闭 P0/P1 缺陷。
5. SQL 和配置齐备。
6. 回滚方案明确。
7. 生产验证点明确。
8. 对敏感信息、生产越权、P0 业务或安全风险必须给出阻断建议，实际阻断由既有流程或负责人决策。

## 7. 安全约束

1. Agent 不自动发布。
2. Agent 不修改生产配置。
3. Agent 不操作生产环境。
4. Agent 不输出生产账号、密钥或客户敏感信息。

## 8. 下一步

1. 检查通过后进入人工发布审批。
2. 有条件发布时补充负责人确认和缓解措施。
3. 不建议发布时修复后重新检查。
