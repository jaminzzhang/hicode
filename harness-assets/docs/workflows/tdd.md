# TDD 工作流

## 1. 触发条件

编码前需要明确测试标准，或需求涉及核心业务逻辑、金额、日期、状态、责任、权限、外部系统、历史缺陷时，触发本工作流。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/PRD_CONTEXT.md`
3. `docs/PROJ_CONTEXT.md`
4. `docs/CODING_RULES.md`
5. `docs/TESTING_GUIDE.md`
6. `docs/DEFECT_CASES.md`
7. 编码计划

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 编码计划 | 改动目标、影响范围、风险和实现步骤 |
| 需求上下文 | 业务规则、验收标准、测试关注点 |
| 代码上下文 | 相关类、接口、模块和已有测试 |

## 4. 执行步骤

1. 明确测试目标和测试范围。
2. 列出正常、边界、异常和历史缺陷场景。
3. 生成 Given-When-Then 用例。
4. 设计 Mock 策略。
5. 明确测试数据和脱敏要求。
6. 明确断言规则。
7. 输出单元测试代码或测试代码建议。

## 5. 输出

1. TDD 测试设计报告。
2. 测试场景清单。
3. Given-When-Then 用例。
4. 边界和异常场景。
5. Mock 策略。
6. 测试数据。
7. 断言规则。
8. 单元测试代码或建议。
9. 上下文更新建议。

## 6. 质量标准

1. 核心逻辑必须覆盖正常、边界、异常。
2. 金额断言必须使用精确数值口径。
3. 日期测试必须覆盖起保日、终保日、越界和闰年。
4. 外部系统必须 Mock。
5. 单元测试不得访问真实数据库。

## 7. 安全约束

1. 测试数据不得包含真实客户敏感信息。
2. 不使用真实账号、Token、Cookie、Session。
3. 不读取生产配置或密钥文件。

## 8. 下一步

1. TDD 设计通过后，进入辅助编码。
2. 如发现规则不清，回到需求评审或编码计划补澄清问题。
3. 如发现历史缺陷模式，建议更新 `docs/DEFECT_CASES.md`。
