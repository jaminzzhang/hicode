# 核心场景测试工作流

## 1. 触发条件

需求进入测试阶段，或变更涉及保险核心业务逻辑、历史缺陷、高风险金额/交易/状态/权限/外部调用时，触发本工作流。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/TESTING_GUIDE.md`
3. `docs/DOMAIN_KNOWLEDGE.md`
4. `docs/PRD_CONTEXT.md`
5. `docs/PROJ_CONTEXT.md`
6. `docs/DEFECT_CASES.md`
7. 编码计划
8. 代码 diff

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 需求上下文 | 本次需求目标、范围、规则、测试关注点 |
| 领域知识 | 核心业务域和核心场景库 |
| 历史缺陷 | 相似缺陷和防范规则 |
| 已有测试用例 | 可复用回归资产 |

## 4. 执行步骤

1. 识别本次需求影响业务场景。
2. 按金融核心系统风险标准补充保险核心业务逻辑严谨性、金额精度、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚场景。
3. 生成必测场景。
4. 生成建议回归场景。
5. 补充边界测试和异常测试。
6. 明确测试数据要求。
7. 判断自动化可行性。
8. 关联历史缺陷。
9. 输出测试风险。

## 5. 输出

1. 核心场景测试报告。
2. 必测场景。
3. 建议回归场景。
4. 边界测试。
5. 异常测试。
6. 测试数据要求。
7. 自动化可行性。
8. 历史缺陷关联。
9. 测试风险。

## 6. 质量标准

1. 必测场景要覆盖本次需求直接影响范围。
2. 建议回归场景要覆盖相关核心链路。
3. 历史缺陷相似场景必须标出。
4. 不使用真实客户敏感数据。

## 7. 安全约束

1. 测试数据必须脱敏或虚构。
2. 不访问生产环境。
3. 不输出生产账号、密钥或真实客户信息。

## 8. 下一步

1. 核心场景生成后进入提测门禁。
2. 若发现新缺陷模式，输出 `docs/DEFECT_CASES.md` 更新建议；经测试和研发共同确认后再沉淀。
3. 若发现领域规则缺口，输出 `docs/DOMAIN_KNOWLEDGE.md` 或 `docs/PRD_CONTEXT.md` 更新建议；经对应负责人确认后再沉淀。
