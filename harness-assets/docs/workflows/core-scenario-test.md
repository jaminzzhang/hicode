# 核心场景测试工作流

## 1. 触发条件

需求进入测试阶段，或变更涉及保险核心业务逻辑、历史缺陷、高风险金额/交易/状态/权限/外部调用时，触发本工作流。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/TESTING_GUIDE.md`
3. `docs/DOMAIN_KNOWLEDGE.md`
4. `docs/PRD_CONTEXT.md`
5. `docs/PROJ_CONTEXT.md`
6. `docs/DEFECT_CASES.md`
7. `docs/CODING_RULES.md`，涉及代码、配置、日志、权限或敏感信息时读取
8. `docs/RELEASE_GUIDE.md`，输出将用于发布检查时读取
9. 编码计划
10. TDD 报告、代码审查报告、提交检查报告，如已有
11. 代码 diff
12. 已有测试用例、测试报告、测试命令或覆盖率结果，如已有

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 需求上下文 | 本次需求目标、范围、规则、测试关注点 |
| 领域知识 | 核心业务域和核心场景库 |
| 历史缺陷 | 相似缺陷和防范规则 |
| 已有测试用例 | 可复用回归资产 |
| 代码 diff | 本次变更范围和影响对象 |
| 测试报告或命令 | 已执行结果或可受限执行的本地测试命令 |

## 4. 执行步骤

1. 按需求层、项目层、领域层、测试层、缺陷层和变更层整理证据。
2. 识别本次需求影响业务场景。
3. 按金融核心系统风险标准补充保险核心业务逻辑严谨性、金额精度、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚场景。
4. 生成必测场景。
5. 生成建议回归场景。
6. 补充边界测试和异常测试。
7. 明确测试数据要求。
8. 判断自动化可行性。
9. 在目标项目本地、非生产环境中按已确认命令执行受限测试，或说明不能执行的原因。
10. 关联历史缺陷。
11. 输出测试风险和测试侧建议结论。

## 5. 输出

1. 核心场景测试报告。
2. 测试侧结论：建议进入发布检查 / 有条件进入发布检查 / 不建议进入发布检查 / 待确认。
3. 证据分层和证据缺口。
4. 必测场景。
5. 建议回归场景。
6. 边界测试。
7. 异常测试。
8. 测试数据要求。
9. Mock 与环境约束。
10. 自动化可行性。
11. 受限命令执行记录。
12. 历史缺陷关联。
13. 测试风险。

## 6. 质量标准

1. 必测场景要覆盖本次需求直接影响范围。
2. 建议回归场景要覆盖相关核心链路。
3. 历史缺陷相似场景必须标出。
4. 需求层或变更层证据缺失时，只能输出 `待确认`。
5. 不使用真实客户敏感数据。
6. 未实际执行的测试不得写成已通过。

## 7. 安全约束

1. 测试数据必须脱敏或虚构。
2. 不访问生产环境。
3. 不输出生产账号、密钥或真实客户信息。
4. 不访问真实数据库或真实外部系统。
5. 不读取生产配置、生产数据、密钥文件或 `.env`。
6. 不执行数据库变更、发布、回滚、生产连接或删除未确认文件。

## 8. 下一步

1. 核心场景生成后进入提测门禁。
2. 若发现新缺陷模式，输出 `docs/DEFECT_CASES.md` 更新建议；经测试和研发共同确认后再沉淀。
3. 若发现领域规则缺口，输出 `docs/DOMAIN_KNOWLEDGE.md` 或 `docs/PRD_CONTEXT.md` 更新建议；经对应负责人确认后再沉淀。
