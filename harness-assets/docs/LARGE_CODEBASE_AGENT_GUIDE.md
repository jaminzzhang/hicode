# 大型代码库 Agent 工作方法论

> 本文基于 `docs/references/How Claude Code works in large codebases Best practices and where to start.md` 的本地参考资料转化而来，用于沉淀 hicode 在大型保险/金融核心系统代码库中的 Agent 使用方法。本文不是 Claude Code 原文摘录，也不代表目标项目已经完成相关工具接入。

## 1. 定位

大型代码库中的 Agent 效果不只取决于模型能力，更取决于代码库是否可导航、上下文是否分层、工具是否可验证、治理是否有人负责。

hicode 应把大型代码库适配看成一套 Harness 工程，而不是把所有规则都写进入口文件。

适用场景：

1. 多模块单体、Monorepo、多仓服务群或长期演进的遗留系统。
2. Java、SQL、批处理、消息、外部接口和保险核心业务规则交织的目标项目。
3. 需要让 Agent 在大范围代码中做需求影响分析、编码计划、TDD、代码审查、专项审查或发布检查。
4. 目标项目准备引入 `.hicode/agents/`、`.hicode/prompts/`、`.hicode/skills/`、`.hicode/hooks/`、MCP、LSP 或内部工具集成。

不适用：

1. 用本文替代 `AGENTS.md`、`PROJ_CONTEXT.md`、Prompt、Skill 或门禁。
2. 让 Agent 全仓扫描所有文件后再判断任务范围。
3. 把参考资料直接作为目标项目运行时必读上下文。
4. 授权自动合并、自动发布、生产连接、生产 SQL、生产日志读取或生产配置修改。

## 2. 核心判断

参考资料的关键经验可以转化为 hicode 的四条原则：

1. **实时导航优先于陈旧索引**：Agent 应从当前工作区读取文件、搜索引用、跟随代码关系；不能依赖可能过期的外部索引来判断真实代码状态。
2. **Harness 优先于单一模型能力**：入口规则、项目上下文、Skill、Hook、子 Agent、LSP、MCP、安装 profile 和治理机制共同决定结果质量。
3. **默认上下文必须克制**：入口文件只保留全局规则、索引和高风险边界；专项知识放入 Skill、专项规则、子目录上下文或按需文档。
4. **探索与修改要分离**：大型代码库中，先用只读方式建立局部地图、证据和影响范围，再进入受控修改、测试和 Review。

## 3. hicode Harness 分层

| 层级 | hicode 对应资产 | 主要职责 | 常见误用 |
|---|---|---|---|
| 入口上下文 | `AGENTS.md`、`docs/PROJ_CONTEXT.md` | 给出项目全局规则、上下文索引、模块地图和安全边界 | 把所有细则塞进入口文件 |
| 任务规则 | `.hicode/prompts/` | 定义场景目标、检查维度、输出要求 | 用 Prompt 替代流程准入或审批 |
| 流程能力 | `.hicode/skills/` | 按任务类型组织步骤、输入准入、验证记录 | 每次会话全量加载所有 Skill |
| 角色委托 | `.hicode/agents/` | 分离需求、计划、TDD、编码、Review、安全、Java、发布等角色 | 所有任务默认触发所有 Agent |
| 门禁触发 | `.hicode/hooks/`、`.hicode/gates/` | 在关键动作前后形成 advisory 或 blocking 建议 | 用 Hook 包装生产命令或自动审批 |
| 代码智能 | LSP / IDE 代码智能 | 按符号定位定义、引用和类型错误 | 只用文本搜索判断复杂调用关系 |
| 内部工具 | MCP / 内部 API | 连接已批准的文档、工单、知识库或质量平台 | 在基础上下文未稳定前先做重集成 |
| 分发治理 | `harness-assets/install/`、profile、插件化分发 | 把有效配置分发给目标团队 | 让有效配置停留在个人经验里 |

## 4. 大型代码库导航流程

Agent 在大型代码库工作时，默认按以下顺序推进：

1. **确认任务边界**：明确用户要做需求评审、计划、编码、Review、测试还是发布检查；明确是否允许修改文件。
2. **读取入口和项目地图**：读取 `AGENTS.md`、`docs/PROJ_CONTEXT.md` 中的模块结构、关键流程、局部命令、排除目录和负责人信息。
3. **缩小工作目录**：优先进入与任务相关的模块或服务目录，不从仓库根目录盲目展开。
4. **实时搜索证据**：使用 `rg`、语言工具、测试配置、构建文件和代码引用查找当前真实实现。
5. **按符号确认关系**：在有 LSP 或等价代码智能时，优先用定义、引用、类型和调用关系验证影响范围。
6. **只读探索先行**：复杂任务先输出模块地图、影响面、待确认问题和风险，不立即修改。
7. **受控修改**：只有具备需求上下文、编码计划、TDD/测试先行证据和用户确认时，才进入代码修改。
8. **局部验证优先**：优先运行当前模块的测试、lint、构建或静态检查；全量验证只在风险或流程要求时执行。
9. **记录证据与缺口**：输出已读取资产、搜索方式、命令结果、未验证原因、降级情况和上下文更新建议。

## 5. 上下文分层原则

### 5.1 入口文件保持短

`AGENTS.md` 应只维护：

1. 项目定位。
2. 必读上下文索引。
3. 任务路由和子 Agent 委托规则。
4. 安全红线。
5. 输出规范。

不应维护：

1. 长篇业务规则。
2. 完整历史缺陷库。
3. 所有测试命令和构建命令。
4. 所有目录的详细说明。
5. 可按需加载的专项知识。

### 5.2 项目上下文提供地图

`docs/PROJ_CONTEXT.md` 应补充大型代码库导航所需的稳定信息：

1. 顶层目录和模块职责。
2. 每个模块的本地测试、构建、lint 和启动命令。
3. 生成文件、第三方代码、构建产物和低价值扫描目录。
4. LSP、MCP、内部搜索、质量平台等可用工具。
5. 跨模块依赖和高风险变更边界。

### 5.3 Skill 和专项规则按需加载

专项能力应放在 Skill、review-rules 或 Agent 中，不放入入口文件。只有当任务触发对应场景时才加载，例如：

1. 安全相关变更触发 `security-reviewer` 和安全规则。
2. Java/Spring/SQL/事务变更触发 `java-reviewer` 和专项规则。
3. 发布材料审查触发 `release-reviewer` 和发布指南。

### 5.4 子目录上下文文件

当目标项目目录超过 3 层或模块超过 10 个时，应在关键模块目录下创建局部上下文文件，例如 `module/AGENTS.md` 或 `module/.hicode/CONTEXT.md`。Agent 进入子目录工作时自动加载该文件，向上遍历时与上层上下文合并。

子目录上下文文件应包含以下 5 类内容：

1. **模块职责摘要**：本模块做什么、不做什么、与哪些模块有直接依赖。
2. **本地命令**：本模块的测试、构建、lint、启动和特定验证命令。
3. **关键入口**：本模块对外暴露的主要类、接口、API 或消息入口。
4. **排除路径**：本模块内应排除扫描的生成代码、第三方或构建产物目录。
5. **Owner**：本模块的技术负责人或 DRI，标注姓名或"待指定"。

子目录上下文文件不应包含：

1. 长篇业务规则（应放入 `docs/DOMAIN_KNOWLEDGE.md` 或专项规则）。
2. 完整缺陷库（应放入 `docs/DEFECT_CASES.md`）。
3. 其他模块的命令或接口（应放入对应模块的子目录上下文或 `docs/PROJ_CONTEXT.md`）。

层级加载规则：

1. Agent 进入子目录时自动加载该目录的局部上下文文件。
2. 向上遍历时合并上层上下文，子目录内容优先。
3. `docs/PROJ_CONTEXT.md` 3.1 导航地图应标注哪些模块已有子目录上下文文件及文件路径。
4. 子目录上下文文件的维护责任由对应模块 Owner 承担，全局导航地图由 hicode 配置维护 DRI 负责。

## 6. 搜索、排除与命令

### 6.1 搜索原则

1. 首选 `rg` 搜索文件、符号、配置和测试。
2. 搜索前先限定目录、扩展名或模块。
3. 命中太多时，先收窄词、路径和文件类型，不把大量无关输出带入上下文。
4. 对同名方法、同名类和跨语言符号，必须用定义、引用、导入路径或 LSP 结果确认。

### 6.2 排除原则

目标项目应明确以下目录或文件是否默认排除：

1. 构建产物。
2. 生成代码。
3. 第三方依赖。
4. 大型二进制资产。
5. 历史归档和迁移临时目录。

排除规则不能掩盖安全、合规、生产配置或客户隐私风险。若生成代码本身是当前任务对象，必须在任务说明中显式恢复读取范围。

### 6.3 排除路径可执行化

`docs/PROJ_CONTEXT.md` 的排除路径应从文档描述同步写入平台可执行的排除配置，确保 Agent 在搜索时自动跳过。

配置要求：

1. 排除路径表中的每一行应同步写入 `.hicode/settings.json` 的 `permissions.deny` 或等价平台排除配置。
2. 配置格式示例：

```json
{
  "permissions": {
    "deny": [
      { "path": "build/", "reason": "构建产物" },
      { "path": "generated/", "reason": "生成代码" },
      { "path": "node_modules/", "reason": "第三方依赖" },
      { "path": "vendor/", "reason": "第三方依赖" },
      { "path": "archive/", "reason": "历史归档" }
    ]
  }
}
```

3. 安装器或适配层应将 `docs/PROJ_CONTEXT.md` 排除路径表转化为平台原生排除规则（如 `.gitignore`、`.cursorignore`、IDE 排除配置等）。
4. 排除规则不能掩盖安全、合规或生产配置风险。被排除目录中的安全、合规或生产配置变更仍需在门禁和 Review 中覆盖。

### 6.4 局部命令原则

`docs/PROJ_CONTEXT.md` 应记录每个模块的局部命令：

| 模块 | 测试命令 | 构建命令 | lint/格式化 | 适用条件 | 备注 |
|---|---|---|---|---|---|

Agent 修改某个模块时，应优先运行该模块命令。全量命令适用于：

1. 跨模块公共契约变更。
2. 构建脚本、依赖或基础框架变更。
3. 发布前检查要求。
4. 局部命令无法覆盖影响范围。

## 7. 子 Agent 使用原则

大型代码库中，子 Agent 的价值是隔离上下文和职责，不是增加角色噪音。

推荐用法：

1. **只读探索 Agent**：在复杂模块或陌生系统中，先让只读 Agent 输出模块地图、调用链、风险点和建议读取文件。
2. **专项审查 Agent**：当风险明确触发时再委托安全、Java/SQL、保险业务或发布专项审查。
3. **主 Agent 保留修改上下文**：探索结果回收后，由主 Agent 在明确范围内修改文件并验证。

不推荐：

1. 所有任务同时触发所有 Agent。
2. 让探索 Agent 直接修改生产代码。
3. 用子 Agent 输出替代门禁、人工 Review 或发布审批。

## 8. Hook 使用原则

Hook 在大型代码库中的主要价值有两类：

1. **确定性检查**：在写文件、提交、推送或合并前触发格式、lint、敏感信息、流程证据和门禁报告检查。
2. **持续改进**：在会话结束或关键节点提出上下文更新建议、命令更新建议和风险规则更新建议。

hicode 首批 Hook 仍保持收敛：

1. 编码准入门禁 Hook。
2. 合并门禁 Hook。

Hook 不得：

1. 自动合并。
2. 自动发布。
3. 自动回滚。
4. 连接生产。
5. 执行生产 SQL。
6. 读取生产日志原文。
7. 修改生产配置。

## 9. LSP 和 MCP 使用原则

### 9.1 LSP

对 Java、C/C++、C#、TypeScript 等大型强类型代码库，LSP 或等价代码智能是高价值能力。

适用：

1. 查找定义和引用。
2. 区分同名函数、同名类和同名字段。
3. 识别类型错误和导入错误。
4. 降低全仓文本搜索噪音。

边界：

1. LSP 结果仍需结合业务上下文判断。
2. LSP 不替代测试、Review 或门禁。
3. LSP 未配置时，应在输出中说明导航可信度下降。

#### 9.1.1 LSP 未就绪时的降级导航策略

当 LSP 尚未配置或不可用时，Agent 应采用以下降级行为，并在输出中标注"本次分析未使用 LSP，同名符号和跨模块调用关系需人工复核"。

**推荐 LSP 配置**：

| 语言 | 推荐 LSP | 最小可用配置步骤 |
|---|---|---|
| Java | `jdtls`（Eclipse JDT Language Server） | 1. 安装 `jdtls`；2. 设置 workspace 路径指向项目根目录；3. 启动命令 `jdtls -data /path/to/workspace`；4. 等待索引完成 |
| TypeScript | `typescript-language-server` | 1. 全局安装 `npm install -g typescript-language-server typescript`；2. 在项目根目录执行 `tsc --init`（如无 `tsconfig.json`）；3. 启动命令 `typescript-language-server --stdio` |

**降级行为**：

1. 搜索时必须限定目录和文件类型，不做全仓 grep。
2. 对同名类/方法，必须交叉验证导入路径和包名/命名空间。
3. 跨模块调用关系必须从 import/require 语句手动追踪。
4. 输出中标注"本次分析未使用 LSP，同名符号和跨模块调用关系需人工复核"。

**LSP 准备就绪判断标准**：能成功响应 `textDocument/definition` 和 `textDocument/references` 请求。

### 9.2 MCP

MCP 适合连接已批准的内部工具，例如知识库、工单系统、质量平台、代码搜索或指标平台。

接入前提：

1. 已有稳定入口上下文、Skill、门禁和安全边界。
2. 数据源不包含未脱敏客户敏感信息、密钥或未授权生产数据。
3. 有权限、审计、脱敏和失败降级口径。

不应在基础 Harness 未稳定前优先建设重型 MCP 集成。

## 10. 维护与治理

大型代码库 Agent Harness 需要持续维护。

建议机制：

1. 设定 hicode 资产 Owner 或 DRI，负责入口文件、profile、Skill、Hook 和权限策略。DRI 应在 `AGENTS.md` 项目定位表中明确标注为"技术负责人"，在 `docs/PROJ_CONTEXT.md` 3.1 导航地图中标注各模块 Owner。
2. 每 3 至 6 个月进行一次上下文与规则复核。
3. 模型、Agent 平台、LSP、Hook 或权限能力明显变化后，进行专项复核。
4. 将有效配置沉淀为可分发资产，不停留在个人经验里。
5. 在安全、工程、测试、架构和合规角色之间建立跨职能确认机制。

**DRI 缺位时的降级处理**：

1. `docs/PROJ_CONTEXT.md` 3.1 导航地图中每个模块必须填写 Owner 或标注"待指定"。
2. Agent 发现 Owner 字段为空或"待指定"时，应在输出中提示"本模块无确认 Owner，上下文更新建议未获确认，需项目负责人复核"。
3. 长期上下文更新建议不自动写入，标记为待 DRI 确认。
4. 上下文捕获 Hook 的建议列表中，缺少 Owner 确认的条目应单独标注，提醒 hicode 配置维护 DRI 跟进。

复核重点：

1. 入口文件是否过长。
2. 是否有过期命令、废弃模块、错误路径。
3. 是否有可下沉到 Skill 或专项规则的内容。
4. 是否有过度保守、限制新模型能力的旧规则。
5. 是否仍符合金融核心系统风险标准。

## 11. 保险核心系统专项要求

大型代码库方法论不能降低金融核心系统风险标准。任何导航、搜索、委托、Hook 或 MCP 能力都必须显式覆盖：

1. 保险核心业务逻辑严谨性。
2. 金额精度、舍入、币种和对账。
3. 交易一致性、事务边界、补偿和重试。
4. 保单、批单、理赔前置、收付费等状态流转。
5. 幂等、并发、锁和重复提交。
6. 权限、审计、操作来源和留痕。
7. 客户隐私、日志脱敏和监管合规。
8. SQL、配置、回滚和发布准入。

命中 P0/P1、生产越权、敏感信息或关键证据缺失时，不得以“代码库太大”或“上下文不足”为理由降级推进。

## 12. 落地检查清单

| 检查项 | 通过标准 | 证据位置 |
|---|---|---|
| 入口短而清晰 | `AGENTS.md` 只保留索引、路由、安全和输出规则 | `AGENTS.md` |
| 项目地图齐备 | `PROJ_CONTEXT.md` 有模块、目录、命令、排除和工具说明 | `docs/PROJ_CONTEXT.md` |
| 搜索有边界 | Agent 输出说明搜索路径、关键词、命中范围和未覆盖范围 | 分析或 Review 报告 |
| 局部验证优先 | 修改后优先运行相关模块命令，无法运行时说明原因 | 验证记录 |
| 专项能力按需加载 | 安全、Java/SQL、发布等只在触发时加载 | Agent 路由记录 |
| Hook 边界清晰 | advisory/blocking 边界清楚，不授权生产或自动审批 | `.hicode/hooks/`、门禁报告 |
| LSP/MCP 有边界 | 说明可用工具、权限、数据脱敏和降级方式 | `docs/PROJ_CONTEXT.md`、工具说明 |
| 治理有 Owner | 有 hicode 配置维护人和复核节奏 | 目标项目治理记录 |

## 13. 禁止事项

1. 不全仓盲扫后再判断任务范围。
2. 不把所有规则塞进 `AGENTS.md`。
3. 不把 `full-library` 理解为全量默认加载。
4. 不让所有子 Agent 默认同时运行。
5. 不把 Hook 设计变成自动合并或发布系统。
6. 不用 LSP、MCP 或外部索引替代测试、Review、门禁和负责人确认。
7. 不处理未脱敏客户信息、生产数据、密钥、生产配置或生产日志原文。
8. 不连接生产、执行生产 SQL、读取生产日志、自动回滚或自动发布。
