# ADR README

## 1. 目录定位

本目录保存目标项目的架构决策记录 ADR，用于记录重要、难逆转、存在真实权衡且未来读者需要理解背景的技术决策。

- 维护对象：架构师、技术负责人
- 更新时机：发生需要记录的架构决策时
- 主要使用者：编码计划、代码审查、架构评审 Agent
- 当前版本：V1 初版规则，Agent 只做 ADR 判断和草稿建议

## 2. 使用原则

1. Agent 可以判断是否建议产生 ADR。
2. Agent 可以生成 ADR 草稿。
3. Agent 不能自动确认架构决策。
4. Agent 不能自动合入 ADR。
5. 正式 ADR 必须由技术负责人或架构师确认。

## 3. 何时需要 ADR

同时满足以下条件时，建议创建 ADR：

1. 决策难以逆转，后续变更成本较高。
2. 没有背景说明时，未来维护者难以理解。
3. 存在真实备选方案和权衡。

## 4. V1 ADR 判断场景

| 场景 | 是否建议 ADR | 说明 |
|---|---|---|
| 修改系统架构分层 | 需要 | 影响模块职责和依赖方向 |
| 引入新技术组件 | 需要 | 影响技术栈、运维和维护成本 |
| 改变核心数据模型 | 需要 | 影响数据兼容、迁移和历史数据 |
| 改变关键接口契约 | 需要 | 影响上下游调用方 |
| 改变事务边界 | 需要 | 影响一致性和失败补偿 |
| 改变幂等策略 | 需要 | 影响重复请求和外部交互 |
| 改变一致性方案 | 需要 | 影响数据正确性和用户体验 |
| 改变发布或回滚策略 | 需要 | 影响上线风险控制 |
| 改变核心业务处理链路 | 需要 | 影响投保、批改、退保、核保、理赔前置、收付费等核心流程 |
| 普通字段新增 | 通常不需要 | 除非影响接口契约、数据模型或兼容性 |
| 简单 Bug 修复 | 通常不需要 | 除非修复方式改变架构或关键规则 |
| 小范围页面调整 | 通常不需要 | 除非影响权限、流程或核心交互 |
| 普通文案修改 | 不需要 | 不涉及架构决策 |

## 5. 判断输出要求

编码计划或 Review 中涉及 ADR 判断时，应输出：

1. 是否需要新增或更新 ADR：是 / 否。
2. 判断理由。
3. 涉及的决策点。
4. 备选方案，若已知。
5. 建议确认人。
6. 是否已有相关 ADR。

## 6. 文件命名

建议格式：

```text
ADR-0001-short-title.md
ADR-0002-short-title.md
```

## 7. 状态枚举

| 状态 | 含义 |
|---|---|
| Proposed | 草稿，待确认 |
| Accepted | 已确认 |
| Deprecated | 已废弃 |
| Superseded | 已被其他 ADR 替代 |

## 8. ADR 合入规则

1. `Proposed` 状态可以由 Agent 生成草稿。
2. `Accepted` 状态必须由技术负责人或架构师确认后更新。
3. `Deprecated` 和 `Superseded` 必须说明替代方案或废弃原因。
4. ADR 不保存生产账号、密钥、客户敏感信息或未脱敏示例。
5. ADR 中的事实、约束和结论必须可追溯到需求、代码、评审或负责人确认。
