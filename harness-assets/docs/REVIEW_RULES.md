# REVIEW_RULES

## 1. 文档定位

本文档记录目标项目的 AI Review 和人工 Review 规则，用于检查需求一致性、架构规范、业务逻辑、测试、安全和可维护性。

- 维护对象：技术负责人、架构师、模块 Owner
- 更新时机：Review 规则变化、缺陷复盘、架构约束变化后
- 主要使用者：代码审查、提交检查、合并门禁 Agent
- 当前版本：V1 初版规则，AI Review 不替代人工 Review

## 2. Review 输入

执行 AI Review 前应尽量准备：

1. 代码 diff。
2. `docs/PRD_CONTEXT.md`。
3. `docs/PROJ_CONTEXT.md`。
4. `docs/CODING_RULES.md`。
5. `docs/TESTING_GUIDE.md`。
6. `docs/DOMAIN_KNOWLEDGE.md`。
7. `docs/DEFECT_CASES.md`。
8. 编码计划。
9. 单测结果。

## 3. 审查结论

| 结论 | 使用条件 | 下一步 |
|---|---|---|
| 通过 | 未发现阻断问题，测试和上下文证据充分 | 可进入提交检查 |
| 有条件通过 | 存在 P1/P2 问题，但已有修复计划或明确豁免 | 修复或补充说明后进入提交检查 |
| 不建议提交 | 存在 P0 问题，或 P1 问题未处理且风险不可接受 | 修复后重新 Review |

## 4. 问题分级

| 等级 | 含义 | 处理要求 | 示例 |
|---|---|---|---|
| P0 | 可能导致生产事故、数据错误、安全漏洞或重大合规问题 | 必须修复，不建议提交 | 金额计算错误、敏感信息泄露、权限绕过、生产配置误用 |
| P1 | 可能导致功能异常、边界错误、性能问题或核心场景漏测 | 修复或由负责人明确豁免 | 日期边界错误、状态流转遗漏、外部调用无异常处理 |
| P2 | 可维护性、规范性或测试完整性问题 | 建议本次修复或登记后续处理 | 命名不清、重复代码、测试数据不完整 |
| P3 | 风格、可读性或优化建议 | 可选处理 | 文档说明可改进、局部结构可优化 |

## 5. 检查维度

| 维度 | 检查重点 | 关联文档 | 常见等级 |
|---|---|---|---|
| 需求一致性 | 是否实现需求目标、业务范围、验收标准，是否遗漏需求规则 | `docs/PRD_CONTEXT.md` | P0-P2 |
| 架构规范 | 是否符合分层、职责、依赖方向和模块边界 | `docs/PROJ_CONTEXT.md`、`docs/CODING_RULES.md` | P1-P2 |
| 业务逻辑 | 金额、日期、状态、责任、权限、幂等是否正确 | `docs/DOMAIN_KNOWLEDGE.md`、`docs/PRD_CONTEXT.md` | P0-P1 |
| 异常处理 | 是否吞异常、异常类型是否合理、失败路径是否明确 | `docs/CODING_RULES.md` | P0-P2 |
| 日志审计 | 核心交易是否有必要日志，日志是否脱敏 | `docs/CODING_RULES.md` | P1-P2 |
| 性能风险 | 是否存在循环查库、N+1 查询、大事务、重复远程调用 | `docs/PROJ_CONTEXT.md`、代码 diff | P1-P2 |
| 安全风险 | 敏感信息、权限绕过、SQL 注入、日志泄露、密钥输出 | `docs/CODING_RULES.md` | P0-P1 |
| 测试覆盖 | 是否覆盖正常、边界、异常和核心业务场景 | `docs/TESTING_GUIDE.md` | P1-P2 |
| 可维护性 | 是否过度复杂、重复代码、命名不清、上下文缺失 | `docs/PROJ_CONTEXT.md` | P2-P3 |
| 历史缺陷匹配 | 是否与历史缺陷模式相似 | `docs/DEFECT_CASES.md` | P1-P2 |

## 6. 高风险业务 Review 清单

| 主题 | 必查问题 | 未满足时建议等级 |
|---|---|---|
| 金额 | 是否使用精确数值类型，公式、精度、舍入和边界是否明确 | P0/P1 |
| 日期 | 是否明确起止边界、越界、闰年、时区或含首含尾规则 | P1 |
| 状态流转 | 是否校验允许流转和禁止流转，异常回退是否明确 | P0/P1 |
| 责任判断 | 产品责任、保险责任、核保或理赔规则是否有依据 | P0/P1 |
| 权限 | 是否存在越权访问、绕过审批或角色判断遗漏 | P0/P1 |
| 客户隐私 | 是否输出或记录未脱敏客户敏感信息 | P0 |
| 外部调用 | 是否有超时、异常处理、日志、幂等和补偿说明 | P1 |
| 事务与幂等 | 是否可能重复扣款、重复退费、重复批改或状态不一致 | P0/P1 |

## 7. 测试覆盖检查

| 检查项 | 通过条件 | 未满足时建议等级 |
|---|---|---|
| 单测结果 | 单测通过或失败原因清楚且不影响提交 | P1/P2 |
| 正常场景 | 主流程有测试覆盖 | P1/P2 |
| 边界场景 | 金额、日期、状态、数量、权限边界有覆盖 | P1 |
| 异常场景 | 外部失败、非法输入、状态非法、权限不足有覆盖 | P1 |
| Mock 策略 | 单元测试不访问真实外部系统或真实数据库 | P1 |
| 敏感信息 | 测试数据和日志不包含真实敏感信息 | P0 |

## 9. 豁免规则

| 场景 | 豁免条件 | 确认人 | 记录位置 |
|---|---|---|---|
| P0 问题 | 原则上不允许豁免 | 技术负责人和业务负责人共同确认 | Review 报告和 MR 说明 |
| P1 问题 | 风险可接受且有补救计划 | 技术负责人确认 | Review 报告 |
| P2 问题 | 不影响当前交付且已登记 | 模块 Owner 确认 | Review 报告或待办 |
| P3 建议 | 可自主选择 | 开发负责人确认 | 可选 |

## 10. 规则沉淀

Review 发现以下情况时，应提出文档更新建议：

1. 新的可复用编码规则：更新 `docs/CODING_RULES.md`。
2. 新的测试规则或缺失场景：更新 `docs/TESTING_GUIDE.md`。
3. 新的历史缺陷模式：更新 `docs/DEFECT_CASES.md`。
4. 新的模块、接口、状态、表或外部依赖：更新 `docs/PROJ_CONTEXT.md`。
5. 新的需求规则或澄清结论：更新 `docs/PRD_CONTEXT.md`。
