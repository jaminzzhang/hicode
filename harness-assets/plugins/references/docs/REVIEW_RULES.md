# REVIEW_RULES

## 1. 定位

代码审查通用总则和规则加载路由，不承载全部细则。

- 使用者：代码审查、提交检查、合并门禁 Agent

原则：

1. 关键禁令、分级和路由放在本文档前部。
2. Java、SQL、安全、保险业务细则放入 `docs/review-rules/`。
3. Prompt 或 Skill 只按 diff 和技术栈加载必要细则。
4. 若映射到 Copilot 等短指令工具，优先保留本文档前部内容。

## 2. 固定红线

以下问题必须阻断或不建议提交：

1. 敏感信息、密钥、Token、Cookie、Session、连接串、生产 IP 或生产凭证泄露。
2. 未脱敏客户数据、生产数据、生产日志进入代码、测试、报告或示例。
3. 越权访问、绕过审批、绕过认证或仅前端控制权限。
4. 金额、保单、批单、客户权益、收付费、责任判断、状态流转可能错误。
5. 自动合并、自动发布、自动修改生产配置或直接操作生产环境。
6. 删除测试、降低断言、跳过 Review 或隐藏风险来推动通过。

## 3. 必读输入

代码 diff、`docs/PRD_CONTEXT.md`、`docs/PROJ_CONTEXT.md`、`docs/CODING_RULES.md`、`docs/TESTING_GUIDE.md`、`docs/DOMAIN_KNOWLEDGE.md`、`docs/DEFECT_CASES.md`、编码计划、单测、CI 或静态扫描结果。

输入缺失列为 `待确认`，不得编造需求、业务规则、测试结果或系统行为。

## 4. 规则加载路由

先读本文档，再按 diff 补充读取：

| 触发条件 | 补充读取 |
|---|---|
| Java、Spring、事务、异常、日志、分层、外部调用 | `docs/review-rules/java.md` |
| SQL、DDL、DML、索引、批量更新、数据修复、配置脚本 | `docs/review-rules/sql.md` |
| 权限、认证、脱敏、日志、密钥、外部接口、客户敏感信息 | `docs/review-rules/security.md` |
| 投保、批改、退保、核保、理赔前置、收付费、责任、费率、保单/批单状态 | `docs/review-rules/insurance-domain.md` |

无法判断是否触发时，按风险更高的一侧处理，并说明依据。

## 5. 分级

| 等级 | 含义 | 处理要求 |
|---|---|---|
| P0 | 生产事故、资金/保单/客户权益错误、安全漏洞、重大合规问题、不可回滚发布 | 必须修复或给出阻断建议 |
| P1 | 核心流程异常、边界错误、数据不一致、性能问题、核心场景漏测 | 修复或负责人豁免 |
| P2 | 可维护性、规范性、日志、测试完整性或局部性能问题 | 建议修复并记录 |
| P3 | 风格、可读性或优化建议 | 可选采纳 |

## 6. 通用必查

1. 需求一致性：覆盖 `docs/PRD_CONTEXT.md` 的目标、范围、规则和验收标准。
2. 金融核心系统风险：保险核心业务逻辑、金额、交易一致性、状态、幂等、权限、审计、隐私、监管、生产变更和回滚。
3. 测试证据：正常、边界、异常和核心业务场景有测试或明确豁免。
4. 历史缺陷：命中 `docs/DEFECT_CASES.md` 中已复盘的问题模式。
5. 上下文缺口：需要补充领域、项目、需求或缺陷上下文。

## 7. 输出要求

必须输出：

1. 审查结论：通过 / 有条件通过 / 不建议提交 / 待确认。
2. 已读取的上下文和分场景规则文件。
3. P0/P1/P2/P3 问题清单。
4. 测试补充建议。
5. 是否建议进入提交检查。
6. 上下文或 Harness 资产更新建议。

问题清单格式：

| 等级 | 文件/位置 | 问题说明 | 依据 | 修复建议 | 责任人 |
|---|---|---|---|---|---|

## 8. 规则沉淀

发现新规则时，只输出更新建议，不直接写成已确认事实：

1. 通用分级或路由：建议更新 `docs/REVIEW_RULES.md`；技术负责人确认后沉淀。
2. Java、SQL、安全、保险业务细则：建议更新 `docs/review-rules/`；对应负责人确认后沉淀。
3. 编码、测试、缺陷、项目或需求上下文：建议更新对应 `docs/`；对应负责人确认后沉淀。
