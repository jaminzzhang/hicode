# 编码计划工作流

## 1. 触发条件

重点需求、跨模块需求、核心业务逻辑需求，或涉及数据库、接口、金额、日期、状态、外部系统、配置变更时，触发本工作流。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/PRD_CONTEXT.md`
3. `docs/PROJ_CONTEXT.md`
4. `docs/CODING_RULES.md`
5. `docs/TESTING_GUIDE.md`
6. `docs/DOMAIN_KNOWLEDGE.md`
7. `docs/DEFECT_CASES.md`
8. `docs/ADR/README.md`

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 需求上下文 | 当前需求目标、范围、业务规则、澄清点 |
| 仓库上下文 | 模块、接口、表、状态、外部依赖 |
| 相关代码片段 | 本次需求可能涉及的类、接口、配置、SQL |
| 历史缺陷 | 相似问题和防范规则 |

## 4. 执行步骤

1. 提炼需求摘要和改动目标。
2. 识别影响范围，包括系统、模块、类、接口、数据库、配置、批处理。
3. 按金融核心系统风险标准识别保险核心业务逻辑严谨性、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚风险。
4. 给出建议实现步骤。
5. 输出 TDD 测试重点和核心场景测试建议。
6. 判断是否需要新增或更新 ADR。
7. 提出上下文更新建议。

## 5. 输出

1. 编码计划。
2. 影响范围清单。
3. 建议修改点。
4. 核心实现步骤。
5. 测试重点。
6. 风险点。
7. 待确认问题。
8. ADR 触发判断。
9. `docs/PROJ_CONTEXT.md` 或 `docs/PRD_CONTEXT.md` 更新建议。

## 6. 质量标准

1. 重点需求必须给出 ADR 判断。
2. P0 待确认问题不得遗漏。
3. 影响范围必须覆盖代码、数据库、配置、接口和外部依赖。
4. 计划应能直接支撑 TDD 和辅助编码。

## 7. 安全约束

1. 不读取生产配置、密钥或真实客户数据。
2. 不直接修改代码。
3. 不自动生成已确认 ADR，只输出判断和草稿建议。

## 8. 下一步

1. 若编码计划无 P0 待确认问题，进入 `tdd.md`。
2. 若 ADR 判断为需要，提交技术负责人确认。
3. 若影响范围不清，先输出 `docs/PROJ_CONTEXT.md` 补充建议；经模块负责人或技术负责人确认后再沉淀。
