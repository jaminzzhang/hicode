# 提交检查工作流

## 1. 触发条件

Git 提交、创建 MR / PR 前，或开发完成准备进入提测前，触发本工作流。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/PRD_CONTEXT.md`
3. `docs/CODING_RULES.md`
4. `docs/TESTING_GUIDE.md`
5. `docs/REVIEW_RULES.md`
6. 编码计划
7. 代码审查结果

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| git diff | 本次提交变更 |
| 分支信息 | 当前分支和目标分支 |
| 关联需求/故事链接 | 提交对应需求 |
| 单测结果 | 测试通过情况 |
| 静态扫描/覆盖率结果 | 若项目已有 |
| 配置/SQL/脚本清单 | 若本次有变更 |

## 4. 执行步骤

1. 检查是否关联需求或故事。
2. 检查分支、变更范围和 MR 描述。
3. 检查单测、Review、覆盖率和静态扫描结果，确认代码审查已按 `docs/REVIEW_RULES.md` 路由加载必要的 `docs/review-rules/` 细则。
4. 检查金融核心系统风险基线是否已有结论，尤其是保险核心业务逻辑、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
5. 检查 SQL、配置、脚本、接口文档是否遗漏。
6. 检查是否存在敏感信息。
7. 输出提交检查报告和 MR 描述建议。

## 5. 输出

1. 提交前检查报告。
2. 是否建议提交。
3. 待修复问题。
4. MR 描述建议。
5. 风险提示。

## 6. 质量标准

1. V1 默认输出提醒型检查报告和建议动作。
2. 存在 P0 Review 问题时必须给出阻断建议。
3. 存在敏感信息、密钥或未脱敏生产数据时必须给出阻断建议。
4. SQL、配置、脚本变更必须提示发布影响。

## 7. 安全约束

1. 不自动提交、不自动合并。
2. 不输出密钥或客户敏感信息。
3. 不读取生产配置。

## 8. 下一步

1. 检查通过后进入提测或 MR Review。
2. 有条件通过时补充说明或修复问题。
3. 不建议提交时修复后重新检查。
