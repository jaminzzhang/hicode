# 发布准入门禁

## 1. 定位

本门禁用于发布申请前的提醒型检查，判断发布对象是否具备发布需求清单、分支、制品、测试、缺陷、SQL、配置、回滚和生产验证点等发布准入证据。

本门禁包装发布检查 Skill 的证据和结论，但统一使用门禁建议结论。它不替代发布负责人、研发负责人、测试负责人、安全负责人、审批流程、CI/CD 或发布平台的最终判断，不自动发布、不自动回滚、不自动合并、不修改生产配置、不操作生产环境。

## 2. 适用场景

使用本门禁的场景：

1. 发布申请提交前。
2. 发布检查 Skill 已输出报告，需要形成发布准入门禁结论。
3. 发布范围、分支、制品、SQL、配置、测试报告、缺陷、回滚方案或生产验证点发生变化后，需要重新核对发布准入证据。
4. 存在未关闭 P0/P1 缺陷、测试失败、核心场景缺失、SQL/配置/回滚缺失、敏感信息或生产越权风险，需要判断是否建议阻断发布申请。

不适用场景：

1. 当前目标是生成核心场景测试报告；应先使用 `core-scenario-test` Skill。
2. 当前目标是执行发布前汇总检查；应先使用 `release-check` Skill。
3. 当前目标是 MR 合并前检查；应使用 `merge-gate.md`。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
5. 用户要求自动发布、自动回滚、修改生产配置、连接生产环境、读取生产日志、执行生产 SQL 或输出生产操作命令。

## 3. 输入材料

至少需要：

1. 神兵版本链接或版本标识、发布申请摘要、发布目标和发布影响范围。
2. 发布需求清单，包括本次发布范围和本次不发布范围说明。
3. 代码分支、目标分支、Commit/MR 清单、构建记录和制品版本。
4. 发布检查 Skill 报告。
5. 核心场景测试报告、测试报告、覆盖率、CI 结果和缺陷关闭情况。
6. SQL、配置、脚本、接口文档、发布说明和负责人确认记录，如适用。
7. 回滚方案，包括触发条件、回滚路径、前置条件和负责人角色。
8. 生产验证点和上线后观察建议，包括验证目标、预期结果、观察窗口和人工确认的数据来源类型。
9. 代码审查、提交检查、人工 Review、风险接受或豁免记录，如已有。

缺少输入时，必须在报告中标注缺口，不得编造发布范围、需求清单、分支、制品、测试结果、缺陷状态、SQL、配置、回滚方案、验证点、负责人确认或发布结论。

## 4. 发布检查结论映射

发布检查 Skill 使用发布侧结论；本门禁统一转换为门禁建议结论：

| 发布检查 Skill 结论 | 门禁建议结论 | 使用规则 |
|---|---|---|
| 建议发布 | 建议通过 | 证据齐备且未发现阻断建议项 |
| 有条件发布 | 有条件通过 | 无阻断建议项，但存在需补充材料、确认或风险接受的问题 |
| 不建议发布 | 建议阻断 | 存在 P0/P1、测试失败、SQL/配置/回滚缺失、敏感信息、生产越权或关键证据缺失 |
| 待确认 | 待确认 | 发布对象、范围、证据或负责人确认不足，无法形成有效建议 |

不得在门禁报告中使用“批准发布”“允许发布”“最终审批通过”或“自动发布”等表达。

## 5. 建议结论规则

| 结论 | 使用条件 |
|---|---|
| 建议通过 | 发布对象和范围明确；发布检查 Skill 结论为建议发布；发布需求清单完整；分支、Commit/MR 和制品一致；测试通过且核心场景证据可追溯；无未关闭 P0/P1 缺陷；SQL 和配置齐备或明确不适用；回滚方案明确；生产验证点明确；未发现敏感信息、密钥、未脱敏生产数据或生产越权 |
| 有条件通过 | 无阻断建议项，但存在 P2/P3 待处理、材料归档、非阻断缺陷、负责人风险接受、生产验证点补充或上线后观察增强要求 |
| 建议阻断 | 发布范围、需求清单或制品信息缺失；发布检查 Skill 结论为不建议发布；测试失败未解释；核心场景缺失；存在未关闭 P0/P1 缺陷；SQL、配置或回滚方案缺失且适用；分支与制品不一致；发现敏感信息、密钥、未脱敏生产数据、生产越权、自动发布或自动回滚诉求 |
| 待确认 | 发布对象、发布范围、发布需求清单、分支、制品、测试证据、缺陷状态、SQL、配置、回滚或生产验证点材料不足，无法判断是否阻塞 |

## 6. 检查项

| 检查项 | 通过条件 | 阻断或降级规则 |
|---|---|---|
| 发布对象明确 | 有发布申请、版本标识、发布目标和影响范围 | 缺失时结论为待确认 |
| 发布需求清单 | 发布需求完整，本次发布和不发布范围明确 | 缺失时建议阻断或待确认 |
| 分支和制品 | 代码分支、Commit/MR、构建记录和制品版本一致 | 不一致时建议阻断 |
| 发布检查报告 | 已提供发布检查 Skill 报告，且结论可映射 | 缺失时待确认；不建议发布时建议阻断 |
| 测试和核心场景 | 测试通过，核心场景测试报告可追溯 | 测试失败未解释或核心场景缺失时建议阻断 |
| 缺陷状态 | 无未关闭 P0/P1 缺陷；P1 豁免必须有负责人确认和后续动作 | P0/P1 未关闭且无确认时建议阻断 |
| SQL | SQL 清单、执行顺序、影响范围、回滚 SQL 或补救方案齐备，或明确不适用 | 适用但缺失时建议阻断 |
| 配置 | 配置清单、环境差异、生产人工确认和配置回退路径齐备，或明确不适用 | 适用但缺失时建议阻断 |
| 回滚方案 | 触发条件、回滚路径、前置条件、负责人角色和验证方式明确 | 缺失时建议阻断 |
| 生产验证点 | 验证目标、预期结果、观察窗口、责任角色和人工确认数据来源类型明确 | 缺失时建议阻断或有条件通过 |
| 金融核心系统风险基线 | 保险核心业务逻辑、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚风险已有检查结论 | 涉及相关域但无结论时建议阻断或待确认 |
| 敏感信息与生产越权 | 无未脱敏客户敏感信息、生产数据、密钥、生产连接、生产 SQL、生产日志查询或自动发布/回滚诉求 | 命中时建议阻断并要求脱敏或转人工安全流程 |

## 7. 受限命令边界

本门禁优先消费发布检查 Skill、测试报告、CI、缺陷系统、SQL/配置清单、回滚方案、生产验证计划和人工确认记录。若目标项目已有命令、文档或用户授权，本门禁可记录或执行以下受限命令：

1. `git status`
2. `git log`
3. `git diff --stat`
4. 本地构建结果复核
5. 本地测试结果复核
6. 本地静态检查或格式检查结果复核

执行前提：

1. 命令来自项目已有脚本、文档、工具配置或用户明确确认。
2. 命令运行在目标项目本地非生产环境。
3. 命令不连接生产、不读取生产日志、不调用生产接口、不读取密钥、不处理未脱敏生产数据。
4. 命令不执行发布、回滚、数据库变更、生产连接、提交、推送、合并或删除未确认文件。

本门禁不执行生产验证命令。生产验证只能输出验证计划和人工确认入口。

## 8. 生产验证计划边界

可以输出：

1. 验证点、验证目标和预期结果。
2. 观察窗口、责任角色和人工确认的数据来源类型。
3. 上线后观察项、观察周期、异常处理路径和责任角色。

禁止输出：

1. 生产命令、生产 SQL、生产日志查询语句或生产接口调用命令。
2. 生产 IP、账号、Token、配置值、连接串或内部密钥。
3. 真实客户、保单、证件、手机号、银行卡或地址信息。
4. 可直接执行的生产回滚命令。

必须依赖生产数据时，只能提示发布负责人在既有平台中脱敏确认。

## 10. 安全约束

1. 不读取或输出真实客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不将发布准入门禁结论写成最终发布审批、发布平台批准或允许发布。
4. 不自动发布、自动回滚、自动合并、修改生产配置或操作生产环境。
5. 不输出生产命令、生产 SQL、生产日志查询、生产接口调用、生产配置值或真实客户/保单信息。
6. 不为了推动发布而隐藏 P0/P1 风险、忽略测试缺口、跳过回滚方案或弱化敏感信息风险。

## 11. 验收要求

本门禁资产验收时应确认：

1. 覆盖发布需求清单、分支、制品、测试、缺陷、SQL、配置、回滚和生产验证点。
2. 明确发布检查 Skill 结论到门禁建议结论的映射。
3. 明确 Agent 不自动发布，只提供检查结论和风险建议。
4. 阻断建议项与普通风险提示项分离。
5. 受限命令边界明确，不授权生产操作、自动发布、自动回滚或生产配置修改。
6. 保留审计证据、待确认问题和下一步建议。
