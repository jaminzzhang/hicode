# 门禁资产目录

## 1. 定位

`harness-assets/gates/` 用于维护 V1 节点门禁规则和报告模板源资产。后续安装到目标项目后，对应目标项目 `.ai-harness/gates/`。

V1 门禁是提醒型、可人工执行、可审计的 Markdown 资产。门禁报告提供标准化检查依据、风险提示和阻断建议，不替代需求负责人、研发负责人、测试负责人、发布负责人、安全负责人或既有流程的最终审批。

本目录不实现自动化 Agent、CI/CD 插件、发布平台审批、自动合并、自动发布或生产操作。

## 2. P5-WP1 交付边界

P5-WP1 只交付：

1. `README.md`：门禁目录规范、统一结论口径、后续门禁路由和安全边界。
2. `_gate-template.md`：通用节点门禁报告模板。

P5-WP1 不创建具体门禁规则文件。以下文件由后续工作包按顺序创建：

| 工作包 | 文件 | 门禁节点 | 主要输入来源 |
|---|---|---|---|
| P5-WP2 | `requirement-entry-gate.md` | 需求进入开发前 | 需求评审 Skill、`docs/PRD_CONTEXT.md`、澄清问题状态 |
| P5-WP2 | `coding-entry-gate.md` | 编码开始前 | 编码计划 Skill、ADR 判断、上下文清晰门槛、P0 待确认问题 |
| P5-WP3 | `coding-to-test-gate.md` | 开发进入提测前 | TDD、辅助编码结果、代码审查、提交检查、核心场景测试建议 |
| P5-WP3 | `merge-gate.md` | MR 合并前 | AI Review、人工 Review、CI、覆盖率、未关闭 P0/P1 问题 |
| P5-WP4 | `release-gate.md` | 发布申请前 | 发布检查 Skill、测试报告、缺陷、SQL、配置、回滚、生产验证计划 |

## 3. 统一结论

门禁报告只能输出建议性质的结论：

| 结论 | 使用条件 | 禁止误用 |
|---|---|---|
| 建议通过 | 检查对象、范围和证据齐备，未发现阻断建议项 | 不代表最终审批通过 |
| 有条件通过 | 无阻断建议项，但存在需补充确认、补测、补材料或风险接受的问题 | 不得用于绕过负责人确认 |
| 建议阻断 | 存在 P0/P1、高风险安全、敏感信息、密钥、未脱敏生产数据、生产越权、自动合并/发布、关键证据缺失等问题 | 不代表 Agent 自动拦截流程 |
| 待确认 | 检查对象、范围、输入材料或证据不足，无法形成有效建议 | 不得编造检查结论 |

禁止使用“批准通过”“最终通过”“强制拦截”“允许发布”“已完成审批”等表达。

## 4. 阻断建议项与风险提示项

门禁报告必须分离：

1. 阻断建议项：P0/P1、高风险安全、敏感信息、密钥、未脱敏生产数据、生产越权、自动合并/发布、关键证据缺失等不应继续推进的问题。
2. 普通风险提示项：P2/P3、材料补充、验证增强、流程提醒或可接受风险说明。

阻断建议项必须写明解除条件和建议确认人。普通风险提示项必须写明影响、建议动作和是否需要人工接受。

## 5. 默认命令边界

通用门禁模板默认不授权执行命令。门禁报告优先记录来自 Prompt、Skill、CI、测试报告、Review 报告、提交检查报告、发布检查报告和人工确认的证据。

后续具体门禁如需允许受限命令，必须在对应门禁文件中单独声明命令范围、执行条件、禁止事项和审计记录要求。

任何门禁均不得执行：

1. 自动提交、推送、合并、发布或回滚。
2. 生产连接、生产 SQL、生产日志查询、生产接口调用或生产配置修改。
3. 数据库变更、删除未确认文件、读取 `.env`、读取密钥文件或读取生产凭证。
4. 处理未脱敏客户敏感信息或未脱敏生产数据。

## 6. 报告模板

所有具体门禁报告应基于 `_gate-template.md`，并至少保留以下信息：

1. 门禁节点和检查对象。
2. 建议性质结论。
3. 输入材料和证据来源。
4. 检查项明细。
5. 阻断建议项。
6. 普通风险提示项。
7. 受限命令执行记录。
8. 审计证据。
9. 待确认问题。
10. 下一步建议。

具体门禁可以在模板基础上增加节点专属检查项，但不得删除安全约束、阻断建议项、受限命令记录和审计证据章节。

## 7. 安全约束

门禁资产必须遵守：

1. 不读取或输出生产账号、密码、Token、Cookie、Session、连接串、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不自动合并 MR / PR，不自动发布，不自动回滚。
5. 不把 AI 门禁建议写成负责人最终审批。
6. 不通过删除测试、降低断言、跳过 Review 或隐藏风险推动流程通过。

## 8. 维护规则

新增或修改门禁时：

1. 优先复用 `_gate-template.md`。
2. 结论枚举必须保持一致。
3. 涉及稳定术语时，先更新根目录 `CONTEXT.md`。
4. 涉及权限、命令或审计边界时，同步检查 `harness-assets/docs/TOOL_PERMISSION_AUDIT_MATRIX.md` 是否需要后续更新。
5. 示例、测试数据和报告样本必须脱敏，不冒充真实试点结果。
