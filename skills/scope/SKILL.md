# hicode scope

## 定位

`hicode:scope` 是需求到编码前的澄清和拆解入口。它帮助 Coding Agent 与用户一起评审需求、追问模糊点、确定范围边界、识别金融核心系统风险，并把可实现范围拆成小任务，作为后续 `hicode:tdd` 的输入。

本 Skill 不生成业务实现代码，不修改本地代码，不进入 TDD，不输出最终审批。

## 必读规则

执行前按需读取：

1. `../../references/rules/coding_rules.md`
2. `../../references/templates/scope/scope-report.md`

同时读取目标项目中与当前需求直接相关的入口规则、PRD、项目上下文、领域知识、历史缺陷、ADR、代码结构说明或已脱敏材料。

不得读取历史资产原文、归档资产、旧 Prompt、旧 Gate、旧 Schema 或旧细粒度 Skill 作为当前规则源。

## 核心原则

1. 先澄清，再计划；先拆小，再编码。
2. 一次只问一个关键问题，除非用户明确要求一次性列出问题清单。
3. 能从本地文档、代码、测试或已确认记录查证的问题，先查证，不把可查证问题丢给用户。
4. 每个追问都必须给出推荐答案、推荐理由和对范围或风险的影响。
5. 对模糊词、冲突术语、隐含范围和过大任务立即收敛。
6. 涉及保险/金融核心系统时，默认按高风险标准检查业务规则、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
7. 任务拆分必须避免“一次生成一大堆代码”；每个实施小任务应能独立理解、独立验证、独立回滚或重新执行。

## 执行流程

### 1. 判断任务类型

先判断用户当前需要的是：

1. 需求评审：判断目标、规则、验收标准和风险是否清楚。
2. 需求梳理：与用户确认边界、术语、异常、非目标和待确认点。
3. 范围收敛：把过大、过散或跨系统需求拆成可独立推进的范围。
4. 实施任务拆分：把已确认范围拆成后续 TDD/编码可执行的小任务。
5. 编码准入检查：判断是否具备进入 `hicode:tdd` 的输入证据。

若用户实际要求编码实现、测试先行、Review 或发布检查，应先完成 Scope 必要澄清，再建议转到 `hicode:tdd`、`hicode:review` 或 `hicode:release`。

### 2. 探索上下文

先查找并读取与当前需求直接相关的材料：

1. 用户提供的需求摘要、PRD、故事链接、会议纪要或脱敏材料。
2. 目标项目入口规则，例如 `AGENTS.md`、`CLAUDE.md`。
3. 目标项目上下文，例如 `docs/PRD_CONTEXT.md`、`docs/PROJ_CONTEXT.md`、`docs/DOMAIN_KNOWLEDGE.md`、`docs/DEFECT_CASES.md`。
4. 相关 ADR、规则文档、接口说明、测试说明和代码结构说明。
5. 与需求直接相关的代码、测试、配置、SQL、批处理或外部依赖说明。

不得读取 `.env`、密钥文件、生产配置、生产凭证、未脱敏客户信息或未脱敏生产数据。

### 3. 评审需求清晰度

必须检查：

1. 需求目标是否能用一句话表达。
2. 范围内、范围外和非目标是否明确。
3. 验收标准是否可测试、可观察、可复核。
4. 业务规则、异常路径、边界值、数据来源和负责人是否可追溯。
5. 关键术语是否与项目上下文冲突。
6. 是否涉及金额、状态机、幂等、并发、事务、权限、审计、隐私、监管、生产变更或回滚。
7. 影响范围是否能定位到模块、接口、类、方法、表、配置、SQL、批处理或外部系统。

发现冲突或模糊点时，先说明证据，再提出一个最关键问题，并给出推荐答案。

### 4. 追问与边界收敛

当需求不能安全进入计划时，进入追问循环：

1. 每轮只问一个会改变范围、风险或实现路径的问题。
2. 问题必须包含：
   - 当前证据。
   - 不确定点。
   - 推荐答案。
   - 选择该答案的理由。
   - 若用户不同意，可能影响哪些范围、风险或后续任务。
3. 用户回答后，立刻更新本轮需求摘要、范围边界、风险判断和待确认问题状态。
4. 若回答产生新术语或稳定概念，只输出上下文更新建议，不把未确认推断写入长期上下文。
5. 若需求过大，先拆成独立子需求，并推荐优先处理的第一个子需求。

不要为了显得完整一次性追问大量问题；优先解决会阻断编码准入的最高风险问题。

### 5. 比较方案

当需求有多种合理实现边界或拆分方式时，提出 2-3 个方案：

1. 推荐方案放在最前。
2. 每个方案说明适用条件、收益、代价、风险和不选原因。
3. 对金融核心系统高风险变更，必须说明哪个方案更容易验证、审计和回滚。
4. 用户未确认方案前，不生成最终实施任务清单。

### 6. 拆分实施小任务

只有在目标、范围、关键规则和风险边界足够清楚后，才能拆任务。

任务拆分规则：

1. 每个任务只服务一个明确结果。
2. 每个任务应能在小范围内完成，避免跨多个独立子系统的大改动。
3. 每个任务必须包含输入、输出、涉及文件或待定位对象、验证方式和停止条件。
4. 每个任务都要能追溯到需求、上下文、代码证据或用户确认。
5. 涉及代码改动的任务必须转交 `hicode:tdd`，并带上测试重点。
6. 不允许写“完善异常处理”“补充测试”“实现相关逻辑”这类不可验收任务。

推荐任务格式：

```markdown
### Task N: 任务名称

- 目标：一句话说明本任务完成什么。
- 输入：需求片段、确认结论、上下文或代码证据。
- 范围内：具体文件、模块、接口、表或待定位对象。
- 范围外：本任务明确不处理的内容。
- TDD 起点：先写或先补的测试场景。
- 验证方式：要运行或人工复核的检查。
- 停止条件：遇到哪些缺口必须回到 Scope 澄清。
```

### 7. 判断停止条件

命中以下情况时，不得建议进入编码：

1. 需求目标、范围、验收标准或核心业务规则不清。
2. 关键术语与项目上下文冲突且未确认。
3. P0/P1 风险未关闭，或无法排除高风险。
4. 影响范围无法定位到可执行对象。
5. 需要 ADR 但未形成决策草案或确认路径。
6. 任务仍然过大，无法拆成可独立验证的小任务。
7. 输入包含未脱敏客户信息、生产数据、密钥、生产账号、生产连接串或生产操作诉求。

### 8. 形成 Scope 结果

输入充分时，输出：

1. 需求评审结论。
2. 已确认的需求摘要。
3. 范围内、范围外和非目标。
4. 关键业务规则与验收标准。
5. 影响范围和证据。
6. 风险清单与阻断建议。
7. 已关闭和未关闭澄清问题。
8. 推荐方案与取舍。
9. 小任务实施计划。
10. TDD 输入和测试重点。
11. ADR 判断。
12. 上下文更新建议。

## 输出要求

默认使用 Markdown，包含：

1. 建议结论：`READY_FOR_TDD`、`NEEDS_CONFIRMATION`、`SPLIT_REQUIRED` 或 `BLOCKED`。
2. 最高风险等级：`P0`、`P1`、`P2`、`P3` 或 `NONE`。
3. 依据和输入缺口。
4. 需求摘要、范围内、范围外和非目标。
5. 澄清问题队列：已关闭、待用户回答、建议确认人。
6. 影响范围和证据。
7. 风险与阻断建议。
8. 推荐方案与取舍。
9. 小任务实施计划。
10. TDD 输入和测试重点。
11. ADR 判断。
12. 上下文更新建议。

不得输出“准许编码”“审批通过”“可以上线”或替代负责人判断。
