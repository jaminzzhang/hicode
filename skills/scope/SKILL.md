# hicode scope

## 定位

`hicode:scope` 是需求进入编码前的澄清、评审和拆分入口。它负责把用户想做的事变成可审查、可验证、可移交给 `hicode:tdd` 的小任务。

本 Skill 不写业务实现代码，不进入 TDD，不替代产品、研发、测试、架构或安全负责人审批。

## 必读材料

按需读取，不默认全量加载：

本 Skill 内置材料：

1. `feature_context.md`
2. `requirement-review-report.md`
3. `scope-report.md`
4. `task-split-plan.md`
5. `ADR-template.md`

目标项目事实文档：

1. 当前需求直接相关的入口规则、PRD、故事、会议纪要和需求材料。
2. `docs/rules/` 下由 `hicode:init` 创建或更新的项目规则文件，以及目标项目已有的等价规则文件；缺少时提示先补 `hicode:init`，不得读取本仓库规则替代。
3. `docs/features/<feature-id>/feature_context.md`、需求评审报告、Scope 报告和任务拆分计划；存在时读取目标项目文件，不读取模板代替事实。
4. `docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md`、`docs/adr/`、历史缺陷、代码结构说明、相关代码、测试、配置、SQL、批处理或外部依赖说明。

模板只在目标项目对应文档缺失且需要创建草稿时读取；不得把模板内容当成目标项目事实或已确认上下文。

不得读取 `.env`、密钥文件、生产配置、生产凭证、未脱敏客户信息、未脱敏生产数据或生产日志原文。

## 快速示例

用户说“帮我梳理这个需求，拆成可编码任务”时，先确认 `feature-id`，读取现有需求材料和 `feature_context.md`，做需求评审；对阻断编码的问题一次只问一个并给推荐答案；目标、边界和风险足够清楚后，再拆成可交给 `hicode:tdd` 的小任务。

## 核心原则

1. 先澄清，再计划；先拆小，再编码。
2. 能从本地文档、代码或已确认记录查证的问题，先查证，不直接丢给用户。
3. 一次只问一个会改变范围、风险或实现路径的关键问题，除非用户要求问题清单。
4. 每个追问都给出当前证据、推荐答案、理由，以及对范围或风险的影响。
5. 对模糊词、冲突术语、隐含范围和过大任务立即收敛。
6. 金融核心系统相关需求默认检查业务逻辑、金额、交易一致性、状态流转、幂等、权限、审计、隐私、监管、生产变更和回滚。
7. `feature_context.md` 可作为单需求过程上下文更新；`DOMAIN_KNOWLEDGE.md`、`PROJ_CONTEXT.md` 和 ADR 正式写入前必须由用户或负责人确认。

## 执行流程

### 1. 固定需求目录

按目标项目 `AGENTS.md` 或 `CLAUDE.md` 中的 hicode 单需求文档生命周期执行；入口缺少 hicode section 时，先提示补 `hicode:init`。目录固定为 `docs/features/<feature-id>/`；`feature-id` 不明确时先问用户；缺少 `feature_context.md`、`requirement-review-report.md`、`scope-report.md` 或 `task-split-plan.md` 时，先读取本 Skill 内置模板再按需创建。

Scope 只写已确认事实、证据、待确认问题、范围边界、风险判断、ADR 草稿和拆分任务；不得编造负责人、业务规则、验收结论或验证结果。

### 2. 判断 Scope 模式

先判断当前任务属于哪类：

| 模式 | 目标 |
|---|---|
| 需求评审 | 判断目标、规则、验收标准和风险是否清楚 |
| 需求澄清 | 确认术语、边界、异常、非目标和待确认点 |
| 范围收敛 | 把过大或跨系统需求拆成可独立推进的范围 |
| 任务拆分 | 把已确认范围拆成 TDD 可执行小任务 |
| 编码准入 | 判断是否具备进入 `hicode:tdd` 的证据 |

若用户实际要求编码、测试、Review 或发布检查，先完成必要 Scope，再建议转到 `hicode:tdd`、`hicode:review` 或 `hicode:release`。

### 3. 需求评审

必须检查：

1. 需求目标是否能用一句话表达。
2. 范围内、范围外和非目标是否明确。
3. 验收标准是否可测试、可观察、可复核。
4. 业务规则、异常路径、边界值、数据来源和负责人是否可追溯。
5. 关键术语是否与项目上下文冲突。
6. 是否涉及金额、状态机、幂等、并发、事务、权限、审计、隐私、监管、生产变更或回滚。
7. 影响范围是否能定位到模块、接口、类、方法、表、配置、SQL、批处理或外部系统。

需求评审结束后，输出或更新 `requirement-review-report.md`。

### 4. 澄清与范围确认

当需求不能安全进入计划时，进入追问循环：

1. 每轮只问一个会改变范围、风险或实现路径的问题。
2. 问题包含：当前证据、不确定点、推荐答案、推荐理由、不同选择的影响。
3. 用户回答后，更新需求摘要、范围边界、风险判断和待确认状态。
4. 若回答产生稳定术语或上下文，只输出更新建议；未确认内容不得写入长期上下文。
5. 若需求过大，先拆成独立子需求，并推荐优先处理的第一个子需求。

范围确认时按需更新目标项目 `docs/features/<feature-id>/feature_context.md`。需要 ADR 时，只有同时满足难逆、无上下文会意外、有真实取舍三个条件，才基于 `ADR-template.md` 生成 ADR 草稿。

### 5. 比较方案

存在多种合理边界或拆分方式时，给 2-3 个方案：

1. 推荐方案放最前。
2. 每个方案说明适用条件、收益、代价、风险和不选原因。
3. 金融核心系统高风险变更必须说明哪个方案更容易验证、审计和回滚。
4. 用户未确认方案前，不生成最终任务清单。

### 6. 拆分小任务

只有目标、范围、关键规则和风险边界足够清楚后，才能拆任务。

每个任务必须包含：

1. 目标：一句话说明完成什么。
2. 输入：需求片段、确认结论、上下文或代码证据。
3. 范围内和范围外。
4. 涉及文件、模块、接口、表或待定位对象。
5. TDD 起点和测试重点。
6. 验证方式。
7. 停止条件：遇到哪些缺口必须回到 Scope。

任务必须小、可独立理解、可独立验证、可独立回滚或重新执行。不得写“完善相关逻辑”“补充测试”“处理异常”这类不可验收任务。

拆分完成后输出或更新 `task-split-plan.md`。

## 分流与停止条件

命中以下情况时，不得输出 `READY_FOR_TDD`：

1. 需求目标、范围、验收标准或核心业务规则不清。
2. 关键术语与项目上下文冲突且未确认。
3. P0/P1 风险未关闭，或无法排除高风险。
4. 影响范围无法定位到可评审、可拆分或可验证的对象。
5. 需要 ADR 但未形成草稿或确认路径。
6. 任务仍然过大，无法拆成可独立验证的小任务。
7. 输入包含未脱敏客户信息、生产数据、密钥、生产账号、生产连接串或生产操作诉求。

命中安全红线时停止 Scope，输出阻断建议并转人工安全流程。

## 输出要求

默认输出 Markdown，包含：

1. 建议结论：`READY_FOR_TDD`、`NEEDS_CONFIRMATION`、`SPLIT_REQUIRED` 或 `BLOCKED`
2. 最高风险等级：`P0`、`P1`、`P2`、`P3` 或 `NONE`
3. 依据、已读材料和输入缺口
4. 需求评审报告
5. `feature_context.md` 更新记录
6. 需求摘要、范围内、范围外和非目标
7. 澄清问题：已关闭、待回答、建议确认人
8. 影响范围、风险和阻断建议
9. 推荐方案与取舍
10. 拆分任务计划
11. TDD 输入和测试重点
12. ADR 判断、草稿或更新建议
13. `DOMAIN_KNOWLEDGE.md` 与 `PROJ_CONTEXT.md` 更新结果或待确认建议
14. 本次创建、更新、跳过或缺失的 feature 文档清单

不得输出“准许编码”“审批通过”“可以上线”或替代负责人判断。
