# hicode scope

## 定位

`hicode:scope` 是需求进入编码前的收敛入口。它用设计树把用户想做的事整理成可审查、可验证、可移交给 `hicode:tdd` 的准入证据和小任务。

设计树的根是业务目标，主干是正常逻辑闭环，分支是异常、边界、状态、权限、幂等、回滚等处理方案，叶子是可独立执行的 TDD 任务。

本 Skill 不写业务实现代码，不进入 TDD，不替代产品、研发、测试、架构或安全负责人审批。

## 读取边界

按需读取，不默认全量加载：

1. 目标项目入口规则、当前需求材料、PRD、故事、会议纪要和用户确认记录。
2. 目标项目 `docs/rules/`，以及已有等价规则文件；缺少时提示先补 `hicode:init`，不得读取本仓库规则替代。
3. 目标项目 `docs/features/<feature-id>/feature_context.md` 和 `scope-plan.md`；存在时读取目标项目文件，不读取模板代替事实。
4. 目标项目 `docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md` 的 Feature 索引、`docs/adr/`、历史缺陷、代码结构说明、相关代码、测试、配置、SQL、批处理或外部依赖说明。

本 Skill 内置模板只在目标项目对应文档缺失且需要创建草稿时读取：

1. `feature_context.md`
2. `scope-plan.md`
3. `ADR-template.md`

不得读取 `.env`、密钥文件、生产配置、生产凭证、未脱敏客户信息、未脱敏生产数据或生产日志原文。

## Scope 设计树闭环

### 1. 固定需求目录和证据

按目标项目 `AGENTS.md` 或 `CLAUDE.md` 中的 hicode 单需求文档生命周期执行；入口缺少 hicode section 时，先提示补 `hicode:init`。目录固定为 `docs/features/<feature-id>/`；`feature-id` 不明确时，先用 `docs/PROJ_CONTEXT.md` 的 Feature 索引定位候选，再向用户确认。

完成标准：

1. `feature-id`、候选 Feature 索引命中情况、需求目录、已读材料和输入缺口已列明。
2. 目标项目已有 feature 文档优先读取；缺少文档时使用本 Skill 同名模板创建草稿。
3. 只写已确认事实、证据、待确认问题、范围边界、风险判断、ADR 草稿和拆分任务；不得编造负责人、业务规则、验收结论或验证结果。

### 2. 评审需求准入

先判断当前任务重点：需求评审、需求澄清、范围收敛、任务拆分或编码准入。用户实际要求编码、测试、Review 或发布检查时，先完成必要 Scope，再建议转到 `hicode:tdd`、`hicode:review` 或 `hicode:release`。

必须逐项检查：

1. 需求目标是否能用一句话表达。
2. 范围内、范围外和非目标是否明确。
3. 验收标准是否可测试、可观察、可复核。
4. 业务规则、异常路径、边界值、数据来源和负责人是否可追溯。
5. 关键术语是否与项目上下文冲突。
6. 是否涉及高严谨领域业务规则、金额或关键数值、交易或数据一致性、状态流转、幂等、并发、事务、权限、审计、隐私、适用监管与合规、生产变更或回滚。
7. 影响范围是否能定位到模块、接口、类、方法、表、配置、SQL、批处理或外部系统。
8. 输入是否足以构建设计树：业务目标、主流程、关键分支、边界条件、验证口径至少有证据或待确认项。

完成标准：输出或更新 `scope-plan.md` 中的需求准入评审，并给出最高风险等级、证据缺口、待确认问题和是否可进入设计树收敛或拆分。

复杂或高风险时，按需委托 `requirement-reviewer` 检查需求完整性和高严谨业务风险，委托 `coding-planner` 检查影响范围、实现切片、TDD 重点和 ADR 判断。子 Agent 不可用时由本 Skill 直接执行，并说明降级影响；不得因此降低 P0/P1 风险等级。

### 3. 构建并收敛设计树

围绕业务目标，参考行业的专家方案和最佳实践，构建设计树，不直接跳到任务清单。

设计树必须覆盖：

1. 根：业务目标、成功结果、验收口径和非目标。
2. 主干：正常路径的触发条件、输入、处理步骤、状态或数据变化、输出和验证点，形成逻辑闭环。
3. 分支：异常路径、边界值、幂等/并发、权限/审计、状态回退、外部依赖失败、配置/SQL/脚本和回滚处理方案。
4. 边界：范围内、范围外、延期处理、明确不做和需要负责人确认的部分。
5. 证据：每个节点的需求来源、项目上下文、规则依据、风险等级和测试切入点。

设计树节点不得只写“处理异常”“校验数据”这类不可验收描述；必须能回答“什么条件触发、怎么处理、结果是什么、如何验证”。

完成标准：`scope-plan.md` 已包含设计树方案；每个主干和分支节点都有边界、证据、风险和测试切入点。缺少关键节点时，不进入 TDD 拆分。

### 4. 澄清并收敛设计树

当设计树不能安全闭合时，进入追问循环：

1. 每轮只问一个会改变范围、风险或实现路径的问题。
2. 问题包含：当前证据、不确定点、推荐答案、推荐理由、不同选择的影响。
3. 用户回答后，更新需求摘要、设计树节点、范围边界、风险判断和待确认状态。
4. 若回答产生稳定术语或上下文，只输出 `DOMAIN_KNOWLEDGE.md`、`PROJ_CONTEXT.md` 或 ADR 更新建议；正式写入前必须由用户或负责人确认。
5. 若需求过大，先拆成独立子需求，并推荐优先处理的第一个子需求。

存在多种合理设计树方案、边界或拆分方式时，给 2-3 个方案：

1. 推荐方案放最前。
2. 每个方案说明主干逻辑、分支处理、范围边界、收益、代价、风险和不选原因。
3. 高严谨业务系统的高风险变更必须说明哪个方案更容易验证、审计和回滚。
4. 用户未确认设计树方案前，不生成最终 TDD 任务清单。

完成标准：`feature_context.md` 已按需更新或给出草稿；每个未关闭问题都有风险等级、影响、推荐答案和建议确认人。需要 ADR 时，只有同时满足难逆、无上下文会意外、有真实取舍三个条件，才基于 `ADR-template.md` 生成 ADR 草稿。

### 5. 从设计树拆分 TDD 小任务

只有设计树的业务目标、主干、关键分支、范围边界和风险足够清楚后，才能拆任务。

每个任务必须包含：

1. 目标：一句话说明完成什么。
2. 对应设计树节点：主干节点或分支节点。
3. 输入：需求片段、确认结论、上下文或代码证据。
4. 范围内和范围外。
5. 涉及文件、模块、接口、表或待定位对象。
6. TDD 起点和测试重点。
7. 验证方式。
8. 停止条件：遇到哪些缺口必须回到 Scope。

任务必须小、可独立理解、可独立验证、可独立回滚或重新执行。每个设计树主干和已确认分支都必须被任务覆盖、明确排除或标注待确认。不得写“完善相关逻辑”“补充测试”“处理异常”这类不可验收任务。

完成标准：输出或更新 `scope-plan.md` 中的 TDD 任务计划；每个任务都能交给 `hicode:tdd` 独立执行，且不会依赖未确认业务规则或隐藏范围。

### 6. 给出准入结论

只在以下条件全部满足时输出 `TDD_INPUT_READY`：

1. 需求目标、范围、非目标、验收标准和核心业务规则清楚，并有证据来源。
2. 关键术语未与项目上下文冲突，或冲突已确认处理方式。
3. 高严谨业务系统风险基线已逐项评审，无未关闭 P0/P1 阻断问题。
4. 设计树已闭合：主干、已确认分支、范围边界、风险和验证点都有证据或明确处理。
5. 影响范围能定位到可评审、可拆分、可验证的对象。
6. ADR 判断已给出；需要 ADR 时已有草稿或确认路径。
7. 任务已从设计树拆到可独立理解、验证、回滚或重新执行。
8. 目标项目 feature 文档的创建、更新、跳过或缺失状态已列明。

否则按以下矩阵选择结论：

| 场景 | 建议结论 |
|---|---|
| 输入包含未脱敏客户信息、生产数据、密钥、生产账号、生产连接串、生产操作诉求，或要求自动合并、自动发布、跳过 Review、隐藏风险 | `BLOCKED` |
| 目标、范围、验收标准、核心业务规则、设计树主干/分支、影响范围、ADR 判断或 P0/P1 风险证据不足 | `NEEDS_CONFIRMATION` |
| 需求过大、设计树过宽、跨系统、跨版本或存在多种未确认边界，尚不能拆成独立 TDD 任务 | `SPLIT_REQUIRED` |
| 全部准入条件满足 | `TDD_INPUT_READY` |

## 输出要求

默认输出 Markdown，包含：

1. 建议结论：`TDD_INPUT_READY`、`NEEDS_CONFIRMATION`、`SPLIT_REQUIRED` 或 `BLOCKED`
2. 最高风险等级：`P0`、`P1`、`P2`、`P3` 或 `NONE`
3. 依据、已读材料和输入缺口
4. 需求准入评审
5. `feature_context.md` 和 `scope-plan.md` 更新记录
6. 需求摘要、范围内、范围外和非目标
7. 设计树方案：业务目标、主干逻辑、分支处理、范围边界、验证点
8. 澄清问题：已关闭、待回答、建议确认人
9. 影响范围、风险和阻断建议
10. 推荐方案与取舍
11. 设计树到 TDD 任务计划
12. TDD 输入和测试重点
13. ADR 判断、草稿或更新建议
14. `DOMAIN_KNOWLEDGE.md` 与 `PROJ_CONTEXT.md` 更新结果或待确认建议
15. 本次创建、更新、跳过或缺失的 feature 文档清单

`TDD_INPUT_READY` 只表示当前证据足以建议进入 TDD，不代表负责人批准编码或风险已最终接受。

不得输出“准许编码”“审批通过”“可以上线”或替代负责人判断。
