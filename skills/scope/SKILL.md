# hicode scope

## 定位

`hicode:scope` 是需求进入编码前的收敛入口。它把用户想做的事整理成可审查、可验证、可移交给 `hicode:tdd` 的准入证据和小任务。

本 Skill 不写业务实现代码，不进入 TDD，不替代产品、研发、测试、架构或安全负责人审批。

## 读取边界

按需读取，不默认全量加载：

1. 目标项目入口规则、当前需求材料、PRD、故事、会议纪要和用户确认记录。
2. 目标项目 `docs/rules/`，以及已有等价规则文件；缺少时提示先补 `hicode:init`，不得读取本仓库规则替代。
3. 目标项目 `docs/features/<feature-id>/feature_context.md`、需求评审报告、Scope 报告和任务拆分计划；存在时读取目标项目文件，不读取模板代替事实。
4. 目标项目 `docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md` 的 Feature 索引、`docs/adr/`、历史缺陷、代码结构说明、相关代码、测试、配置、SQL、批处理或外部依赖说明。

本 Skill 内置模板只在目标项目对应文档缺失且需要创建草稿时读取：

1. `feature_context.md`
2. `requirement-review-report.md`
3. `scope-report.md`
4. `task-split-plan.md`
5. `ADR-template.md`

不得读取 `.env`、密钥文件、生产配置、生产凭证、未脱敏客户信息、未脱敏生产数据或生产日志原文。

## Scope 闭环

### 1. 固定需求目录和证据

按目标项目 `AGENTS.md` 或 `CLAUDE.md` 中的 hicode 单需求文档生命周期执行；入口缺少 hicode section 时，先提示补 `hicode:init`。目录固定为 `docs/features/<feature-id>/`；`feature-id` 不明确时，先用 `docs/PROJ_CONTEXT.md` 的 Feature 索引定位候选，再向用户确认。

完成标准：

1. `feature-id`、候选 Feature 索引命中情况、需求目录、已读材料和输入缺口已列明。
2. 目标项目已有 feature 文档优先读取；缺少文档时使用本 Skill 同名模板创建草稿。
3. 只写已确认事实、证据、待确认问题、范围边界、风险判断、ADR 草稿和拆分任务；不得编造负责人、业务规则、验收结论或验证结果。

### 2. 评审需求准入

先判断当前任务重点：需求评审、需求澄清、范围收敛、任务拆分或编码准入。用户实际要求编码、测试、Review 或发布检查时，先完成必要 Scope，再建议转到 `hicode:tdd`、`hicode:review` 或 `hicode:release`。

必须逐项检查：

1. 需求目标是否能用一句话表达。
2. 范围内、范围外和非目标是否明确。
3. 验收标准是否可测试、可观察、可复核。
4. 业务规则、异常路径、边界值、数据来源和负责人是否可追溯。
5. 关键术语是否与项目上下文冲突。
6. 是否涉及保险核心业务逻辑、金额、交易一致性、状态流转、幂等、并发、事务、权限、审计、隐私、监管、生产变更或回滚。
7. 影响范围是否能定位到模块、接口、类、方法、表、配置、SQL、批处理或外部系统。

完成标准：输出或更新 `requirement-review-report.md`，并给出最高风险等级、证据缺口、待确认问题和是否可进入拆分。

复杂或高风险时，按需委托 `requirement-reviewer` 检查需求完整性和保险核心风险，委托 `coding-planner` 检查影响范围、实现切片、TDD 重点和 ADR 判断。子 Agent 不可用时由本 Skill 直接执行，并说明降级影响；不得因此降低 P0/P1 风险等级。

### 3. 澄清并收敛范围

当需求不能安全进入计划时，进入追问循环：

1. 每轮只问一个会改变范围、风险或实现路径的问题。
2. 问题包含：当前证据、不确定点、推荐答案、推荐理由、不同选择的影响。
3. 用户回答后，更新需求摘要、范围边界、风险判断和待确认状态。
4. 若回答产生稳定术语或上下文，只输出 `DOMAIN_KNOWLEDGE.md`、`PROJ_CONTEXT.md` 或 ADR 更新建议；正式写入前必须由用户或负责人确认。
5. 若需求过大，先拆成独立子需求，并推荐优先处理的第一个子需求。

存在多种合理边界或拆分方式时，给 2-3 个方案：

1. 推荐方案放最前。
2. 每个方案说明适用条件、收益、代价、风险和不选原因。
3. 金融核心系统高风险变更必须说明哪个方案更容易验证、审计和回滚。
4. 用户未确认方案前，不生成最终任务清单。

完成标准：`feature_context.md` 已按需更新或给出草稿；每个未关闭问题都有风险等级、影响、推荐答案和建议确认人。需要 ADR 时，只有同时满足难逆、无上下文会意外、有真实取舍三个条件，才基于 `ADR-template.md` 生成 ADR 草稿。

### 4. 拆分 TDD 小任务

只有目标、范围、关键规则和风险边界足够清楚后，才能拆任务。

每个任务必须包含：

1. 目标：一句话说明完成什么。
2. 输入：需求片段、确认结论、上下文或代码证据。
3. 范围内和范围外。
4. 涉及文件、模块、接口、表或待定位对象。
5. TDD 起点和测试重点。
6. 验证方式。
7. 停止条件：遇到哪些缺口必须回到 Scope。

任务必须小、可独立理解、可独立验证、可独立回滚或重新执行。不得写“完善相关逻辑”“补充测试”“处理异常”这类不可验收任务。

完成标准：输出或更新 `task-split-plan.md`；每个任务都能交给 `hicode:tdd` 独立执行，且不会依赖未确认业务规则或隐藏范围。

### 5. 给出准入结论

只在以下条件全部满足时输出 `TDD_INPUT_READY`：

1. 需求目标、范围、非目标、验收标准和核心业务规则清楚，并有证据来源。
2. 关键术语未与项目上下文冲突，或冲突已确认处理方式。
3. 金融核心风险基线已逐项评审，无未关闭 P0/P1 阻断问题。
4. 影响范围能定位到可评审、可拆分、可验证的对象。
5. ADR 判断已给出；需要 ADR 时已有草稿或确认路径。
6. 任务已拆到可独立理解、验证、回滚或重新执行。
7. 目标项目 feature 文档的创建、更新、跳过或缺失状态已列明。

否则按以下矩阵选择结论：

| 场景 | 建议结论 |
|---|---|
| 输入包含未脱敏客户信息、生产数据、密钥、生产账号、生产连接串、生产操作诉求，或要求自动合并、自动发布、跳过 Review、隐藏风险 | `BLOCKED` |
| 目标、范围、验收标准、核心业务规则、影响范围、ADR 判断或 P0/P1 风险证据不足 | `NEEDS_CONFIRMATION` |
| 需求过大、跨系统、跨版本或存在多种未确认边界，尚不能拆成独立 TDD 任务 | `SPLIT_REQUIRED` |
| 全部准入条件满足 | `TDD_INPUT_READY` |

## 输出要求

默认输出 Markdown，包含：

1. 建议结论：`TDD_INPUT_READY`、`NEEDS_CONFIRMATION`、`SPLIT_REQUIRED` 或 `BLOCKED`
2. 最高风险等级：`P0`、`P1`、`P2`、`P3` 或 `NONE`
3. 依据、已读材料和输入缺口
4. 需求评审报告
5. `feature_context.md` 更新记录
6. 需求摘要、范围内、范围外和非目标
7. 澄清问题：已关闭、待回答、建议确认人
8. 影响范围、风险和阻断建议
9. 推荐方案与取舍
10. 拆分任务计划
11. TDD 输入和测试重点
12. ADR 判断、草稿或更新建议
13. `DOMAIN_KNOWLEDGE.md` 与 `PROJ_CONTEXT.md` 更新结果或待确认建议
14. 本次创建、更新、跳过或缺失的 feature 文档清单

`TDD_INPUT_READY` 只表示当前证据足以建议进入 TDD，不代表负责人批准编码或风险已最终接受。

不得输出“准许编码”“审批通过”“可以上线”或替代负责人判断。
