# hicode tdd

## 定位

`hicode:tdd` 覆盖测试先行、缺陷复现、受控实现和小步重构。它把已经明确的需求、Scope 拆分任务或修复目标转化为一个个可验证的行为切片。

核心原则：测试必须通过公开接口验证可观察行为，而不是绑定私有方法、内部调用顺序或实现细节。实现按纵向切片推进：一个测试、一个最小实现、一次验证，再进入下一个行为。

本 Skill 可以在用户确认范围后修改本地代码、测试或必要文档；不得绕过测试先行证据、删除测试、降低断言或隐藏失败。

## 必读材料

执行前按需读取：

1. `../../references/rules/coding_rules.md`
2. `../../references/templates/feature/tdd-report.md`

同时读取目标项目中与当前任务直接相关的材料：

1. 入口规则：`AGENTS.md` 或 `CLAUDE.md`。
2. 单需求上下文：`docs/features/<feature-id>/feature_context.md`。
3. Scope 产物：`docs/features/<feature-id>/requirement-review-report.md`、`scope-report.md`、`task-split-plan.md`。
4. 项目级上下文：`docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md`、`docs/adr/`、`docs/rules/`。
5. 相关代码、测试、脚本、构建配置和已脱敏失败证据。

若当前需求目录或 TDD 报告不存在：

1. 当前需求目录固定使用 `docs/features/<feature-id>/`；`feature-id` 不明确时先询问用户。
2. 缺少 `tdd-report.md` 时，先读取 `../../references/templates/feature/tdd-report.md`，再按需创建。
3. 缺少 `feature_context.md`、需求评审报告、Scope 报告或拆分计划时，不编造内容；说明缺口、影响和是否需要回到 `hicode:scope`。
4. 创建或更新报告只能记录真实测试设计、命令、结果、修改文件、风险和待确认问题。

不得读取 `.env`、密钥文件、生产配置、生产凭证、未脱敏客户信息或未脱敏生产数据。

## 任务模式

先判断当前属于哪种模式，并按对应准入执行：

| 模式 | 允许动作 | 准入条件 |
|---|---|---|
| 测试设计 | 输出测试计划、场景、Given-When-Then、Mock、数据和断言 | 需求目标、行为和验收标准基本明确 |
| 修复复现 | 先补失败测试或可验证失败证据，再最小修复 | 缺陷现象、期望行为和影响范围明确 |
| 受控实现 | 修改测试和生产代码，按一个行为一轮推进 | Scope 任务、允许修改范围和测试重点明确 |
| 重构 | 在行为保护测试通过后整理代码 | 重构目标、范围和不改变行为的验证方式明确 |
| 只读解释 | 解释代码、测试、失败或风险，不修改文件 | 无需 TDD 准入，但不得输出实现假结论 |

范围不清、P0/P1 待确认问题未关闭、测试目标无法观察或用户未确认可修改范围时，只能做只读分析、测试设计或补齐问题，不能修改生产实现。

## 执行流程

### 1. 固定公开接口和行为

进入本地修改前先确认：

1. 本轮要验证的公开接口：API、命令、服务方法、页面交互、事件、批处理入口或其他调用方可见入口。
2. 本轮可观察行为：返回值、状态变化、事件、错误、审计记录、幂等结果或外部边界调用结果。
3. 不测试的实现细节：私有方法、内部类拆分、内部调用次数、临时数据结构或可重构的中间步骤。
4. 本轮不处理的范围，防止借 TDD 扩大到其他需求。

若公开接口本身不清楚，先和用户确认接口形态、输入输出和最重要的行为；不要先写测试绑定临时实现结构。

### 2. 设计第一个 tracer bullet

先选一个端到端路径最短、价值最高、风险能暴露的行为作为 tracer bullet：

1. 优先覆盖主路径、关键业务规则、金额/状态/幂等/权限/审计等高风险点。
2. 写一个测试只验证一个可观察行为。
3. 测试名使用项目领域语言，能读成行为规格。
4. 先运行并确认失败；失败原因必须是目标行为尚未实现或缺陷可复现，而不是测试环境错误。

不得先批量写完所有测试再批量实现。那是横向切片，会让测试绑定想象中的结构，而不是实际行为。

### 3. RED-GREEN-REFACTOR 循环

每轮只处理一个行为：

1. RED：写或补一个失败测试，记录命令和失败摘要。
2. GREEN：只写足够代码让当前测试通过，不预实现未来行为。
3. RECORD：记录修改文件、命令、结果、未验证原因和残余风险。
4. REFACTOR：只有全部相关测试为 GREEN 后，才整理重复、命名、边界、模块深度或类型表达；每次重构后立即运行相关测试。

修复模式必须先有失败用例、复现测试或可验证失败证据；重构模式必须先有行为保护测试。RED 状态下不得重构。

### 4. 测试质量规则

优先写集成风格的行为测试：

1. 通过公开接口进入真实代码路径，验证调用方关心的结果。
2. 描述“系统做什么”，不要描述“内部怎么做”。
3. 断言金额、日期、状态、异常、权限、幂等、审计字段和边界条件。
4. 测试应能经受内部重命名、方法拆分或模块整理。
5. 一个测试只表达一个核心行为；需要多个行为时拆成多轮循环。

避免以下坏测试：

1. 测私有方法、内部 helper 或内部调用顺序。
2. Mock 自己可控制的内部类、内部模块或内部协作者。
3. 只断言调用次数而不验证业务结果。
4. 绕过公开接口直接查库、查缓存或查临时结构来证明行为。
5. 为未来功能、假设字段或未确认流程提前写测试。

### 5. Mock 与测试数据

Mock 只放在系统边界：

1. 外部 API、支付、邮件、短信、文件系统、时间、随机数、第三方 SDK。
2. 数据库按项目测试策略处理；优先使用本地测试库、内存库、容器或仓储边界，不连接真实生产库。
3. 对外部系统设计可替换边界：依赖注入、明确接口、每个外部操作一个清晰方法，避免一个泛化 fetcher 承载所有条件分支。

不得使用真实账号、Cookie、Session、生产 Token、生产响应、未脱敏客户、真实保单或真实支付数据。测试数据必须虚构或脱敏。

### 6. 本地验证

优先运行最小必要的本地非生产命令：

1. 单个测试或指定测试文件。
2. 指定模块测试。
3. 相关构建、类型检查、静态检查或格式检查。
4. 敏感信息扫描或项目已有安全检查。

命令必须来自项目脚本、文档、工具配置或用户确认。命令可能连接真实外部系统、读取生产配置或造成破坏时，不执行，改为说明原因和替代验证建议。

## 停止条件

命中以下情况时停止修改，并输出 `NEEDS_CONFIRMATION` 或 `BLOCKED`：

1. 需求、Scope 任务、公开接口、预期行为或验收标准不清。
2. P0/P1 风险未关闭，或无法判断金额、状态、幂等、权限、隐私、监管、生产变更风险。
3. 用户未确认允许修改的文件范围。
4. 失败测试无法证明目标行为，或失败来自环境/依赖问题且无法隔离。
5. 需要连接生产、读取生产日志、执行生产 SQL、修改生产配置、发布或回滚。
6. 输入包含密钥、生产凭证、未脱敏客户信息或未脱敏生产数据。
7. 用户要求删除测试、降低断言、跳过 RED、隐藏失败、自动提交、自动推送、自动合并或自动发布。

## 输出要求

默认使用 Markdown，包含：

1. 建议结论：`PASS`、`CONDITIONAL_PASS`、`BLOCKED` 或 `NEEDS_CONFIRMATION`。
2. 任务模式和最高风险等级。
3. 读取依据、输入缺口和未覆盖范围。
4. 公开接口、可观察行为和本轮不测试的实现细节。
5. 测试场景和 Given-When-Then。
6. Mock、测试数据和断言规则。
7. RED-GREEN-REFACTOR 记录，逐轮说明测试、实现、命令和结果。
8. 修改文件清单。
9. 受限命令执行记录和未执行原因。
10. 风险、待确认问题和上下文更新建议。
11. 本次创建、更新、跳过或缺失的 feature 文档清单。

不得输出最终合并、发布、上线审批或负责人确认结论。
