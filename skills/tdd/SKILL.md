# hicode tdd

## 定位

`hicode:tdd` 是编码执行入口。它把已澄清的需求、`hicode:scope` 拆分任务或明确缺陷，转化为一轮轮可验证的行为切片：先写失败测试，再做最小实现，再重构。

本 Skill 可以在用户确认范围后修改本地代码、测试和必要文档；不得跳过测试先行、删除测试、降低断言、隐藏失败、自动合并、自动发布或替代负责人审批。

## 按需加载材料

按需读取，不默认全量加载：

本 Skill 内置材料：

1. `tdd-report.md`

目标项目事实文档：

1. 当前任务直接相关的入口规则。
2. `docs/rules/` 下由 `hicode:init` 创建或更新的项目规则文件，以及目标项目已有的等价规则文件；缺少时提示先补 `hicode:init`，不得读取本仓库规则替代。
3. `docs/features/<feature-id>/feature_context.md`、需求评审报告、Scope 报告、任务拆分计划和已有 TDD 报告；存在时读取目标项目文件，不读取模板代替事实。
4. `docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md` 的 Feature 索引、`docs/adr/`、相关代码、测试、脚本、构建配置和已脱敏失败证据。

模板只在目标项目缺少 `tdd-report.md` 且需要创建或更新报告时读取；不得把模板内容当成目标项目事实。

不得读取 `.env`、密钥文件、生产配置、生产凭证、未脱敏客户信息、未脱敏生产数据或生产日志原文。

## 快速示例

用户说“按 TDD 实现 task 1”时，先确认 `feature-id`、任务范围和允许修改的文件；读取 Scope 产物和相关代码；选择一个公开接口行为作为 tracer bullet；写一个会失败的测试并运行；用最小代码让它通过；记录命令、结果、修改文件和残余风险；再进入下一轮。

## 文档规则

按目标项目 `AGENTS.md` 或 `CLAUDE.md` 中的 hicode 单需求文档生命周期执行；入口缺少 hicode section 时，先提示补 `hicode:init`。`feature-id` 或任务范围不清时，先用 `docs/PROJ_CONTEXT.md` 的 Feature 索引定位候选，再读取对应单需求文档。缺少 `tdd-report.md` 时，先读取本 Skill 内置模板再按需创建；缺少 `feature_context.md`、需求评审报告、Scope 报告或任务拆分计划时，不编造内容，说明缺口、影响，以及是否需要回到 `hicode:scope`。

历史 feature 只能作为定位、对比和风险提示，不能替代当前 Scope 任务、失败证据或用户确认。

TDD 报告只能记录真实测试设计、命令、结果、修改文件、风险和待确认问题。

## 任务模式

先判断当前属于哪类：

| 模式 | 允许动作 | 准入条件 |
|---|---|---|
| 测试设计 | 只输出测试计划、场景、数据、Mock 和断言 | 需求目标、行为和验收标准基本明确 |
| 修复复现 | 先补失败测试或可验证失败证据，再最小修复 | 缺陷现象、期望行为和影响范围明确 |
| 受控实现 | 修改测试和生产代码，按一个行为一轮推进 | Scope 任务、相关修改范围和测试重点明确 |
| 行为保护重构 | 在行为测试保护下整理代码 | 重构目标、范围和不改变行为的验证方式明确 |
| 只读解释 | 解释代码、测试、失败或风险，不修改文件 | 无需 TDD 准入，但不得输出实现假结论 |

范围不清、P0/P1 待确认问题未关闭、行为无法观察或修改范围无法从用户请求与上下文中界定时，只能做只读分析、测试设计或追问，不能修改生产实现。

## 核心原则

1. 测试先于实现，证据先于结论。
2. 测试验证公开接口的可观察行为，不绑定私有方法、内部调用顺序或临时结构。
3. 一个测试只表达一个核心行为；一个实现只服务当前测试。
4. 先做最短端到端 tracer bullet，再扩展异常、边界和高风险路径。
5. GREEN 前不重构；重构后必须重新运行相关测试。
6. 金融核心系统相关代码默认检查金额、状态、幂等、事务、并发、权限、审计、隐私、监管、生产变更和回滚风险。
7. 命令必须来自项目脚本、文档、工具配置或用户确认。

## 执行流程

### 1. 确认本轮切片

进入本地修改前固定：

1. 公开接口：API、命令、服务方法、页面交互、事件、批处理入口或其他调用方可见入口。
2. 可观察行为：返回值、状态变化、事件、错误、审计记录、幂等结果或外部边界调用结果。
3. 不测试的实现细节：私有方法、内部类拆分、内部调用次数、临时数据结构或可重构中间步骤。
4. 本轮范围外：明确不处理的需求、异常、模块或风险。

公开接口不清楚时，先确认接口形态、输入输出和最重要的行为；不要先写绑定临时实现结构的测试。

用户明确要求“实现”“修复”“按 TDD 做”且任务范围可从 Scope、缺陷描述或上下文界定时，可视为允许修改相关测试、生产代码和必要文档。删除文件、覆盖已有冲突内容、跨模块大改、修改配置/SQL/发布脚本或触及敏感/生产材料时，必须单独确认。

### 2. 设计 tracer bullet

选择端到端路径最短、业务价值最高、风险能暴露的行为：

1. 优先覆盖主路径、关键业务规则、金额、状态、幂等、权限或审计。
2. 测试名使用项目领域语言，能读成行为规格。
3. 测试数据必须虚构或脱敏。
4. 先运行并确认失败；失败原因必须是目标行为未实现或缺陷可复现，而不是环境错误。

不得先批量写完所有测试再批量实现。

### 3. RED-GREEN-REFACTOR

每轮只处理一个行为：

1. RED：写或补一个失败测试，运行最小命令，记录失败摘要。
2. GREEN：只写足够代码让当前测试通过，不预实现未来行为。
3. RECORD：记录修改文件、命令、结果、未验证原因和残余风险。
4. REFACTOR：全部相关测试为 GREEN 后，才整理重复、命名、边界、模块深度或类型表达；每次重构后立即运行相关测试。

修复模式必须先有失败用例、复现测试或可验证失败证据；重构模式必须先有行为保护测试。

### 4. 测试质量

优先写集成风格的行为测试：

1. 从公开接口进入真实代码路径，验证调用方关心的结果。
2. 描述系统做什么，不描述内部怎么做。
3. 断言金额、日期、状态、异常、权限、幂等、审计字段和边界值。
4. 能经受内部重命名、方法拆分或模块整理。

避免：

1. 测私有方法、内部 helper 或内部调用顺序。
2. Mock 自己可控制的内部类、内部模块或内部协作者。
3. 只断言调用次数而不验证业务结果。
4. 绕过公开接口直接查库、查缓存或查临时结构来证明行为。
5. 为未来功能、假设字段或未确认流程提前写测试。

### 5. Mock 与测试数据

Mock 只放在系统边界：外部 API、支付、邮件、短信、文件系统、时间、随机数、第三方 SDK。

数据库按项目测试策略处理；优先使用本地测试库、内存库、容器或仓储边界，不连接真实生产库。

外部系统边界应可替换：依赖注入、明确接口、每个外部操作一个清晰方法；避免一个泛化 fetcher 承载所有条件分支。

不得使用真实账号、Cookie、Session、生产 Token、生产响应、未脱敏客户、真实保单或真实支付数据。

### 6. 本地验证

优先运行最小必要的本地非生产命令：

1. 单个测试或指定测试文件。
2. 指定模块测试。
3. 相关构建、类型检查、静态检查或格式检查。
4. 敏感信息扫描或项目已有安全检查。

命令可能连接真实外部系统、读取生产配置或造成破坏时，不执行；说明原因和替代验证建议。

## 停止条件

命中以下情况时停止修改，输出 `NEEDS_CONFIRMATION` 或 `BLOCKED`：

1. 需求、Scope 任务、公开接口、预期行为或验收标准不清。
2. P0/P1 风险未关闭，或无法判断金额、状态、幂等、权限、隐私、监管、生产变更风险。
3. 修改范围无法从用户请求、Scope 任务、缺陷证据或文件上下文中界定。
4. 失败测试无法证明目标行为，或失败来自环境/依赖问题且无法隔离。
5. 需要连接生产、读取生产日志、执行生产 SQL、修改生产配置、发布或回滚。
6. 输入包含密钥、生产凭证、未脱敏客户信息或未脱敏生产数据。
7. 用户要求删除测试、降低断言、跳过 RED、隐藏失败、自动提交、自动推送、自动合并或自动发布。

## 输出要求

默认输出 Markdown，包含：

1. 建议结论：`LOCAL_VERIFIED`、`PARTIAL_VERIFICATION`、`BLOCKED` 或 `NEEDS_CONFIRMATION`
2. 任务模式和最高风险等级
3. 读取依据、输入缺口和未覆盖范围
4. 公开接口、可观察行为和本轮不测试的实现细节
5. 测试场景和 Given-When-Then
6. Mock、测试数据和断言规则
7. RED-GREEN-REFACTOR 记录：测试、实现、命令和结果
8. 修改文件清单
9. 受限命令执行记录和未执行原因
10. 风险、待确认问题和上下文更新建议
11. 本次创建、更新、跳过或缺失的 feature 文档清单

`LOCAL_VERIFIED` 只表示本地已覆盖范围的验证通过；`PARTIAL_VERIFICATION` 表示存在未覆盖范围、未执行命令或残余风险。二者都不是合并、发布或负责人确认结论。

不得输出最终合并、发布、上线审批或负责人确认结论。
