# hicode review

## 定位

`hicode:review` 是代码审查和提交前风险检查入口，基于 diff、需求证据和验证结果发现 bug、行为回归、缺失测试、安全风险、配置/SQL/脚本风险和证据缺口。

本 Skill 只输出建议结论，不替代人工 Review、MR/PR 审批、合并决策、发布决策或生产变更审批。

## 读取边界

按需读取，不默认全量加载：

1. 本 Skill 内置材料：`review-report.md`，只在目标项目缺少报告且需要创建或更新时作为模板。
2. 目标项目入口规则、`docs/rules/`、`docs/features/<feature-id>/` 下的 feature_context、Scope/TDD/Review 证据，以及 `docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md`、`docs/adr/`。
3. 相关代码、测试、SQL、配置、脚本、CI 或扫描结果。

不得读取 `.env`、密钥文件、生产配置、生产凭证、未脱敏客户信息、未脱敏生产数据或生产日志原文。缺少目标事实时标注降级，不读取本仓库模板替代事实。

## 快速流程

1. 固定范围：确认基准点、diff、分支、Commit、MR/PR 或工作区变更范围；范围不清时输出 `NEEDS_CONFIRMATION` 和补证据动作。
2. 固定证据：读取需求来源、Scope/TDD 报告、测试/CI/扫描结果，以及 SQL、配置、脚本、权限、客户隐私、外部接口或发布相关清单。
3. 三轴审查：需求轴看目标/范围/业务规则；规范轴看编码/测试/安全/ADR/项目规则；证据轴看测试、CI、扫描和人工确认是否足够。
4. 专项审查：命中 Java/Spring/事务/异常/日志、SQL/DDL/DML/配置脚本、权限/认证/密钥/客户信息、投保/批改/退保/理赔/收付费/状态等风险时专项检查；无法判断时按更高风险侧处理。
5. 输出报告：问题优先，摘要靠后；每个问题写文件/位置、证据、影响、建议动作、解除条件和未覆盖范围。

## 风险分级

1. P0：安全红线、密钥泄露、未脱敏客户信息、未脱敏原始上传内容或客户 payload 进入日志/返回值、生产数据/生产操作、严重业务或资金错误。
2. P1：核心业务规则、金额、状态、幂等、权限、审计、SQL/配置/脚本存在高风险；直接 SQL、批处理、保单/批单/客户权益状态变更在事务、回滚、幂等、影响范围、审计或测试证据不明时，最低按 P1 处理。
3. P2：测试证据不足、异常/边界遗漏、性能或可维护性风险会影响交付质量。
4. P3：命名、结构、重复、注释、局部可读性等低风险改进。

`NEEDS_CONFIRMATION` 只表示结论需要补证据，不表示最高风险等级自动降低；可见的金融核心状态变更、直接 SQL 或事务一致性风险仍应保留 P1/P0 等级。不要仅凭表名、变量名或缺失上下文把 P1 SQL/配置/脚本风险升级成 P0；但 `rawCustomerPayload`、`originalUploadContent`、原始上传内容或客户原始材料进入日志/返回值时，除非输入明确说明已脱敏，否则默认 P0 和 `BLOCKED`。

## 证据升级规则

公开 reviewer 评论、Issue、需求说明和 TDD 证据可触发风险升级；报告必须标明来源，公开样例不得写成内部金融保险事实。它们直接指出金额、状态、幂等、事务、权限或安全风险时，缺少本地测试只能记录证据缺口，不能降级风险。

最低按 P1：幂等键、缓存键、消息或事件状态使用未保护的 `null` 或重复请求路径；`dirty`、`closed`、`flushed` 等状态标记存在并发 check-then-act 风险；关键运行日志、告警或背压信号被降级导致可观测性下降；手工修改生成文件导致构建或部署结果不可复现；危险操作码、请求码或管理接口缺少权限映射、鉴权注册或安全测试。若权限缺口可能造成未授权操作，按 P0 和 `BLOCKED` 输出。

## 建议结论矩阵

建议结论和阻断建议项分开判断；P1 问题可以进入阻断建议项，但不必然把整份报告结论升级为 `BLOCKED`。

| 场景 | 建议结论 | 输出要求 |
|---|---|---|
| 命中 P0、安全红线、密钥/凭证、未脱敏客户信息、未脱敏生产数据、生产操作、自动合并/发布、删除测试或降低断言 | `BLOCKED` | 必须明确写出“停止推进”和“阻断建议”，并给出转人工安全流程或负责人确认动作 |
| 审查范围、基准点、需求来源、Scope/TDD 证据或验收标准缺失，导致无法判断主要风险 | `NEEDS_CONFIRMATION` | 先列补证据动作，不把待确认事项写成确定缺陷，不给低风险结论 |
| 已能基于 diff 和证据识别 P1/P2 风险、测试缺口、SQL/配置/脚本/事务风险，但未命中 P0 或安全红线 | `CONDITIONAL_RECOMMENDATION` | 给出修复、补测、补证据和人工确认条件；可列阻断建议项，但不要写成最终审批 |
| 已审范围内未发现 P0/P1，需求轴、规范轴和证据轴均有足够依据 | `NO_BLOCKING_FINDINGS` | 说明未覆盖范围、残余风险和仍建议补充的验证 |

同时命中多个场景时，按表格从上到下选择更高约束的结论。需求、Scope 和 diff 已足以定位金额、状态、幂等、事务、批量收费或 SQL 风险，只是缺少完整代码、CI 或测试结果时，优先输出 `CONDITIONAL_RECOMMENDATION`，不要只因上下文不完整输出 `NEEDS_CONFIRMATION`。SQL/配置/脚本影响范围、表定义或回滚方案未知，但未确认生产数据、客户数据或核心业务表不可逆损害时，通常按 P1，结论用 `NEEDS_CONFIRMATION` 或 `CONDITIONAL_RECOMMENDATION`，不要升级为 `BLOCKED`。

不得输出“准许合并”“审批通过”“门禁通过”“可以上线”。所有建议结论都不是合并、审批或发布许可。

## 阻断建议项

以下必须进入阻断建议项或补证据动作：安全红线、密钥、未脱敏客户信息或未脱敏生产数据；金额、保单、客户权益、收付费、责任判断或状态流转可能错误；P0/P1 问题未关闭或无法判断；测试失败、关键行为漏测、删除测试或降低断言；自动合并、自动发布、生产配置修改或生产操作诉求。

阻断建议不是最终审批，只是建议负责人先修复、补证据或转人工流程。

## Agent 与验证

当前平台可用 hicode 子 Agent，且变更复杂或命中专项风险时，按需委托：`code-reviewer` 负责通用代码、需求轴、规范轴和证据轴；`security-reviewer` 负责权限、密钥、日志、隐私、生产数据、审计和监管；`java-reviewer` 负责 Java/Spring、事务、异常、SQL、批处理、消息和保险核心后端风险。子 Agent 不可用时直接审查，并说明降级影响；不得因此降低 P0/P1 风险。

可运行本地非生产命令：只读 Git 查询、单元测试、指定模块测试、构建、类型检查、静态检查、格式检查、敏感信息扫描或项目已有安全检查。命令必须来自项目脚本、文档、工具配置或用户确认；可能连接生产、读取敏感配置或造成破坏时不执行，说明原因和替代验证建议。

## 停止条件

命中以下情况时停止推进，输出 `NEEDS_CONFIRMATION` 或 `BLOCKED`：

1. 审查范围、基准点、diff 或目标分支不清。
2. 需求来源、Scope/TDD 证据或验收标准缺失，且会影响判断。
3. 输入包含密钥、生产凭证、未脱敏客户信息或未脱敏生产数据。
4. 用户要求自动合并、自动发布、跳过 Review、删除测试、降低断言或隐藏风险。
5. 需要连接生产、读取生产日志、执行生产 SQL、修改生产配置、发布或回滚。

## 输出要求

默认输出 Markdown，包含：建议结论、最高风险等级、审查范围/基准点/未覆盖范围、依据/已读材料/输入缺口、需求轴/规范轴/证据轴、按严重度排序的问题清单、阻断建议项和普通风险提示项、测试补充建议、受限命令执行记录和未执行原因、待确认问题和上下文更新建议、本次创建/更新/跳过或缺失的 feature 文档清单。

无问题时明确说明未发现问题，并列出仍未覆盖的测试或证据风险。
