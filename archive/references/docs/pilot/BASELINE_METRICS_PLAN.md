# BASELINE_METRICS_PLAN

## 1. 文档定位

本文档用于 P6-WP2，作为 V1 试点前后指标采集的指标字典、采集计划和空值记录模板。它定义效率、质量、过程和安全指标的数据来源、采集周期、统计口径和责任角色，用于后续试点复盘。

本文档不记录未采集的实际指标值，不把目标建议写成已达成结果，不编造基线值、试点后值、覆盖率、有效问题率、满意度或安全结论。

- 维护对象：AI 工程化小组、研发负责人、测试负责人、安全负责人
- 主要使用者：试点项目负责人、研发代表、测试代表、产品/BA 代表、架构师或技术负责人、发布负责人
- 关联资产：`references/docs/pilot/PILOT_PROJECTS.md`、`references/docs/pilot/PILOT_USAGE_LOG.md`、`references/docs/pilot/V1_REVIEW_REPORT_TEMPLATE.md`

## 2. 使用规则

1. 每个指标使用一个脱敏 `metric_id`，格式建议为 `METRIC-YYYY-NNN`，例如 `METRIC-2026-001`。
2. 每条指标必须关联 `pilot_project_id`；若尚未确定试点项目，先填写 `待确认`。
3. 指标值默认填写 `待采集`、`不适用` 或 `无法取得，需说明原因`，不得填写推断值。
4. 数据来源只记录来源类型，例如需求系统统计、MR/PR 记录、AI Review 报告、测试报告、发布检查报告、人工抽样确认或问卷汇总；不得粘贴真实客户数据、生产日志、生产 SQL、生产地址、密钥或未脱敏材料。
5. 指标目标建议来自 V1 验收口径时，只能作为参考阈值，不代表当前已达成。
6. 涉及安全指标时，必须由安全负责人或等效角色确认采集口径；发现敏感信息泄露、生产越权或未经 Review 的 AI 代码合入，应记录为 P0 风险并转人工流程。

## 3. 指标状态枚举

| 状态 | 含义 |
|---|---|
| 待采集 | 已定义采集口径，但尚未采集实际值 |
| 已采集 | 已按口径采集值，并保留证据来源 |
| 不适用 | 该试点项目不涉及该指标，需说明原因 |
| 无法取得 | 当前无法从系统、报告或人工记录取得，需说明补齐动作 |
| 待确认 | 统计口径、数据来源或责任角色尚未确认 |

## 6. 指标分类口径

### 6.1 效率指标

效率指标用于观察 hicode 是否减少重复整理、提高计划、编码或沟通效率。V1 不直接要求填写节省工时，除非试点团队已有可信的工时记录口径。

| 指标 | 建议来源 | 注意事项 |
|---|---|---|
| AI 代码采纳率 | 试点使用记录、人工确认 | 采纳必须经过人工 Review，不得把采纳写成自动合入 |
| 开发人员满意度 | 问卷汇总或访谈摘要 | 只记录汇总结果和问题摘要，不记录个人评价原文 |

### 6.2 质量指标

质量指标用于观察 hicode 对问题发现、误报漏报、缺陷和发布遗漏的影响。质量指标必须区分“发现问题”和“有效问题”。

| 指标 | 建议来源 | 注意事项 |
|---|---|---|
| 需求评审问题数 | 需求评审报告 | 问题数不等于有效问题数，需后续人工确认 |
| ADR 判断准确率 | 编码计划报告、技术负责人确认 | ADR 是否正式记录由技术负责人确认 |
| AI Review 问题数 | AI Review 报告 | 按 P0/P1/P2/P3 分层统计 |
| AI Review 有效问题率 | 人工抽样确认 | 需记录样本口径，不能凭感觉填写 |
| 单测覆盖率变化 | 测试报告、覆盖率报告 | 必须说明覆盖率工具和统计范围 |
| MR 打回次数 | MR/PR 记录 | 区分 AI 建议导致、人工 Review 导致和 CI 导致 |
| 提测缺陷数 | 缺陷记录、测试报告 | 不贴未脱敏缺陷详情 |
| 发布遗漏数 | 发布检查报告、复盘摘要 | 不贴生产日志和生产数据 |
| 误报记录数 | 使用记录、人工确认 | 用于 Prompt、Skill 和门禁优化 |
| 漏报记录数 | 使用记录、人工确认 | 必须说明漏报发现来源 |

### 6.3 过程指标

过程指标用于观察 V1 核心资产是否被按计划使用。过程覆盖率不得解释为质量达成。

| 指标 | 建议来源 | 注意事项 |
|---|---|---|
| 需求评审覆盖率 | 需求系统统计、需求评审报告 | 仅统计重点需求 |
| 编码计划覆盖率 | 编码计划报告 | 普通低风险需求可不强制 |
| ADR 判断覆盖率 | 编码计划报告、ADR 记录 | 只统计已生成编码计划的需求 |
| AI Review 覆盖率 | MR/PR 记录、AI Review 报告 | 不替代人工 Review |
| 提交前检查覆盖率 | 提交检查报告、MR/PR 材料 | V1 可人工执行 |
| 核心场景沉淀数量 | 核心场景测试报告 | 区分场景建议和已执行用例 |
| 发布前检查覆盖率 | 发布检查报告、发布负责人确认 | 不代表发布批准 |

### 6.4 安全指标

安全指标用于确认 V1 试点没有突破敏感信息、生产操作和人工 Review 边界。

| 指标 | 建议来源 | 注意事项 |
|---|---|---|
| 敏感信息泄露次数 | 输入输出抽检、安全负责人确认 | 目标为 0；发现即按 P0 阻断复盘 |
| 未经 Review 的 AI 代码合入次数 | MR/PR 审计、人工 Review 记录 | 目标为 0；AI 生成代码必须人工 Review |

## 10. 禁止误用

1. 不编造基线值、试点后值、覆盖率、有效问题率、满意度、安全事件或推广结论。
2. 不把目标建议写成当前已达成。
3. 不把过程覆盖率解释为质量提升结论。
4. 不用未脱敏客户数据、生产日志、生产 SQL、生产地址、生产账号、密钥、Token、Cookie、Session 或连接串作为采集证据。
5. 不替代项目负责人、研发负责人、测试负责人、安全负责人、发布负责人或业务负责人的最终确认。
6. 不为了达成指标降低 P0/P1 风险等级、删除误报漏报记录、隐藏缺失数据或弱化敏感信息风险。
