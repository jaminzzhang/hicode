# Markdown Structured Output

## 1. 定位

hicode 当前使用 Markdown 结构化输出规则，不再维护当前 JSON 结构化校验文件。历史校验文件中仍有效的字段和枚举只作为本规则中的稳定输出口径。

本规则适用于分析、计划、Review、测试、发布、门禁建议和上下文更新类输出。普通简短问答可按需简化，但涉及风险、变更或节点推进时必须保留关键字段。

## 2. 建议结论

Agent 输出的是建议结论，不是最终审批。不得写成“准许合并”“准许发布”“可以上线”或“审批通过”。

| 代码 | 中文建议结论 | 使用条件 |
|---|---|---|
| `PASS` | 未发现 P0/P1 风险，建议进入下一步人工或既有流程 | 已检查范围明确，关键证据充分，未发现阻断风险 |
| `CONDITIONAL_PASS` | 有条件建议继续 | 存在 P2/P3 或材料补强项，需明确条件、负责人和验证要求 |
| `BLOCKED` | 发现高风险，建议先修复或补证据 | 存在 P0/P1、生产越权、安全红线或关键证据缺失 |
| `NEEDS_CONFIRMATION` | 证据不足，待确认后再判断 | 输入不足、范围不清、负责人确认缺失或无法排除 P0/P1 |

## 3. 稳定枚举

| 类型 | 可用值 |
|---|---|
| 风险等级 | `P0` / `P1` / `P2` / `P3` / `NONE` |
| 检查项结果 | `PASS` / `FAIL` / `PENDING` / `NOT_APPLICABLE` |
| 人工确认状态 | `CONFIRMED` / `PENDING` / `NOT_APPLICABLE` |
| 受限命令状态 | `EXECUTED` / `FAILED` / `NOT_EXECUTED` / `SKIPPED` |
| 报告状态 | `DRAFT` / `PENDING_CONFIRMATION` / `ARCHIVED` |

枚举可在 Markdown 表格中保留英文代码，也可以同时给出中文标签。不得为同一语义临时发明新代码。

## 4. 通用输出结构

分析、评审、计划、测试和发布类输出默认包含：

1. `结论`：建议结论、最高风险等级、一句话依据。
2. `范围与输入`：检查对象、输入材料、未读取或不可用材料。
3. `依据或证据`：引用的文件、diff、报告、命令结果、人工确认或不适用说明。
4. `问题清单`：问题、证据、影响、风险等级、建议动作。
5. `阻断建议项`：P0/P1、安全红线、生产越权、关键证据缺失及解除条件。
6. `普通风险提示项`：P2/P3、材料补充、验证增强和负责人接受条件。
7. `验证与命令记录`：实际执行的命令、结果、失败摘要或未执行原因。
8. `待确认问题`：问题、优先级、影响、建议确认人、期望材料。
9. `建议更新的上下文或资产`：建议位置、类型、原因、待确认事项。

具体 Skill 或模板可以调整章节名，但不得丢失结论、证据、风险、建议动作和待确认问题。

## 5. 问题列表规则

问题清单必须可执行：

| 字段 | 要求 |
|---|---|
| 问题 | 描述具体缺陷、缺口或风险，不写泛化评价 |
| 证据 | 给出文件路径、输入材料、命令结果、报告摘要或缺失证据 |
| 影响 | 说明对业务、质量、安全、发布或合规的影响 |
| 风险等级 | 使用 `P0/P1/P2/P3/NONE` |
| 建议动作 | 给出修复、补证据、验证、人工确认或暂停推进动作 |

P0/P1 必须同步进入 `阻断建议项`。P2/P3 不得伪装成 P0/P1，也不得掩盖安全红线。

## 6. 验证记录规则

1. 已执行验证时，记录命令、范围、结果和失败摘要。
2. 未执行验证时，写明原因，例如缺少脚本、缺少权限、超出范围或用户未授权。
3. 不得把未运行测试写成已验证。
4. 不得把只读分析写成运行时验证。
5. 涉及生产、密钥、未脱敏数据或平台动作的验证，只能建议人工或既有平台流程执行。

## 7. 上下文更新规则

发现稳定缺口时，可以输出上下文或 hicode 资产更新建议，但不得把推断直接写成事实。

建议更新项应包含：

1. 建议位置。
2. 更新类型：新增 / 修改 / 废弃 / 待确认。
3. 建议内容摘要。
4. 原因或证据。
5. 需要谁确认。
