# 辅助编码 Skill

## 1. 定位

本 Skill 用于目标项目本地的受控辅助编码，覆盖实现、修复、重构和解释四种模式。凡会修改业务代码、测试代码、配置脚本或重构代码行为的任务，都必须先通过编码计划和 TDD 证据校验；解释模式保持只读，不属于代码修改。

本 Skill 的核心职责是执行门槛后的最小范围实现、修复或重构，并记录文件变更、测试处理、受限命令、风险和待确认问题。

本 Skill 不替代开发人员、架构师、Review 人、测试负责人或发布负责人，不自动提交、自动合并或自动发布。

## 2. 适用场景

使用本 Skill 的场景：

1. 实现模式：编码计划和 TDD 测试设计已形成，需要按 RED-GREEN-REFACTOR 的最小范围修改本地代码和测试。
2. 修复模式：已有失败用例、复现测试或可验证失败证据，需要定位并修复。
3. 重构模式：已有行为保护测试、明确范围和不改变业务行为的验证方式，需要小范围重构。
4. 解释模式：需要只读解释代码、业务流程、风险或测试含义，不修改文件。
5. 用户明确要求受控辅助编码，且输入材料满足安全规则和准入条件。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 需求、业务规则、影响范围或 P0/P1 待确认问题尚未关闭。
2. 实现模式缺少编码计划或 TDD 输入。
3. 修复模式缺少失败用例、复现测试或可验证失败证据。
4. 重构模式缺少行为保护测试，或当前测试处于 RED 状态。
5. 用户要求跳过测试、先编码后补测试、删除测试、降低断言或绕过 Review。
6. 用户要求跨需求重构、生成完整核心交易实现、直接操作生产、自动合并、自动发布或修改生产配置。
7. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。

## 4. 必读上下文

执行前按需读取：

1. `AGENTS.md`
2. `docs/workflows/coding-plan.md`
3. `docs/workflows/tdd.md`
4. `.hicode/prompts/coding-assistant.md`
5. `docs/PRD_CONTEXT.md`
6. `docs/PROJ_CONTEXT.md`
7. `docs/CODING_RULES.md`
8. `docs/TESTING_GUIDE.md`
9. `docs/DOMAIN_KNOWLEDGE.md`
10. `docs/DEFECT_CASES.md`
11. 编码计划或 `coding-plan` Skill 输出
12. TDD 测试设计报告或 `tdd` Skill 输出，解释模式可不提供
13. 相关代码、已有测试、测试命令、构建命令、静态检查命令或 CI 摘要
14. 修复模式所需的失败用例、复现测试或可验证失败证据；Review 问题、缺陷摘要或复现说明可作为辅助背景
15. 本目录的 `output-template.md`

只读取当前任务必要上下文。缺少上下文时，输出缺口和影响，不补编业务规则、类、表、接口、配置、测试结果或发布结论。

## 5. 输入材料

要求用户或调用方提供：

1. 任务模式：实现 / 修复 / 重构 / 解释。
2. 需求编号、需求摘要或 `docs/PRD_CONTEXT.md`。
3. 编码计划，包括改动目标、影响范围、建议步骤、风险和 ADR 判断。
4. TDD 测试设计报告，包括行为清单、RED-GREEN-REFACTOR 循环、测试场景、Mock 策略、断言规则和测试代码草案。
5. 允许修改范围，包括模块、类、接口、方法、表、配置、测试文件或 diff 摘要。
6. 相关代码、已有测试、测试命令、构建命令、静态检查命令或 CI 摘要。
7. 修复模式所需的失败用例、复现测试或可验证失败证据；Review 问题、缺陷摘要或复现说明可作为辅助背景。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止并要求先脱敏或改走人工安全流程。
3. 判断任务模式并执行准入检查：
   - 实现模式必须有编码计划和 TDD 输入，并能指向 RED-GREEN-REFACTOR 循环。
   - 修复模式必须有失败用例、复现测试或可验证失败证据。
   - 重构模式必须有行为保护测试，且当前测试应为 GREEN。
   - 解释模式保持只读，不修改文件。
4. 读取必需上下文，列出已读取材料、缺失材料和缺失影响。
5. 确认允许修改范围，拒绝顺手重构、跨需求修改或无关文件改动。
6. 按 TDD 垂直切片执行代码修改：
   - RED：确认当前行为测试失败或缺陷可复现。
   - GREEN：只写通过当前测试的最小实现。
   - REFACTOR：仅在 GREEN 状态下做必要重构，并重新运行相关测试。
7. 同步处理测试，禁止删除测试、降低断言或绕过质量门禁。
8. 检查金融核心系统风险、编码规范、测试规范、安全约束和发布影响。
9. 在具备条件时运行受限测试、构建、静态检查或格式检查；无法运行时说明原因、风险和人工验证动作。
10. 输出变更摘要、验证记录、风险、待确认问题和上下文更新建议。

## 7. 输出

必须输出：

1. 结论：已完成本地修改 / 已输出补丁草案 / 只读解释完成 / 有条件完成 / 未完成 / 待确认。
2. 任务模式。
3. 依据。
4. 准入与 TDD 证据检查。
5. 修改范围。
6. 修改文件清单。
7. RED-GREEN-REFACTOR 执行记录。
8. 核心逻辑说明。
9. 异常处理、日志与审计说明。
10. 测试处理。
11. 受限命令执行记录。
12. 风险清单。
13. 待确认问题。
14. 验证要求。
15. 上下文更新建议。
16. 下一步建议。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的辅助编码 Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 8. 质量标准

输出必须满足：

1. 每个文件修改都能追溯到需求、编码计划、TDD、代码上下文、测试结果或规范文档。
2. 所有代码修改任务都有 TDD 或测试先行证据；缺失时不得继续实现、修复或重构。
3. 修改范围严格限于编码计划、TDD 和用户提供的允许范围。
4. 实现模式按 RED-GREEN-REFACTOR 小步推进，不批量实现投机功能。
5. 修复模式优先补充或复用失败测试，不做无复现依据修复。
6. 重构模式不改变业务行为，并有行为保护测试。
7. 解释模式只读，不修改文件，不运行高风险命令。
8. 风险按 P0/P1/P2/P3 分级，并说明影响、建议动作和建议确认人。

## 9. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不直接操作生产环境，不直接给出生产操作命令。
6. 不自动提交，不自动合并 MR / PR，不自动发布，不自动修改生产配置。
7. 不删除测试，不降低断言，不为了通过测试而绕过质量门禁。
8. 不绕过异常处理、日志、权限、审计、事务、幂等或外部调用保护。
9. 受限命令仅限本地单元测试、构建、静态检查或格式检查；不得执行数据库变更、发布、回滚、生产连接或删除未确认文件。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 10. 验证要求

完成后必须说明：

1. 实际读取的上下文。
2. 任务模式和准入是否满足。
3. TDD 或测试先行证据来源。
4. 修改文件是否都在允许范围内。
5. RED、GREEN、重构和最终验证命令是否实际执行；如未执行，说明原因和风险。
6. 是否存在测试、Review、发布或负责人确认缺口。
7. 是否需要更新目标项目 `docs/`、目标项目 `.hicode/` 或本仓库 hicode 根目录源资产。

## 11. 上下文更新

如发现稳定业务规则、测试规则、历史缺陷、模块约束、接口约束、ADR 决策或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库、测试规范或 ADR 前，必须由对应负责人确认。
