# 合并门禁

## 1. 定位

本门禁用于 MR 或 PR 合并前的提醒型检查，判断合并对象是否具备 AI Review、人工 Review、CI、覆盖率和未关闭 P0/P1 问题关闭证据。

本门禁不替代人工 Reviewer、模块 Owner、研发负责人、质量负责人、CI/CD 或代码托管平台的最终判断，不自动批准、不自动合并、不自动打 Tag、不自动发布。

## 2. 适用场景

使用本门禁的场景：

1. MR 或 PR 准备合并前。
2. 已完成 AI Review、人工 Review、CI 和测试覆盖率检查，需要形成合并节点门禁结论。
3. 需要核对未关闭 P0/P1 问题、未解决讨论、风险豁免和负责人确认。
4. 需要检查 MR 材料是否包含敏感信息、密钥、未脱敏生产数据或生产越权诉求。

不适用场景：

1. 当前变更尚未准备提测或提交检查；应先使用 `coding-to-test-gate.md` 或 `pre-commit-check` Skill。
2. 当前目标是深度代码审查；应先使用 `code-review` Skill。
3. 当前目标是发布申请前检查；应使用后续 `release-gate.md`。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。

## 3. 输入材料

至少需要：

1. MR 或 PR 链接、编号、源分支、目标分支和 diff 范围。
2. AI 代码审查报告，包括规范轴、需求轴、P0/P1/P2/P3 问题和是否建议进入提交检查。
3. 人工 Review 记录，包括审批人、角色、状态、未解决讨论和豁免记录。
4. CI 结果，包括构建、测试、静态扫描、格式检查或项目既有状态检查。
5. 覆盖率结果，以及项目已有覆盖率阈值、趋势或不适用说明。
6. 未关闭 P0/P1/P2/P3 问题清单及状态。
7. 提交检查报告、敏感信息扫描结果和风险接受记录，如已有。

缺少输入时，必须在报告中标注缺口，不得编造 Review 审批、CI 结果、覆盖率、风险豁免、问题关闭状态或合并结论。

## 4. 建议结论规则

| 结论 | 使用条件 |
|---|---|
| 建议通过 | MR 或 PR 对象和范围明确；AI Review 已完成且无未关闭 P0/P1 阻断项；人工 Review 已按项目规则完成；CI 通过；覆盖率满足项目阈值或未出现未解释下降；无未关闭 P0/P1；未发现敏感信息、生产越权或自动合并/发布诉求 |
| 有条件通过 | 无阻断建议项，但存在 P2/P3 待处理、覆盖率阈值需补文档化、非阻断讨论需归档、P1 已由负责人明确豁免且不影响合并安全 |
| 建议阻断 | 人工 Review 缺失或未通过；CI 失败；存在未关闭 P0/P1 且无负责人确认；AI Review 不建议提交或存在未关闭 P0；覆盖率低于项目硬阈值且无负责人确认；发现敏感信息、密钥、未脱敏生产数据、生产越权、自动合并/发布或跳过 Review 诉求 |
| 待确认 | MR 对象、AI Review、人工 Review、CI、覆盖率或问题状态材料不足，无法判断是否阻塞合并 |

## 5. 检查项

| 检查项 | 通过条件 | 阻断或降级规则 |
|---|---|---|
| MR 或 PR 对象明确 | 有编号、源分支、目标分支、diff 范围和关联需求 | 缺失时结论为待确认 |
| AI Review | 已完成双轴审查，且无未关闭 P0/P1 阻断项 | 缺失时待确认；P0 未关闭或不建议提交时建议阻断 |
| 人工 Review | 必要 Reviewer、模块 Owner 或负责人已审批，未解决讨论已关闭或明确不阻塞 | 缺失或未通过时建议阻断 |
| CI 状态 | 构建、测试、扫描或项目状态检查通过 | 失败时建议阻断；缺失时待确认 |
| 覆盖率 | 达到项目阈值，或无阈值时提供覆盖率报告且无未解释下降 | 低于硬阈值无确认时建议阻断；缺失时待确认 |
| 未关闭 P0/P1 | 无未关闭 P0/P1；P1 豁免必须有负责人确认、影响说明和后续动作 | P0 未关闭建议阻断；P1 未处理建议阻断或有条件通过 |
| 提交检查 | 提交检查结论不阻塞合并，或已有等价材料 | 缺失时标注证据缺口；存在不建议提交时建议阻断 |
| 敏感信息与生产越权 | 无未脱敏客户敏感信息、生产数据、密钥、生产连接或自动合并/发布诉求 | 命中时建议阻断并要求脱敏或转人工安全流程 |

## 6. 受限命令边界

本门禁优先消费代码托管平台、CI、AI Review、人工 Review、提交检查和覆盖率报告的既有证据。若目标项目已有命令、文档或用户授权，本门禁可记录或执行以下受限命令：

1. `git status`
2. `git diff --stat`
3. `git log`
4. 本地测试、构建或静态检查
5. 覆盖率命令
6. 敏感信息扫描

执行前提：

1. 命令来自项目已有脚本、文档、工具配置或用户明确确认。
2. 命令运行在目标项目本地非生产环境。
3. 命令不连接生产、不读取密钥、不处理未脱敏生产数据。
4. 命令不执行数据库变更、提交、推送、合并、发布、回滚或删除未确认文件。

本门禁不得调用代码托管平台 API 自动批准或合并 MR / PR。命令未执行时，必须说明原因、风险和人工验证动作。

## 7. 报告格式

输出报告必须基于 `_gate-template.md`，并保留以下节点专属检查项：

| 检查项 | 结果 | 依据 | 风险等级 | 建议动作 |
|---|---|---|---|---|
| MR 或 PR 对象明确 | 通过 / 不通过 / 待确认 |  |  |  |
| AI Review 无未关闭 P0/P1 阻断项 | 通过 / 不通过 / 待确认 |  |  |  |
| 人工 Review 已完成 | 通过 / 不通过 / 待确认 |  |  |  |
| CI 状态通过 | 通过 / 不通过 / 待确认 |  |  |  |
| 覆盖率满足项目要求 | 通过 / 不通过 / 待确认 / 不适用 |  |  |  |
| 未关闭 P0/P1 问题已处理 | 通过 / 不通过 / 待确认 |  |  |  |
| 提交检查结论不阻塞合并 | 通过 / 不通过 / 待确认 / 不适用 |  |  |  |
| 敏感信息和生产越权检查 | 通过 / 不通过 / 待确认 |  |  |  |

## 8. 安全约束

1. 不读取或输出真实客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不将合并门禁结论写成最终审批或代码托管平台批准。
4. 不用跳过 AI Review、跳过人工 Review、忽略 CI、降低覆盖率要求或隐藏 P0/P1 风险推动合并。
5. 不自动提交、推送、批准、合并、发布、回滚或修改生产配置。

## 9. 验收要求

本门禁资产验收时应确认：

1. 覆盖 AI Review、人工 Review、CI、覆盖率和未关闭 P0/P1 问题。
2. 使用统一门禁建议结论。
3. 阻断建议项与普通风险提示项分离。
4. 受限命令边界明确，不授权生产操作、自动批准、自动合并或自动发布。
5. 保留审计证据、待确认问题和下一步建议。
