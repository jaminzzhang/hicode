# TDD 工作流

## 1. 触发条件

编码前需要明确测试标准，或任何会修改业务代码、测试代码、配置脚本或重构代码行为的任务准备启动时，触发本工作流。

纯只读解释任务不触发 TDD；实现、修复和重构任务必须提供 TDD 或测试先行证据。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/PRD_CONTEXT.md`
3. `docs/PROJ_CONTEXT.md`
4. `docs/CODING_RULES.md`
5. `docs/TESTING_GUIDE.md`
6. `docs/DEFECT_CASES.md`
7. 编码计划

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 编码计划 | 改动目标、影响范围、风险和实现步骤 |
| 需求上下文 | 业务规则、验收标准、测试关注点 |
| 代码上下文 | 相关类、接口、模块和已有测试 |

## 4. 执行步骤

1. 明确测试目标和测试范围。
2. 列出正常、边界、异常和历史缺陷场景。
3. 按金融核心系统风险标准补充保险核心业务逻辑严谨性、金额精度、交易一致性、状态流转、幂等、权限、审计、隐私和监管测试。
4. 按可观察行为和公开接口生成 Given-When-Then 用例，避免测试私有实现细节。
5. 按垂直切片规划 RED-GREEN-REFACTOR 循环：一次只覆盖一个行为，禁止先批量写完所有测试再批量实现。
6. 设计 Mock 策略。
7. 明确测试数据和脱敏要求。
8. 明确断言规则。
9. 输出单元测试代码或测试代码建议。

## 5. 输出

1. TDD 测试设计报告。
2. 测试场景清单。
3. Given-When-Then 用例。
4. 边界和异常场景。
5. Mock 策略。
6. 测试数据。
7. 断言规则。
8. 单元测试代码或建议。
9. 上下文更新建议。

## 6. 质量标准

1. 核心逻辑必须覆盖正常、边界、异常。
2. 金额断言必须使用精确数值口径。
3. 日期测试必须覆盖起保日、终保日、越界和闰年。
4. 外部系统必须 Mock。
5. 单元测试不得访问真实数据库。
6. 测试必须验证行为和公开接口，不测试私有方法或内部实现细节。
7. RED 状态不得重构；重构必须在 GREEN 后进行，并重新运行相关测试。

## 7. 安全约束

1. 测试数据不得包含真实客户敏感信息。
2. 不使用真实账号、Token、Cookie、Session。
3. 不读取生产配置或密钥文件。

## 8. 下一步

1. TDD 设计通过后，进入辅助编码。
2. 如发现规则不清，回到需求评审或编码计划补澄清问题。
3. 如发现历史缺陷模式，输出 `docs/DEFECT_CASES.md` 更新建议；经测试和研发共同确认后再沉淀。
