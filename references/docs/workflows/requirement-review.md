# 需求评审工作流

## 1. 触发条件

当需求准备进入开发前，或需求存在范围、规则、边界、测试、发布风险不清晰时，触发本工作流。

## 2. 必读上下文

1. `AGENTS.md`
2. `docs/DOMAIN_KNOWLEDGE.md`
3. `docs/PROJ_CONTEXT.md`
4. `docs/PRD_CONTEXT.md`
5. `docs/DEFECT_CASES.md`

## 3. 输入材料

| 输入 | 说明 |
|---|---|
| 需求文档 | 原始需求、故事、PRD 或需求链接 |
| 历史需求 | 相似需求或历史变更，若有 |
| 领域知识 | 业务域、高风险主题、核心场景 |
| 历史缺陷 | 相似缺陷模式和防范规则 |

## 4. 执行步骤

1. 摘要需求目标和业务范围。
2. 检查背景、目标、流程、规则、验收标准是否完整。
3. 按金融核心系统风险标准检查保险核心业务逻辑严谨性、金额、交易一致性、状态流转、幂等、权限、审计、客户隐私、监管、生产变更和回滚等高风险点。
4. 检查需求是否具备可开发性和可测试性。
5. 输出澄清问题、风险和测试关注点。
6. 提出 `docs/PRD_CONTEXT.md` 更新建议。

## 5. 输出

1. 需求评审结论：通过 / 有条件通过 / 不建议进入开发。
2. 需求摘要。
3. 完整性、明确性、一致性和可测试性检查。
4. 风险清单。
5. 澄清问题清单。
6. 测试关注点。
7. `docs/PRD_CONTEXT.md` 更新建议。

## 6. 质量标准

1. P0 澄清问题必须明确标出。
2. 高风险业务逻辑必须标注风险等级和建议确认人，保险核心业务规则不清晰时必须列为 `待确认`。
3. 不编造未确认业务规则。
4. 输出可直接用于编码计划输入。

## 7. 安全约束

1. 不输出客户敏感信息。
2. 示例数据必须脱敏或虚构。
3. 不读取生产配置或密钥文件。

## 8. 下一步

1. 若评审通过或有条件通过，输出 `docs/PRD_CONTEXT.md` 更新建议；经需求、研发或测试负责人确认后再沉淀。
2. 若存在 P0 澄清问题，先由负责人确认后再进入开发。
3. 进入 `coding-plan.md` 工作流。
