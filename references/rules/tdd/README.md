# TDD Rules

## 定位

本目录保存测试先行、RED-GREEN-REFACTOR、辅助编码和受控本地修改规则。它服务 `hicode:tdd`。

## 准入条件

1. 已有需求范围、编码计划或明确修复目标。
2. P0/P1 待确认问题已关闭，或当前任务只做只读测试设计。
3. 已确认允许修改的文件范围和验证方式。
4. 测试数据已脱敏或使用虚构数据。

## RED-GREEN-REFACTOR

1. RED：先写或说明一个失败测试，覆盖一个可观察行为。
2. GREEN：只做最小实现使该测试通过，不扩大重构范围。
3. REFACTOR：在测试保护下整理实现，保留行为不变。
4. 每轮记录测试目标、修改文件、命令结果和未验证原因。

## 测试设计规则

| 维度 | 要求 |
|---|---|
| 场景 | 覆盖正常、边界、异常、历史缺陷和安全场景 |
| 核心风险 | 金额、日期、状态、责任、权限、外部调用、幂等和审计必须有测试或豁免依据 |
| Mock | 单元测试不得访问真实数据库、真实外部系统、真实账号、Cookie、Session 或生产配置 |
| 数据 | 客户、保单、账号、Token、生产响应必须脱敏或虚构 |
| 断言 | 金额用精确断言；日期、状态、异常、外部调用和审计字段要有明确断言 |

## 辅助编码规则

1. 修改必须受需求、计划、测试和用户确认范围约束。
2. 不删除测试、不降低断言、不绕过异常处理、权限、日志或审计。
3. 涉及生产代码实现时，必须说明对应测试证据。
4. 无法运行测试时，必须说明原因、风险和人工验证动作。
5. 解释模式只读，不修改文件。

## 输出要求

输出应使用 `references/templates/tdd/tdd-report.md`，并包含：

1. TDD 结论和准入状态。
2. 测试目标、范围和场景清单。
3. Given-When-Then 用例。
4. Mock、测试数据和断言规则。
5. RED-GREEN-REFACTOR 记录。
6. 修改文件清单。
7. 受限命令执行记录。
8. 风险、待确认问题和上下文更新建议。
