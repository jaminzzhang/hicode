# 提测门禁

## 1. 定位

本门禁用于开发完成后进入提测前的提醒型检查，判断当前分支、Commit 或 MR 是否具备编译、单元测试、TDD 证据、AI Review、提交检查、核心测试场景和敏感信息检查证据。

本门禁不替代开发人员、测试负责人、研发负责人、Reviewer、CI/CD 或测试流程的最终判断，不自动提交、不自动推送、不自动创建 MR、不自动流转提测、不自动发布。

## 2. 适用场景

使用本门禁的场景：

1. 开发完成，准备将变更交付测试。
2. 已有 TDD、辅助编码、代码审查或提交检查报告，需要形成提测节点门禁结论。
3. 需要检查编译、单元测试、AI Review、提交检查、核心测试场景和敏感信息证据是否齐备。
4. 存在 P0/P1 Review 问题、测试失败、核心场景缺口或敏感信息风险，需要判断是否建议阻断提测。

不适用场景：

1. 当前需求尚未完成编码准入；应先使用 `coding-entry-gate.md`。
2. 当前目标是设计 TDD 测试；应先使用 `tdd` Skill。
3. 当前目标是深度审查代码；应先使用 `code-review` Skill。
4. 当前目标是提交材料检查或 MR 描述草稿；应先使用 `pre-commit-check` Skill。
5. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。

## 3. 输入材料

至少需要：

1. 需求或故事编号、分支、Commit、MR 或变更范围。
2. 编码准入门禁报告，或等价的编码计划和上下文清晰证据。
3. TDD Skill 报告、测试先行证据或明确不适用说明。
4. 编译或构建结果。
5. 单元测试结果，以及失败说明或未执行原因。
6. AI 代码审查报告，包括 P0/P1/P2/P3 问题状态。
7. 提交检查 Skill 报告，包括敏感信息、SQL、配置、脚本、接口文档和发布影响检查。
8. 核心场景测试建议或核心场景测试报告。
9. P0/P1 问题修复、豁免或负责人确认记录。

缺少输入时，必须在报告中标注缺口，不得编造测试结果、Review 结论、扫描结果、覆盖率、负责人确认或提测结论。

## 4. 建议结论规则

| 结论 | 使用条件 |
|---|---|
| 建议通过 | 检查对象和变更范围明确；代码可编译或构建通过；单元测试通过；TDD 或测试先行证据可追溯；AI Review 无未关闭 P0，P1 已修复或有明确不阻塞提测的负责人确认；提交检查结论为建议提交或有条件提交且条件不阻塞提测；核心测试场景已生成；未发现敏感信息或生产越权 |
| 有条件通过 | 无阻断建议项，但存在 P1 已豁免需归档、P2/P3 待处理、覆盖率或扫描证据需补充、核心场景需测试负责人确认等不阻塞提测的问题 |
| 建议阻断 | 编译或构建失败；单元测试失败且无合理解释；存在未关闭 P0；P1 未修复且无负责人确认；AI Review 或提交检查结论不建议提交；核心业务测试场景缺失且影响测试接收；发现敏感信息、密钥、未脱敏生产数据、生产越权、跳过测试或绕过 Review 诉求 |
| 待确认 | 检查对象、变更范围、编译结果、单测结果、AI Review、提交检查、核心场景或 P0/P1 状态材料不足，无法判断是否阻塞 |

## 5. 检查项

| 检查项 | 通过条件 | 阻断或降级规则 |
|---|---|---|
| 检查对象明确 | 有需求或故事、分支、Commit、MR 或 diff 范围 | 缺失时结论为待确认 |
| 编码准入结果 | 已通过编码准入，或有等价编码计划和上下文清晰证据 | 缺失时待确认；前置阻断未解除时建议阻断 |
| TDD 或测试先行证据 | 已提供 TDD 报告、测试先行记录或明确不适用说明 | 缺失时不得建议通过；核心业务逻辑变更缺失时建议阻断 |
| 编译或构建结果 | 编译或构建通过 | 失败时建议阻断；未执行且无原因时待确认 |
| 单元测试结果 | 单元测试通过，或未执行原因和人工验证动作明确且风险可接受 | 失败未解释时建议阻断；缺失时待确认 |
| AI Review | 无未关闭 P0；P1 已修复或有负责人确认 | P0 未关闭建议阻断；P1 未处理时建议阻断或有条件通过 |
| 提交检查 | 提交检查建议提交或有条件提交且条件不阻塞提测 | 缺失时待确认；不建议提交时建议阻断 |
| 核心测试场景 | 已生成必测、回归、边界、异常和安全隐私场景，或明确不适用 | 涉及核心业务但缺失时建议阻断或待确认 |
| 敏感信息与生产越权 | 无未脱敏客户敏感信息、生产数据、密钥、生产连接或自动发布/合并诉求 | 命中时建议阻断并要求脱敏或转人工安全流程 |

## 6. 受限命令边界

本门禁优先消费已有 TDD、编译、单测、AI Review、提交检查和核心场景测试证据。若目标项目已有命令、文档或用户授权，本门禁可记录或执行以下受限命令：

1. `git status`
2. `git diff --stat`
3. 本地编译或构建
4. 本地单元测试
5. 静态检查或格式检查
6. 敏感信息扫描

执行前提：

1. 命令来自项目已有脚本、文档、工具配置或用户明确确认。
2. 命令运行在目标项目本地非生产环境。
3. 命令不连接生产、不读取密钥、不处理未脱敏生产数据。
4. 命令不执行数据库变更、提交、推送、合并、发布、回滚或删除未确认文件。

命令未执行时，必须说明原因、风险和人工验证动作。敏感信息扫描命中时，不输出完整敏感值，只输出掩码和处理建议。

输出报告必须基于 `_gate-template.md`，并保留以下节点专属检查项：

|---|---|---|---|---|
| 变更范围明确 | 通过 / 不通过 / 待确认 |  |  |  |
| TDD 或测试先行证据可追溯 | 通过 / 不通过 / 待确认 / 不适用 |  |  |  |
| 编译或构建结果通过 | 通过 / 不通过 / 待确认 |  |  |  |
| 单元测试结果通过 | 通过 / 不通过 / 待确认 |  |  |  |
| AI Review 无未关闭 P0/P1 阻断项 | 通过 / 不通过 / 待确认 |  |  |  |

## 8. 安全约束

1. 不读取或输出真实客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不将提测门禁结论写成最终测试接收、流程审批或质量负责人确认。
4. 不用跳过测试、删除测试、降低断言、跳过 Review 或隐藏 P0/P1 风险推动提测。
5. 不自动提交、推送、合并、发布、回滚或修改生产配置。

## 9. 验收要求

本门禁资产验收时应确认：

1. 覆盖编译、单元测试、AI Review、提交检查、核心测试场景和敏感信息。
2. 使用统一门禁建议结论。
3. 阻断建议项与普通风险提示项分离。
4. 受限命令边界明确，不授权生产操作、自动提交、自动合并或自动发布。
5. 保留审计证据、待确认问题和下一步建议。
