# 编码准入门禁

## 1. 定位

本门禁用于编码开始前的提醒型检查，判断已完成需求准入的需求或故事是否具备编码计划、上下文清晰门槛、ADR 判断和 P0 待确认问题关闭证据。

本门禁不替代研发负责人、架构负责人、模块 Owner 或技术负责人的最终判断，不自动创建分支、不自动修改代码、不自动合并、不自动发布。

## 2. 适用场景

使用本门禁的场景：

1. 需求准入门禁已输出建议通过或有条件通过。
2. 编码计划 Skill 已输出报告，需要判断是否可以开始编码。
3. 需要检查上下文是否足够清晰、明确、无歧义、无冲突。
4. 需要检查 ADR 判断、P0 待确认问题和编码前置风险。

不适用场景：

1. 需求尚未完成需求准入。
2. 当前目标是生成编码计划；应先使用 `coding-plan` Skill。
3. 当前目标是进入 TDD 或辅助编码的具体执行；应先通过本门禁，再使用对应 Skill。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。

## 3. 输入材料

至少需要：

1. 需求准入门禁报告。
2. 编码计划 Skill 报告。
3. `docs/PRD_CONTEXT.md` 和 `docs/PROJ_CONTEXT.md`，或已确认等价上下文材料。
4. ADR 判断结果，以及需要 ADR 时的技术负责人确认记录或 ADR 草稿链接。
5. P0/P1 待确认问题清单及状态。
6. 影响范围摘要，包括模块、接口、数据、配置、外部依赖或明确不涉及说明。

缺少输入时，必须在报告中标注缺口，不得编造实现范围、类名、接口、表名、ADR 或负责人确认。

## 4. 建议结论规则

| 结论 | 使用条件 |
|---|---|
| 建议通过 | 需求准入门禁建议通过或有条件通过且条件不阻塞编码；编码计划结论为建议启动编码或有条件启动编码；上下文清晰门槛为充足或有条件充足；无未关闭 P0；ADR 判断已给出；未发现敏感信息或生产越权 |
| 有条件通过 | 无阻断建议项，但存在 P1/P2 待确认、ADR 确认需归档、上下文补充项不阻塞第一轮 TDD 或局部编码 |
| 建议阻断 | 需求未准入；编码计划缺失或不建议启动编码；上下文清晰门槛不足；存在未关闭 P0；ADR 判断缺失且涉及难逆架构或治理决策；发现敏感信息、密钥、未脱敏生产数据或生产越权 |
| 待确认 | 检查对象、需求准入报告、编码计划报告、ADR 判断或影响范围材料不足，无法判断是否阻塞 |

## 5. 检查项

| 检查项 | 通过条件 | 阻断或降级规则 |
|---|---|---|
| 需求准入结果 | 需求准入门禁建议通过，或有条件通过且条件不阻塞编码 | 缺失时待确认；阻断项未解除时建议阻断 |
| 编码计划报告 | 已提供且结论为建议启动编码或有条件启动编码 | 缺失时待确认；不建议启动编码时建议阻断 |
| 上下文清晰门槛 | 充足或有条件充足 | 不足时建议阻断 |
| P0 待确认问题 | 全部关闭 | 任一未关闭 P0 建议阻断 |
| ADR 判断 | 已给出是否需要 ADR；需要时已有确认记录或明确阻断 | 缺失且涉及难逆决策时建议阻断 |
| 影响范围 | 模块、接口、数据、配置、外部依赖等已明确或明确不涉及 | 不足以设计 TDD 时建议阻断或待确认 |
| 敏感信息与生产越权 | 无未脱敏客户敏感信息、生产数据、密钥或生产操作诉求 | 命中时建议阻断并要求脱敏或转人工安全流程 |

## 6. 受限命令边界

本门禁默认不执行命令。只读取需求准入报告、编码计划报告、上下文文档、ADR 判断和人工确认材料。

本门禁不得执行代码修改、测试、构建、数据库变更、发布、回滚、自动提交、自动合并、生产连接或删除文件。测试和构建命令应在后续 TDD、辅助编码、提交检查或具体门禁中按受限命令边界执行并记录。

## 7. 报告格式

输出报告必须基于 `_gate-template.md`，并保留以下节点专属检查项：

| 检查项 | 结果 | 依据 | 风险等级 | 建议动作 |
|---|---|---|---|---|
| 需求准入结论不阻塞编码 | 通过 / 不通过 / 待确认 |  |  |  |
| 编码计划报告结论可追溯 | 通过 / 不通过 / 待确认 |  |  |  |
| 上下文清晰门槛满足 | 通过 / 不通过 / 待确认 |  |  |  |
| P0 待确认问题关闭 | 通过 / 不通过 / 待确认 |  |  |  |
| ADR 判断已给出 | 通过 / 不通过 / 待确认 / 不适用 |  |  |  |
| 影响范围足以进入 TDD | 通过 / 不通过 / 待确认 |  |  |  |
| 敏感信息和生产越权检查 | 通过 / 不通过 / 待确认 |  |  |  |

## 8. 安全约束

1. 不读取或输出真实客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不将编码准入结论写成最终审批或开工命令。
4. 不在上下文不足、P0 未关闭或 ADR 判断缺失时建议直接编码。
5. 不通过跳过 TDD、删除测试、降低断言或隐藏风险推动编码开始。

## 9. 验收要求

本门禁资产验收时应确认：

1. 覆盖编码计划、上下文清晰门槛、ADR 判断和 P0 待确认问题。
2. 使用统一门禁建议结论。
3. 阻断建议项与普通风险提示项分离。
4. 默认不执行命令。
5. 保留审计证据、待确认问题和下一步建议。
