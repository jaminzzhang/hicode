# Skill 名称

## 1. 定位

本 Skill 用于【研发流程节点】中的【具体任务】，负责路由上下文、执行步骤、权限边界、验证记录和输出模板引用。

本 Skill 不替代业务负责人、研发负责人、测试负责人、架构师、Review 人、发布负责人或既有审批流程的最终判断。

## 2. 适用场景

使用本 Skill 的场景：

1. 【触发条件 1】。
2. 【触发条件 2】。
3. 用户明确要求使用本 Skill，且输入材料不违反安全规则。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 【应改用其他 Prompt 或 Skill 的情况】。
2. 缺少关键输入，无法判断范围、目标或风险。
3. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
4. 用户要求自动合并、自动发布、生产操作或越权操作。

## 4. 必读上下文

执行前按需读取：

1. `AGENTS.md`
2. `docs/workflows/【workflow-name】.md`
3. `.hicode/prompts/【prompt-name】.md`
4. `docs/PRD_CONTEXT.md`
5. `docs/PROJ_CONTEXT.md`
6. `docs/DOMAIN_KNOWLEDGE.md`
7. `docs/CODING_RULES.md`
8. `docs/TESTING_GUIDE.md`
9. `docs/REVIEW_RULES.md`
10. `docs/RELEASE_GUIDE.md`
11. `docs/DEFECT_CASES.md`
12. 本目录的 `output-template.md`

只读取当前任务必要上下文。缺少上下文时，输出缺口和影响，不补编事实。

## 5. 输入材料

要求用户或调用方提供：

1. 【输入 1】。
2. 【输入 2】。
3. 【输入 3】。
4. 已知风险、验证结果、负责人确认或豁免说明，如有。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 读取必需上下文，列出已读取材料和缺失材料。
3. 检查输入是否满足准入条件。
4. 按对应 Prompt 的检查维度执行分析、生成或检查。
5. 使用 `output-template.md` 输出结果。
6. 记录验证动作、受限命令执行结果或未执行原因。
7. 输出风险、建议动作、待确认问题和上下文更新建议。
8. 遇到敏感信息、生产越权或 P0/P1 风险时停止推进，并给出阻断或人工确认建议。

## 7. 输出

必须输出：

1. 结论。
2. 依据。
3. 输入范围或检查范围。
4. 风险清单。
5. 建议动作。
6. 待确认问题。
7. 验证要求或执行记录。
8. 上下文更新建议。

输出结构以本目录的 `output-template.md` 为准；如对应 Prompt 有更具体格式，优先使用对应 Prompt 的输出要求。

## 8. 质量标准

输出必须满足：

1. 结论可追溯到输入、上下文、工具结果或负责人确认。
2. 风险按 P0/P1/P2/P3 分级。
3. 不确定内容标注 `需确认` 或 `待确认`。
4. 建议动作具体到文档、测试、负责人、验证方式或下一流程节点。
5. 不复制对应 Prompt 全文，不维护第二套规则。

## 9. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不自动提交，不自动合并 MR / PR，不自动发布，不自动回滚，不自动修改生产配置。
6. 不直接操作生产环境，不直接给出生产操作命令。
7. 不执行数据库变更、生产连接、发布、回滚或删除未确认文件。
8. 不通过删除测试、降低断言、跳过 Review 或隐藏风险推动流程通过。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 10. 验证要求

完成后必须说明：

1. 实际读取的上下文。
2. 实际执行的验证动作或未执行原因。
3. 是否存在证据缺口。
4. 是否需要人工确认。
5. 是否需要更新目标项目 `docs/`、目标项目 `.hicode/` 或本仓库 hicode 根目录源资产。

## 11. 上下文更新

如发现稳定规则、术语、历史缺陷、ADR 决策或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库或 ADR 前，必须由对应负责人确认。
