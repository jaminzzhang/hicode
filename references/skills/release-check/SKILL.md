# 发布检查 Skill

## 1. 定位

本 Skill 用于发布申请前、发布准入门禁前或发布范围变更后的发布前证据汇总检查，负责核对发布范围、需求清单、分支制品、Commit/MR、代码审查、提交检查、核心场景测试、缺陷、SQL、配置、生产验证计划和回滚方案。

本 Skill 默认消费 `core-scenario-test` Skill 的测试侧报告，但不重新设计详细测试用例；核心场景测试证据不足时，应输出证据缺口并建议回到测试负责人流程或 `core-scenario-test` Skill。

本 Skill 不替代发布负责人、研发负责人、测试负责人、审批流程、CI/CD 或发布平台的最终判断，不自动发布、不自动回滚、不自动合并、不修改生产配置、不操作生产环境。

## 2. 适用场景

使用本 Skill 的场景：

1. 准备提交发布申请、发布准入门禁或人工发布审批前。
2. 发布范围、分支、制品、SQL、配置、脚本、接口、测试报告、缺陷或回滚方案发生变化后。
3. 需要汇总代码审查、提交检查、核心场景测试、测试报告、缺陷关闭和发布材料，形成可归档发布检查报告。
4. 需要输出生产验证计划、上线后观察建议和回滚关注点，但不执行生产操作。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 当前目标是设计详细测试用例或整理核心场景测试证据；应先使用 `core-scenario-test` Skill。
2. 当前目标是代码审查、提交检查或 MR/PR 描述草稿；应使用 `code-review` 或 `pre-commit-check` Skill。
3. 缺少发布范围、发布需求清单、分支与制品信息，无法判断发布对象；只能输出待确认和证据缺口。
4. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
5. 用户要求自动发布、自动回滚、修改生产配置、连接生产环境、读取生产日志、执行生产 SQL 或输出生产操作命令。

## 4. 必读上下文

执行前按需读取：

1. `AGENTS.md`
2. `CONTEXT.md`
3. `docs/workflows/release-check.md`
4. `.hicode/prompts/release-check.md`
5. `docs/RELEASE_GUIDE.md`
6. `docs/PRD_CONTEXT.md`
7. `docs/PROJ_CONTEXT.md`
8. `docs/TESTING_GUIDE.md`
9. `docs/DOMAIN_KNOWLEDGE.md`
10. `docs/DEFECT_CASES.md`
11. `docs/CODING_RULES.md`，涉及配置、脚本、日志、权限或敏感信息时读取
12. 需求评审报告、编码计划、TDD 报告、核心场景测试报告、代码审查报告、提交检查报告，如已有
13. 神兵版本链接或版本标识、发布需求清单、代码分支、Commit/MR 清单、制品版本、发布说明和发布影响范围
14. SQL、配置、脚本、接口文档、回滚方案、生产验证点、上线后观察建议和负责人确认记录，如有
15. 测试报告、覆盖率、CI、缺陷关闭情况、风险接受说明和人工豁免记录，如有
16. 本目录的 `output-template.md`

只读取当前发布检查必要上下文。缺少上下文时，输出缺口和影响，不补编需求清单、分支、制品、Commit、MR、SQL、配置、测试结果、缺陷状态、验证点、回滚方案或发布结论。

## 5. 输入材料

要求用户或调用方提供：

1. 神兵版本链接或版本标识、发布目标、发布窗口说明或发布申请摘要。
2. 发布需求清单、需求编号、发布范围和本次不发布范围说明。
3. 代码分支、目标分支、Commit/MR 清单、制品版本和构建记录。
4. SQL、配置、脚本、接口文档、发布说明、回滚说明和生产验证点清单。
5. 核心场景测试报告、测试报告、覆盖率、CI 结果、缺陷关闭情况和风险接受说明。
6. 代码审查报告、提交检查报告、未解决讨论、负责人确认或人工豁免记录，如有。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 准入规则

进入可建议状态前必须满足：

1. 有发布范围，包括发布目标、发布需求清单、本次发布范围和本次不发布范围。
2. 有分支与制品信息，包括代码分支、目标分支、Commit/MR 清单、构建记录或制品版本。
3. 有可追溯测试证据入口，包括核心场景测试报告、测试报告、CI、覆盖率或缺陷关闭情况。
4. 缺少发布范围、发布需求清单或制品信息时，只能输出 `待确认`。
5. 缺少核心场景测试报告时，可以检查其他发布材料缺口，但不得给出无条件建议发布。
6. 存在未关闭 P0/P1、核心场景缺失、测试失败未解释、SQL/配置/回滚缺失、敏感信息或生产越权时，必须输出不建议发布和阻断建议。

## 7. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止检查并要求先脱敏或改走人工安全流程。
3. 执行发布材料准入检查，明确发布范围、需求清单、分支制品和测试证据入口是否齐备。
4. 核对发布需求清单、本次发布范围和本次不发布范围。
5. 核对分支、制品、Commit/MR、构建记录和发布申请一致性。
6. 汇总代码审查、提交检查、核心场景测试、测试报告、覆盖率、CI 和缺陷关闭证据。
7. 检查金融核心系统风险基线是否已有结论。
8. 检查 SQL、配置、脚本、接口兼容性、发布说明和回滚说明。
9. 输出生产验证计划和上线后观察建议，只写计划、数据来源类型、观察窗口和责任角色。
10. 检查回滚方案是否存在、明确、可验证。
11. 在具备条件时执行低风险本地只读核对或本地构建/测试结果复核；无法执行时说明原因、风险和人工验证动作。
12. 输出发布结论、阻断建议、发布风险、待确认问题、验证要求和上下文更新建议。

## 8. 受限命令边界

可执行的受限命令：

1. `git status`
2. `git log`
3. `git diff --stat`
4. 本地构建结果复核。
5. 本地测试结果复核。
6. 本地静态检查或格式检查结果复核。

执行前提：

1. 命令来自项目已有脚本、文档、工具配置或用户确认。
2. 命令运行在目标项目本地非生产环境。
3. 命令不连接生产、不读取生产日志、不调用生产接口、不读取密钥、不处理未脱敏生产数据。
4. 命令不执行发布、回滚、数据库变更、生产连接、提交、推送、合并或删除未确认文件。

本 Skill 不执行生产验证命令。生产验证只能输出验证计划和人工确认入口。

## 9. 生产验证计划边界

可以输出：

1. 验证点。
2. 验证目标。
3. 预期结果。
4. 观察窗口。
5. 责任角色。
6. 人工确认的数据来源类型。
7. 上线后观察项、观察周期、异常处理路径和责任角色。

禁止输出：

1. 生产命令。
2. 生产 SQL。
3. 生产日志查询语句。
4. 生产接口调用命令。
5. 生产 IP、账号、Token、配置值。
6. 真实客户、保单、证件、手机号、银行卡或地址信息。
7. 可直接执行的生产回滚命令。

必须依赖生产数据时，只能提示发布负责人在既有平台中脱敏确认。

## 10. 输出

必须输出：

1. 发布结论：建议发布 / 有条件发布 / 不建议发布 / 待确认。
2. 依据。
3. 发布材料准入检查。
4. 发布范围。
5. 需求清单核对。
6. 分支和制品核对。
7. Commit / MR 核对。
8. 代码审查与提交检查结果。
9. 核心场景测试证据。
10. 测试报告与缺陷关闭情况。
11. SQL 检查。
12. 配置 / 脚本 / 接口检查。
13. 金融核心系统风险基线。
14. 生产验证计划。
15. 回滚方案检查。
16. 发布风险。
17. 上线后观察建议。
18. 受限命令执行记录。
19. 阻断建议。
20. 待确认问题。
21. 验证要求。
22. 上下文更新建议。
23. 下一步建议。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的发布检查 Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 11. 质量标准

输出必须满足：

1. 发布结论只能使用建议发布 / 有条件发布 / 不建议发布 / 待确认，不输出发布批准、最终审批或最终门禁通过。
2. 缺少发布范围、发布需求清单或制品信息时，只能输出待确认。
3. 缺少核心场景测试报告时，不得给出无条件建议发布。
4. 未关闭 P0/P1、核心场景缺失、测试失败、SQL/配置/回滚缺失、敏感信息或生产越权必须给出不建议发布和阻断建议。
5. 每个发布结论都能追溯到输入材料、工具结果、测试报告、Review 报告、提交检查报告或负责人确认。
6. 生产验证计划只写目标、预期、观察和人工确认口径，不写生产操作细节。
7. 输出能支持发布负责人判断是否进入人工发布审批，但不替代发布负责人最终判断。

## 12. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不自动发布，不自动回滚，不自动合并 MR / PR，不自动修改生产配置。
6. 不连接生产环境，不读取生产日志，不调用生产接口，不执行生产 SQL。
7. 不直接给出生产操作命令、生产 SQL、生产日志查询、生产接口调用、生产配置值或真实客户/保单信息。
8. 不为了推动发布而隐藏 P0/P1 风险、忽略测试缺口、跳过回滚方案或弱化敏感信息风险。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 13. 验证要求

完成后必须说明：

1. 实际读取的上下文。
2. 发布范围、需求清单、分支与制品、测试证据入口是否齐备。
3. 是否读取核心场景测试报告，以及该报告是否建议进入发布检查。
4. 是否存在代码审查、提交检查、测试、缺陷、SQL、配置、脚本、接口、回滚或生产验证计划证据缺口。
5. 实际执行的受限命令、结果、失败摘要和未执行原因。
6. 是否需要发布负责人、研发负责人、测试负责人、安全负责人、业务负责人或运维/发布平台负责人确认。
7. 是否需要更新目标项目 `docs/`、目标项目 `.hicode/`、目标项目 `.hicode/gates/` 或本仓库 hicode 根目录源资产。

## 14. 上下文更新

如发现稳定发布规则、发布材料要求、回滚规则、生产验证计划口径、历史缺陷、测试证据缺口或模板缺口，只能输出更新建议。正式写入长期上下文、发布指南、缺陷库、测试规范或门禁资产前，必须由对应负责人确认。
