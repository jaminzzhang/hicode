# 编码计划辅助 Skill

## 1. 定位

本 Skill 用于编码启动前的轻量编码计划和 ADR 判断，负责确认需求上下文是否已经足够清晰、明确、无歧义、无冲突，再输出影响范围、建议步骤、TDD 重点、核心场景建议、风险、待确认问题和上下文更新建议。

本 Skill 借鉴 `grill-with-docs` 的文档驱动追问纪律：对照现有术语和上下文挑战模糊表述，能从仓库文档或代码判断的问题先检索本地证据，无法确认的问题输出具体澄清问题和推荐答案。

本 Skill 不直接修改代码，不生成完整核心交易实现，不替代架构师、研发负责人、测试负责人或业务负责人的最终决策。

## 2. 适用场景

使用本 Skill 的场景：

1. 需求评审已通过或有条件通过，准备进入编码、TDD 或辅助实现前。
2. 已提供需求摘要、`docs/PRD_CONTEXT.md`、需求评审报告、模块上下文或代码范围，需要形成可执行编码计划。
3. 需求涉及保险/金融核心系统风险，需要在实现前明确业务规则、状态、金额、权限、审计、交易一致性、幂等、测试和发布边界。
4. 需求可能触发 ADR，需要判断是否存在难逆、需解释且有真实取舍的架构或治理决策。
5. 用户要求在实现代码前先确认上下文足够理解、需求清晰明确、无歧义、无冲突。

## 3. 不适用场景

不要使用本 Skill 的场景：

1. 需求评审未完成，或 P0/P1 澄清问题仍未关闭；应先回到 `requirement-review` Skill。
2. 目标是直接生成测试用例或执行 TDD；应使用 `tdd` Skill 或 `.hicode/prompts/tdd.md`。
3. 目标是直接修改代码、修复缺陷或重构；应先有本 Skill 的编码计划，再进入辅助编码 Skill。
4. 目标是代码 diff 审查、提交检查、核心场景测试或发布检查；应使用对应 Skill 或 Prompt。
5. 输入包含未脱敏客户敏感信息、生产数据、密钥、`.env`、生产配置或生产凭证。
6. 用户要求自动创建或合入 ADR、自动提交、自动合并、自动发布、连接生产或修改生产配置。

## 4. 必读上下文

执行前按需读取：

1. `AGENTS.md`
2. `docs/workflows/coding-plan.md`
3. `.hicode/prompts/coding-plan.md`
4. `docs/PRD_CONTEXT.md`
5. 需求评审报告，如已有
6. `docs/PROJ_CONTEXT.md`
7. `docs/DOMAIN_KNOWLEDGE.md`
8. `docs/CODING_RULES.md`
9. `docs/TESTING_GUIDE.md`
10. `docs/DEFECT_CASES.md`
11. `docs/ADR/README.md`
12. 相关历史 ADR，如影响架构或治理决策
13. 相关代码目录、接口、表、配置、批处理或检索结果，如已有
14. 本目录的 `output-template.md`

只读取当前需求必要上下文。缺少上下文时，输出缺口和影响，不补编类、表、接口、配置、业务规则或测试结论。

## 5. 输入材料

要求用户或调用方提供：

1. 需求编号、需求摘要或 `docs/PRD_CONTEXT.md`。
2. 需求评审报告、未关闭澄清问题和负责人确认记录，如已有。
3. 相关系统、模块、类、接口、表、配置、批处理或外部依赖范围。
4. 相关代码片段、目录结构、检索结果、diff 摘要或模块负责人说明，如已有。
5. 已知测试要求、历史缺陷、发布限制、回滚要求或 ADR 背景。

输入可以是摘要、脱敏片段、文件路径或工具输出。不得要求真实客户数据、未脱敏生产日志、密钥文件、生产配置文件或生产凭证。

## 6. 执行步骤

1. 判断本 Skill 是否适用；不适用时说明原因并路由到正确 Prompt、Skill 或人工流程。
2. 检查输入是否包含敏感信息、生产数据、密钥或生产越权诉求；如存在，停止计划并要求先脱敏或改走人工安全流程。
3. 读取必需上下文，列出已读取材料、缺失材料和缺失影响。
4. 执行上下文清晰门槛检查：
   - 需求是否有明确目标、范围、业务规则、验收标准和确认人。
   - 术语是否与 `docs/DOMAIN_KNOWLEDGE.md`、`docs/PROJ_CONTEXT.md`、`docs/PRD_CONTEXT.md` 一致。
   - 是否存在“按原逻辑”“正常情况”“适当”“默认”“特殊情况”等模糊表述。
   - 业务规则、流程、状态、金额、日期、权限、审计和异常处理是否有冲突。
   - 能从本地上下文、代码检索或负责人确认回答的问题是否已经有证据。
5. 若上下文不清晰，输出澄清问题、推荐答案、建议确认人和补齐材料；不得建议直接启动编码。
6. 若上下文达到准入门槛，按 `.hicode/prompts/coding-plan.md` 的检查维度输出编码计划。
7. 识别影响范围，包括系统、模块、类、方法、接口、数据库、配置、SQL、脚本、批处理、外部依赖、权限、日志、测试和发布材料。
8. 输出 TDD 测试重点和核心场景测试建议，确保后续实现受测试和风险边界约束。
9. 判断是否需要新增或更新 ADR。ADR 只输出判断和草稿建议，不自动创建、确认或合入。
10. 输出 `docs/PROJ_CONTEXT.md`、`docs/PRD_CONTEXT.md`、`docs/DOMAIN_KNOWLEDGE.md`、`docs/DEFECT_CASES.md`、`docs/ADR/` 或 Harness 资产的更新建议；正式沉淀前必须由对应负责人确认。
11. 记录验证动作、代码检索结果、未执行命令原因和证据缺口。本 Skill 默认为 L1 只读分析和计划建议，不直接修改文件或执行业务代码。

## 7. 输出

必须输出：

1. 计划结论：建议启动编码 / 有条件启动编码 / 不建议启动编码 / 待确认。
2. 上下文清晰门槛结论：充足 / 有条件充足 / 不足 / 待确认。
3. 依据和输入范围。
4. 术语与上下文校准结果。
5. 需求摘要和本次改动目标。
6. 需求清晰度、歧义和冲突检查。
7. 影响范围清单。
8. 建议实现步骤。
9. TDD 测试重点。
10. 核心场景测试建议。
11. 风险清单。
12. 待确认问题。
13. ADR 判断。
14. 验证要求和证据缺口。
15. 上下文更新建议。
16. 下一步建议。

输出结构以本目录的 `output-template.md` 为准；如目标项目已安装的编码计划 Prompt 有更具体格式，优先使用 Prompt 的输出要求。

## 8. 质量标准

输出必须满足：

1. 结论可追溯到需求评审、需求上下文、项目上下文、领域知识、代码检索、历史缺陷或负责人确认。
2. 上下文不清晰、需求有歧义、规则冲突或 P0/P1 待确认问题未关闭时，不得建议直接启动编码。
3. 影响范围必须覆盖代码、数据库、配置、接口、外部依赖、权限、日志、测试和发布材料；没有证据的范围标注 `需确认`。
4. 建议实现步骤只描述小步实现边界，不输出完整核心交易代码。
5. TDD 重点和核心场景测试建议必须覆盖正常、边界、异常、回归和高风险金融核心场景。
6. 风险按 P0/P1/P2/P3 分级，并说明影响、建议动作和建议确认人。
7. ADR 判断必须说明是否难逆、是否有真实备选方案、是否需要未来读者理解背景。
8. 不复制编码计划 Prompt 全文，不维护第二套编码计划规则。

## 9. 安全约束

必须遵守：

1. 不读取或输出密钥、Token、Cookie、Session、连接串、生产账号、生产 IP 或内部密钥。
2. 不读取 `.env`、密钥文件、生产配置文件或生产凭证。
3. 不处理未脱敏真实客户数据或未脱敏生产数据。
4. 不输出客户姓名、证件号、手机号、银行卡号、保单号、客户号、地址、邮箱等敏感信息。
5. 不直接操作生产环境，不直接给出生产操作命令。
6. 不直接修改代码，不生成完整核心交易实现，不删除测试，不降低断言，不绕过 Review。
7. 不自动创建、确认或合入 ADR。
8. 不自动提交，不自动合并 MR / PR，不自动发布，不自动修改生产配置。
9. 不为了推动需求进入实现而隐藏 P0/P1 风险、降低风险等级或删减待确认问题。

如果输入包含敏感信息，请要求用户先脱敏；如已经出现敏感内容，输出中必须使用掩码。

## 10. 验证要求

完成后必须说明：

1. 实际读取的上下文。
2. 是否读取了需求评审报告和 `docs/PRD_CONTEXT.md`。
3. 是否对照领域术语、项目上下文和历史缺陷做过冲突检查。
4. 是否检索或读取了相关代码、接口、表、配置或 ADR。
5. 是否存在无法判断的需求范围、业务规则、系统影响、测试或发布证据缺口。
6. 是否需要业务负责人、产品/BA、研发负责人、测试负责人、安全负责人、架构师或模块负责人确认。
7. 是否需要更新目标项目 `docs/`、目标项目 `.hicode/` 或本仓库 hicode 根目录源资产。

## 11. 上下文更新

如发现稳定业务规则、术语、历史缺陷、模块影响、接口约束、ADR 决策或模板缺口，只能输出更新建议。正式写入长期上下文、缺陷库或 ADR 前，必须由对应负责人确认。
