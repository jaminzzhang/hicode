# 需求评审 Skill 示例案例

> 本示例为 P4-WP7 脱敏结构示例，不是真实试点结果，不代表任何真实客户、保单、系统、分支或发布记录。

## 1. 适用 Skill

`references/skills/requirement-review/SKILL.md`

## 2. 脱敏输入摘要

需求编号：`REQ-DEMO-2026-001`

需求名称：批改单生效日期规则调整

输入来源：

1. 脱敏 PRD 摘要：允许部分批改类型的生效日期回溯至保单起保日，但不得早于保单起保日。
2. 脱敏业务说明：仅面向车险批改示例链路，不涉及真实产品。
3. 当前规则摘要：批改单生效日期不得早于申请日期。
4. 待变更规则摘要：指定批改原因可选择保单起保日至申请日期之间的日期。

已知缺口：

1. 未说明已收付费批改单是否允许回溯。
2. 未说明已出单、已退保、已终止保单的状态限制。
3. 未说明回溯后责任期间、保费重算和审计留痕要求。

## 3. 处理重点

1. 先判断需求是否足够进入开发。
2. 对保险核心业务逻辑、金额、状态流转、审计和监管风险做 P0/P1/P2/P3 分级。
3. 明确哪些问题必须由业务负责人或系统 Owner 确认。
4. 不补编缺失业务规则。

## 4. 示例输出片段

### 4.1 评审结论

结论：有条件通过

说明：需求目标和主要范围基本明确，但存在批改状态、收付费、保费重算和审计留痕等 P1 待确认问题。关闭 P1 问题前，不建议进入详细编码。

### 4.2 依据

| 依据来源 | 关键内容 | 说明 |
|---|---|---|
| 脱敏 PRD 摘要 | 批改单生效日期可回溯至保单起保日 | 可支持目标理解 |
| 当前规则摘要 | 生效日期不得早于申请日期 | 可支持差异识别 |
| 领域知识 | 批改影响责任期间、保费和状态流转 | 触发金融核心系统风险检查 |

### 4.3 完整性检查

| 检查项 | 结果 | 说明 | 风险等级 |
|---|---|---|---|
| 背景与目标 | 通过 | 目标是调整批改生效日期规则 | P2 |
| 范围内/范围外 | 待确认 | 未明确涉及哪些批改原因、产品和渠道 | P1 |
| 业务流程 | 待确认 | 未说明收付费、核保、出单后流程 | P1 |
| 业务规则 | 缺失 | 缺少已退保、已终止、已收费场景规则 | P1 |
| 验收标准 | 待确认 | 未列出边界日期和异常状态验收标准 | P1 |
| 负责人或确认人 | 待确认 | 需业务负责人和核心批改 Owner 确认 | P2 |

### 4.4 风险清单

| 风险 | 等级 | 说明 | 建议动作 | 建议确认人 |
|---|---|---|---|---|
| 回溯日期导致责任期间错误 | P1 | 批改单责任起止和保单期间可能不一致 | 明确责任期间重算规则 | 业务负责人 |
| 保费重算口径不明 | P1 | 回溯批改可能影响应收、实收和退补费 | 补充金额计算和舍入规则 | 精算或业务负责人 |
| 状态流转限制缺失 | P1 | 已退保或已终止保单是否允许批改未明确 | 补充状态矩阵 | 核心系统 Owner |
| 审计字段缺失 | P2 | 回溯原因和操作人留痕未说明 | 补充审计要求 | 合规或系统 Owner |

### 4.5 待确认问题

| 问题 | 优先级 | 影响 | 建议确认人 |
|---|---|---|---|
| 哪些批改原因允许回溯生效日期？ | P1 | 决定开发范围和测试范围 | 业务负责人 |
| 已收付费批改单是否允许回溯？ | P1 | 影响金额和交易一致性 | 财务/业务负责人 |
| 已退保、已终止、已满期保单是否允许批改？ | P1 | 影响状态流转和异常处理 | 核心系统 Owner |
| 回溯原因是否必须写入审计日志？ | P2 | 影响合规和可追溯性 | 合规负责人 |

## 5. 人工复核关注点

1. 业务负责人是否确认可回溯批改原因清单。
2. 金额重算、收付费和状态限制是否形成明确规则。
3. 审计和监管要求是否进入后续编码计划。

## 6. 禁止误用说明

1. 不得把本示例中的批改规则当作真实业务规则。
2. 不得把“有条件通过”当作研发负责人最终批准。
3. 不得补写真正产品、客户、保单或生产数据。
